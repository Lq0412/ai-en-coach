# MS1 面试主链路第 7—10 节详细交互规格

> 日期：2026-07-14
> 上游 PRD：`docs/ms1/week2/林锵/2026-07-14-ms1-interview-prd-design.md`
> 对应原型：`prototype/public/spreak-prototype.html` 与 `prototype/public/spreak-prototype.js`
> 文档性质：产品行为、问题策略、页面交互与验收规格；不决定数据库、通信协议、语音或模型厂商

## 0. 本稿解决什么

第 7—10 节不是四个松散页面，而是一条可恢复、可追溯的面试主链路：

```text
创建面试计划
  -> 确认 1—4 位面试官
  -> 选择某位面试官
  -> 选择完整模拟或该面试官的专项练习
  -> 麦克风检查
  -> 根据岗位 JD、候选人资料与面试官职责进行动态问答
  -> 生成报告
  -> 查看原回答、回听、同题复练或再练一场
```

### 0.1 MS1 统一术语

| 术语                    | 含义                           | 不表示         |
| --------------------- | ---------------------------- | ----------- |
| 面试计划 `InterviewPlan`  | 一份已锁定岗位、JD、候选人资料和练习重点的长期练习容器 | 一场已开始的面试    |
| 面试官 `Interviewer`     | 一个提问视角和风格                    | 传统招聘流程的必经轮次 |
| 场次 `InterviewSession` | 用户和某一位面试官进行的一次独立面试           | 多位面试官同场     |
| 问次                    | 完整模拟或专项练习中的一次提问          | 新的面试轮次      |
| `Turn`                | 已保存的问题 + 一次有效回答音频 + 可用转录     | AI 说了一句话    |
| 同题复练 `RetryAttempt`   | 针对报告中某一题缺口的再回答               | 重开完整面试      |

### 0.2 原型总体改造决策

| 原型内容             | MS1 决策      | 说明                                    |
| ---------------- | ----------- | ------------------------------------- |
| 创建职位的三步骤结构       | 保留并简化       | 每屏只完成一类决定：岗位、候选人资料、练习重点               |
| 4 张面试官卡          | 保留视觉，改交互    | 它们是 4 个独立视角，不是必须依次完成的 4 轮             |
| “完整面试”和专项练习      | 保留概念，重新设计   | 完整模拟可重复多次；每位面试官的专项由 JD、候选人资料与该面试官职责决定 |
| 轮次轮播页            | 改为面试官场次列表   | 每位面试官显示自己的进度和操作                       |
| 实时对话视觉           | 保留          | 补充第几问、AI/用户/处理状态和恢复操作                 |
| “准备话术”           | 改为“面试前准备”   | 展示当前面试官与练习类型的考察方向，不提前透出实际动态问句         |
| 逐句改进、语法/词汇/发音错题本 | MS1 不作为真实交付 | MS1 复习主入口是报告中的 4 张回答复盘卡               |

## 1. 第 7 节：创建面试计划

### 1.1 用户目标与核心规则

用户要快速建立一份可长期重复练习的面试计划。面试官的问题来自四类输入：

```text
岗位 JD + 候选人资料 + 面试官职责 + 用户重点练习内容
```

本节遵守以下规则：

- 所有字段都可以不填，用户零输入也能进入完整体验。
- 用户不需要从简历中选择一段具体经历。系统解析整份简历，后续再根据 JD 与面试官职责选择提问证据。
- 只填岗位名称时，AI 自动生成 JD 摘要；用户已填 JD 时，优先使用用户原文。
- 无简历时可手动填写候选人资料。只要用户填写了任何真实信息，系统就不得使用虚构内容补齐缺失字段。
- 岗位、JD 和候选人资料都为空时，进入明确标注的“示例体验模式”，使用一整套相互一致的虚拟岗位、JD 和简历。
- 点击“生成面试计划”且生成成功后，历史首页立即新增一张计划卡。

### 1.2 三步渐进式创建

创建页不把所有字段、上传区和选项堆在同一屏。每一步只让用户处理一类信息，并使用渐进展开隐藏非当前内容。

#### 第一步：岗位

首屏只突出“岗位名称”，JD 收起在“添加或查看职位描述”中。

| 输入情况 | 系统行为 | 用户看到什么 |
|---|---|---|
| 岗位名称 + 用户 JD | 使用用户 JD，不用 AI 改写为另一个岗位 | 用户原文的收起摘要，可展开编辑 |
| 只有岗位名称 | AI 生成匹配该岗位的 JD 摘要 | 生成结果摘要，可直接继续或展开修改 |
| 岗位名称为空，但有 JD | AI 从 JD 提取岗位名称 | 提取的岗位名称与 JD 摘要，均可修改 |
| 岗位名称和 JD 都为空 | 先使用默认模拟岗位，在第二步继续判断是否进入示例模式 | “将使用示例岗位，你之后仍可修改” |

点击“下一步”后进入候选人资料。AI 生成 JD 失败时不阻塞流程，保留岗位名称，并允许重试或稍后在计划生成时再试。

#### 第二步：候选人资料

页面首先只显示四个简洁选项，选中后仅展开该路径的内容：

| 选项 | 点击后 | 完成条件 |
|---|---|---|
| 使用已保存简历 | 展示账户内简历列表，每项有文件名、更新时间和解析状态 | 选中一份简历 |
| 上传新简历 | 只展示上传器、文件要求和进度 | 上传完成；解析可在后台继续 |
| 手动填写资料 | 显示候选人字段，任何字段均可跳过 | 用户可填任意部分，也可保持为空 |
| 使用示例体验 | 展示示例人物与简历摘要，并持续显示“示例体验”标识 | 用户确认使用示例资料 |

##### 简历自动解析与确认

- 系统解析整份简历，不要求用户从中选择具体经历。
- 解析结果至少按“基本信息、教育背景、实习/工作经历、项目经历、技能”分组。
- 解析完成后默认只展示一屏摘要。用户可直接继续，也可点击“展开修改”编辑字段。
- 修改内容只作为当前面试计划的候选人资料快照，不回写原始简历或账户中保存的解析结果。
- 解析失败时保留已填岗位、JD 与其他输入，同时提供“重试解析”、“换一份简历”和“改为手动填写”。

##### 无简历时的手动字段

| 字段组 | 字段 | 规则 |
|---|---|---|
| 基本信息 | 姓名 | 选填；只用于面试称呼和报告展示 |
| 教育背景 | 学校、专业、学历、毕业时间 | 选填；可新增多条教育经历 |
| 实习/工作经历 | 单位、职位、时间、职责、成果 | 选填；可新增多条 |
| 项目经历 | 项目名称、背景/目标、个人职责、关键行动、成果 | 选填；可新增多条，不要求选出唯一项目 |
| 技能 | 专业能力、工具、语言或证书 | 选填；支持标签与自由补充 |
| 其他 | 个人优势、获奖或其他可验证成果 | 选填 |

用户只填了部分真实字段时，系统只使用这些真实内容，其他字段保持缺失。后续面试官可以在面试中请用户补充，但不得把示例项目混入真实资料。

##### 示例体验模式

- 只有在用户明确选择“使用示例体验”，或岗位、JD 和候选人资料均为空时才启用。
- 岗位、JD、教育、实习/工作、项目和技能使用同一套内部一致的 mock 资料，不临时拼接互相冲突的示例。
- 创建流程、面试页、报告和历史卡都持续显示“示例体验”标识。
- 用户改填任何真实岗位或候选人信息后，退出示例模式，不保留其他 mock 个人经历。

#### 第三步：练习重点

页面先展示少量常用标签，支持多选，再提供一个自由文本补充框。标签和自由文本都可跳过。

初始标签建议为：“自我介绍”、“简历深挖”、“岗位能力”、“项目取舍”、“情景题”、“英语表达”。这些是提问权重与面试官专项生成输入，不是每位面试官都必须显示的固定专项菜单。

### 1.3 “生成面试计划”的提交边界

1. 用户可在任何字段为空时提交。系统根据输入决定使用真实资料、部分资料或示例模式。
2. 提交前展示一行摘要：岗位、候选人资料来源、已选练习重点和是否为示例模式。不新增一个堆满字段的“最终确认页”。
3. 点击后按钮立即显示“正在生成面试计划…”，并禁止重复提交。
4. 成功时保存岗位、最终 JD、解析或手动得到的候选人资料快照、练习重点和示例模式标记。
5. 计划生成成功后立即在“历史”首页新增一张计划卡，再进入第 8 节的面试官配置等待状态；配置成功后直接进入计划详情。
6. 生成失败时不新增计划卡，留在当前页并保留所有输入。
7. 重试复用同一次创建意图，最终只能产生一份有效计划和一张历史卡。

### 1.4 页面状态与恢复

| 状态 | 页面反馈 | 主操作 |
|---|---|---|
| JD 生成中 | 岗位名称保留，JD 区显示生成状态 | 等待 / 跳过后续再试 |
| JD 生成失败 | 保留岗位名称，不清空其他步骤 | 重试 / 手动填写 / 继续 |
| 无已保存简历 | “使用已保存简历”显示空状态，其他三种路径仍可用 | 上传 / 手动填写 / 示例体验 |
| 简历上传中 | 只展示当前文件进度 | 取消 / 返回资料来源选择 |
| 简历解析中 | 显示阶段与进度，不展示伪解析内容 | 继续等待 / 换一份简历 |
| 简历解析失败 | 展示可理解的失败原因 | 重试解析 / 换简历 / 手动填写 |
| 解析摘要可用 | 展示分组摘要和资料完整度提示 | 直接继续 / 展开修改 |
| 编辑未保存 | 切换资料来源或返回上一步时提示 | 保存本次修改 / 放弃 / 继续编辑 |
| 计划生成中 | 当前摘要保留可见，主按钮 loading | 等待；禁止再次提交 |
| 计划生成失败 | 顶部错误条 + 原输入 | 重试 / 返回修改 |

### 1.5 历史首页的计划卡

每次面试计划生成成功后，历史首页新增一张独立卡片。历史首页不直接铺开面试官、专项和每一次练习。

卡片显示：

- 目标岗位。
- 创建时间。
- 候选人资料来源：已保存简历、新上传简历、手动资料或示例体验。
- 已练习次数、最近练习的面试官/专项，以及是否有进行中练习。
- 主操作为“进入面试计划”；有进行中练习时可显示“继续练习”状态提示。

点击卡片进入计划详情。计划详情是选择面试官、完整模拟或专项练习的主入口，并在页面中提供统一的“查看练习历史”按钮。

### 1.6 验收场景

1. 岗位与 JD 均为空时仍可继续，系统明确提示将使用示例岗位。
2. 只填岗位名称时，AI 生成可查看、可修改的 JD 摘要；用户可不修改直接继续。
3. 用户已填 JD 时，系统不用无关的 AI JD 覆盖用户原文。
4. 候选人资料首先只显示四个来源选项，选中后只展开对应内容。
5. 选择已保存简历或上传新简历后，系统解析整份简历，不要求用户选择具体经历。
6. 解析完成后显示可直接继续、也可展开修改的摘要。
7. 解析失败时保留其他输入，并同时提供重试、换简历和手动填写。
8. 无简历时可手动填写任意部分字段；缺失字段不使用 mock 补齐。
9. 零输入进入示例模式时，创建、面试、报告和历史都展示“示例体验”标识。
10. 练习重点支持标签多选与自由文本，两者都可跳过。
11. 计划生成成功后历史首页新增一张计划卡；生成失败时不新增。
12. 生成请求重复到达时，最终只产生一份计划和一张历史卡。

## 2. 第 8 节：生成与编辑面试官

> 评审状态：第 7—10 节已按最新讨论更新。其中第 10 节的一问一答主流程、动态问题生成和时长控制已确认；“实时通话”仅保留入口，其切换与打断交互待后续讨论。本文档仍不代表已批准修改原型。

### 2.1 目标与生成原则

用户创建面试计划后，系统自动为该岗位配置 2—4 位具有不同考察视角的面试官。用户不需要先选面试官类型，生成完成后可直接练习，也可在首次练习前编辑、添加或删除。

生成不从空白开始，而是使用“系统预设 + 一次 AI 轻量适配”：

1. 系统内部面试官库按岗位族维护常见视角，例如招聘/综合、直属业务、专业能力、跨团队协作与高层判断。
2. 系统根据目标岗位自动选出 2—4 个差异化视角，不在页面让用户选择模板。
3. 单次 AI 请求结合岗位 JD、候选人资料和用户练习重点，同时完成整组面试官的轻量适配。
4. AI 不逐字段、逐面试官分开请求，避免等待累积、角色重复和部分成功状态。

### 2.2 一次适配生成的内容

| 对象 | AI/预设需要给出 | 不在本节生成 |
|---|---|---|
| 面试官 | 虚拟姓名、具体职位、一句职责、提问风格、2–3 个关注点与默认音色 | 头像或虚拟人实现方案 |
| 完整模拟 | 预计时长、环节名称与环节顺序 | 正式面试问题 |
| 面试官专项 | 根据该面试官职责生成 2—5 个专项；每个专项包含名称和一句考察说明 | 正式问题清单或固定题库 |

面试官姓名是卡片主要识别信息，职位作为副标题。头像只预留稳定展示位，当前不决定是使用内置头像还是后续接入虚拟人。系统为每位面试官自动分配默认音色，音色不是开始练习的必选项。

正式问题必须在用户开始某次完整模拟或专项练习时再动态生成，以便同一专项可多次练习、问题不完全重复，且后续问题能根据当场回答追问。

### 2.3 生成状态与失败回退

1. 第 7 节点击“创建面试计划”后，计划先保存，再开始一次整组面试官适配。
2. 正常路径使用轻量等待页，文案为“正在根据岗位配置面试官…”，不使用虚假百分比或堆叠多个伪生成步骤。
3. 生成通常应为短等待。用户主动离开、刷新或重新登录时，历史计划卡保留“正在配置面试官”状态，可以稍后恢复。
4. 生成成功后直接进入面试计划详情，整组面试官自动保存且立即可用，不增加独立的“确认面试官”页或“使用这组面试官”按钮。
5. 生成失败时计划不丢失。页面提供“重试配置”和“使用系统预设”，两种操作都不新建重复计划。

### 2.4 计划详情中的面试官卡

生成完成后直接展示面试计划详情。卡片首屏保持简洁，不把所有风格、环节和专项堆在列表中。

| 卡片信息/操作 | 首屏展示 | 点击后 |
|---|---|---|
| 预留形象位 | 头像/虚拟人容器 | 当前不提供形象生成或编辑 |
| 身份 | 虚拟姓名为主标题，具体职位为副标题 | 进入面试官详情 |
| 职责 | 一句职责摘要 | 详情中展示完整文本 |
| 关注点 | 显示 2—3 个标签 | 详情中可查看与首次练习前编辑 |
| 音色 | 显示当前音色名称的简短摘要 | 详情中可试听和更换，不开始 Session |
| 主操作 | 开始/继续练习 | 进入该面试官详情，再选完整模拟或专项 |

面试官详情页展示：

- 姓名、职位、职责、提问风格与关注点。
- 当前音色摘要和“试听与更换”入口。
- “完整模拟”：预计时长和环节顺序，不显示正式问题。
- 2—5 个该面试官专属专项：每项显示名称与一句考察说明。
- 统一“练习历史”入口：查看该面试官的完整模拟和全部专项练习记录。

### 2.5 首次练习前的编辑边界

用户可以在任何练习产生前调整面试官阵容，但不提供含糊的“重新生成”或复杂的停用/恢复管理。

| 操作 | 规则 |
|---|---|
| 编辑面试官 | 可修改姓名、职位、职责、提问风格与关注点 |
| 试听与更换音色 | 系统已自动分配默认音色。选择器展示音色名称、声音特点、口音和试听按钮；用户可直接更换 |
| 编辑专项 | 可修改名称和考察说明，可增加或删除；每位最终保留 1—5 个专项 |
| 编辑完整模拟 | 可对环节拖动排序、隐藏或恢复，不编辑尚未生成的正式问题 |
| 添加面试官 | 少于 4 位时可用。系统从内部预设角色方向快速创建一张已填写卡，用户再编辑；不调用 AI，不显示空白大表单 |
| 删除面试官 | 大于 1 位时可删除；删除前二次确认 |

当用户开始该计划中任意一次完整模拟或专项练习时，整个面试官阵容立即锁定：

- 面试官、专项和完整模拟环节不再允许编辑、添加或删除。
- 音色是用户体验偏好，不随内容配置锁定；首次练习后仍可在面试官详情中试听和更换。
- 更换音色只影响之后的语音播放。已完成练习的问答、音频和报告不重新生成，也不改变历史快照。
- 页面不展示停用、移出计划或重新生成等低频管理入口。
- 用户需要另一套面试官时，新建一份面试计划，不在已练习计划中维护多版配置。

### 2.6 历史与快照

- 面试官详情页是查看该面试官全部练习历史的统一入口。
- 同一面试官的完整模拟和各专项都可重复练习，每次产生独立记录与报告。
- 开始练习时保存该次使用的面试官、练习类型和环节快照。
- 用户从面试官详情进入“练习历史”后，可查看完整模拟和不同专项的每一次记录。

### 2.7 验收场景

1. 产品岗位与研发岗位可从同一预设库中自动选出不同的 2—4 位面试官，页面不要求用户选类型。
2. 整组面试官通过一次 AI 请求完成适配，不发生逐人等待或部分卡片已完成、部分失败。
3. 生成成功后直接进入计划详情，不要求整组确认，也不创建 Session。
4. 卡片首屏仅展示姓名、职位、一句职责和 2—3 个关注点，详细风格、完整模拟与专项收在详情中。
5. 每位面试官有 2—5 个符合其职责的专项，不同面试官不被强制使用相同专项菜单。
6. 完整模拟显示预计时长与环节顺序，专项显示名称和考察说明，两者都不提前生成正式问题。
7. 首次练习前，用户可以编辑面试官与专项，也可排序、隐藏或恢复完整模拟环节。
8. 首次任意练习开始后，整组配置锁定，不再出现编辑、添加、删除、停用或重新生成入口。
9. 添加面试官使用内部预设快速创建，不再调用 AI；面试官总数始终保持 1—4 位。
10. 生成失败、刷新或离开页面不删除计划，用户可从历史计划卡恢复。
11. 每位面试官都有可直接使用的默认音色，用户可在详情页查看音色特点和口音、试听并更换。
12. 首次练习后仍可更换音色，新音色只影响后续播放，不改动旧音频、报告或历史。

## 3. 第 9 节：面试计划详情与场次选择

### 3.1 页面定位

页面标题使用岗位名称或“面试计划”。它是一份长期计划的练习总入口，用户可以自由切换面试官，不表示必须按轮播顺序完成传统面试轮次。

### 3.2 页面结构与入口

#### A. 计划摘要

显示目标岗位、候选人资料来源、已选练习重点、创建时间与示例体验标识（如适用）。

| 操作 | 点击后 |
|---|---|
| 查看面试背景 | 打开只读快照，展示本计划锁定的岗位、JD、候选人资料、练习重点与示例模式标记 |
| 编辑面试官 | 首次练习前进入面试官详情编辑；任意练习开始后仅允许查看 |
| 查看练习历史 | 进入本计划的统一历史，再按面试官、完整模拟/专项和时间查看记录 |

#### B. 面试官轮播

一次聚焦展示一位面试官，用户通过左右滑动、箭头或底部指示点切换。卡片显示姓名、职位、职责、关注点、当前音色摘要、最近练习状态与练习次数。

轮播只是浏览方式，不锁定先后顺序。任何面试官都可以率先开始，不同面试官的进度相互独立。

卡片操作层级：

1. **完整模拟**：卡片主按钮，可重复开始多次。
2. **更多练习**：次按钮，打开底部面板，展示该面试官的 2—5 个专项名称与一句考察说明。
3. **面试官详情**：查看完整设置、更换音色，并在首次练习前编辑配置。
4. **练习历史**：查看该面试官的完整模拟和专项记录。

### 3.3 练习准备页

点击“完整模拟”或底部面板中的某个专项后，不立即请求麦克风。先进入一屏简洁准备页，展示：

- 面试官姓名、职位与当前音色。
- 练习类型：完整模拟或具体专项。
- 建议时长：完整模拟约 15 分钟，专项约 8 分钟。
- 完整模拟的环节顺序，或专项的考察方向。
- “开始练习”与“返回选择”。

点击“开始练习”后，创建仅绑定当前面试官和当前练习类型的 Session，保存计划、面试官、专项/完整模拟和环节快照，再进入麦克风检查。请求中禁止重复点击；创建失败时留在准备页重试。

### 3.4 进行中、完成与历史

| 状态 | 轮播卡主操作 | 行为 |
|---|---|---|
| 从未练习 | 完整模拟 | 进入准备页 |
| 存在进行中 Session | 继续练习 | 打开原 Session，简化麦克风检查后回到当前未完成问题；不新建 Session |
| 最近一次已完成 | 完整模拟 | 可开始新 Session，不覆盖旧报告 |
| 最近一次提前结束 | 完整模拟 | 可开始新 Session；原 Session 保留阶段性报告 |
| 报告生成中/失败 | 查看状态/重试报告 | 已完成问答不回滚，不要求重新面试 |

中途退出时保留已完成的有效 Turn，丢弃未提交音频。用户再次进入时打开原 Session，从当前未完成问题继续。

完整模拟和每个专项均可重复练习。每次都新建独立 Session，不继承上次对话上下文，也不覆盖旧记录与报告。

### 3.5 验收场景

1. 计划详情使用面试官轮播，可自由切换任意人，不锁定练习顺序。
2. 卡片主操作为“完整模拟”，“更多练习”打开不撑高卡片的底部面板。
3. 底部面板只展示当前面试官的专项，切换面试官后不串用上一人的专项。
4. 选择完整模拟或专项后先进入准备页，不立即弹麦克风权限或创建问答。
5. 完整模拟预览约 15 分钟与环节顺序，专项预览约 8 分钟与考察方向，不显示正式问题。
6. 已有进行中 Session 时，“继续练习”恢复原 Session，不重复创建。
7. 完整模拟和专项均可开始多次，新记录不覆盖旧报告。
8. 用户可从计划详情的统一历史入口或面试官详情的历史入口找到同一批练习记录。
9. 连续点击“开始练习”最终只产生一个 Session。

## 4. 第 10 节：实时语音面试

### 4.1 一场面试如何控制时长和问题数

用户暂不选择时长或问题数量。系统根据练习类型自动给出默认控制目标：

| 练习类型 | 默认时长 | 主问题目标 | 单个主问题追问上限 | 结束规则 |
|---|---:|---:|---:|---|
| 完整模拟 | 约 15 分钟 | 最多 6 个 | 最多 1 个 | 达到时间上限、完成核心目标或用户提前结束 |
| 专项练习 | 约 8 分钟 | 最多 3 个 | 最多 1 个 | 达到时间上限、完成专项目标或用户提前结束 |

以上不是要求 AI 一定问满的问题配额。系统控制器同时看“剩余时间”和“剩余目标”：

- 时间充足且目标未覆盖：进入下一个主问题。
- 当前回答存在高价值缺口且仍有时间：允许一次追问。
- 剩余时间不足以展开新主题：不再开新问题，进入结束语和报告生成。
- 已覆盖本场核心目标：可以提前正常完成，不为了凑数继续提问。
- 用户主动提前结束：立即停止后续提问，保留已完成问答并生成标记为“未完整完成”的阶段性报告。

完整模拟是否包含自我介绍由岗位和面试官职责决定；并非所有场次固定以自我介绍开场。专项练习只有在用户选择“自我介绍”专项时才以此为练习目标。

### 4.2 麦克风检查页

#### 标准流程

1. 显示本场面试官、练习类型、预计时长和大致环节，不承诺固定问数。
2. 用户点击“检查麦克风”后再请求权限，不在进页瞬间无说明弹权限框。
3. 权限通过后列出输入设备并展示实时音量。
4. 用户点击“开始试音”并说一句短句；系统只检查是否收到有效声音。
5. 检查成功后主按钮改为“进入面试”，必须由用户主动点击。

试音不保存、不转录、不作为 Turn。若用户在第一问产生前退出，MS1 统一保留该 Session 为“待开始问答”，回到第 9 节后显示“继续面试”，不静默删除 Session。

#### 检查异常

| 问题 | 页面说明 | 操作 |
|---|---|---|
| 权限未决定 | 说明为什么需要麦克风 | 检查麦克风 |
| 权限被拒绝 | 给出当前浏览器的设置路径 | 重新检测 / 返回计划 |
| 无输入设备 | 说明未检测到麦克风 | 刷新设备 / 返回计划 |
| 有设备但无声音 | 提示检查静音、系统输入源和音量 | 重新试音 / 切换设备 |
| 设备在检查后失效 | 进入面试前再做可用性检查 | 重新检测 |

### 4.3 问题从哪里来

开始 Session 时，系统使用以下快照生成本场隐藏的“面试目标与覆盖提纲”，同时只生成第一道具体问题：

1. 岗位名称与最终 JD：决定岗位能力、业务场景和必要考察项。
2. 简历解析结果或手动候选人资料：提供可追问的公司、项目、经历、技能和成果。
3. 当前面试官职责、风格和关注标签：决定问题侧重点与表达方式。
4. 当前练习类型：完整模拟使用阶段提纲；专项练习只使用所选专项的目标。
5. 用户填写的重点练习内容：提高相关目标的优先级，但不能排除岗位必要考察项。
6. 本场默认时长和问题上限：限制能展开的主题数量与追问深度。

隐藏提纲不是提前生成的固定题库，也不向用户展示标准答案。它只记录“这一场还需要确认什么”，例如岗位动机、核心项目、个人贡献、方案取舍、结果验证、协作方式或专业能力。

### 4.4 每轮回答后如何决定下一问

每个有效回答保存后，控制器先做判断，再让 AI 只生成下一道具体问题：

```text
保存本轮回答
  -> 提取回答中明确出现的事实与证据
  -> 标记已覆盖、部分覆盖、未覆盖的面试目标
  -> 检查剩余时间、主问题数和当前题追问次数
  -> 在“追问当前缺口 / 进入下一个目标 / 正常结束”中三选一
  -> 只生成并保存下一道问题
```

生成规则：

- 面试官使用英语提问，用户使用英语回答；页面状态、错误和诊断使用中文。
- 每次只问一个主问题；可以带一个简短限定条件，不能堆叠三四个子问题。
- 优先问“JD 要求 × 简历证据”的交集，再结合该面试官职责选择角度。
- 后续问题可以引用用户刚刚真实说出的项目名、任务、方案或结果，不得补造客户、数据、职级或决策。
- 当前回答偏离问题但出现高价值岗位证据时，控制器可记录该证据，再用一次追问拉回当前目标。
- 当前回答缺少关键信息但追问次数已用完时，记录为“部分覆盖”，继续下一目标，不陷入循环。
- 问题文本必须先保存再播放；播放失败只重播同一文本，不重新生成题目。
- 剩余时间不足时优先完成当前目标，不新开需要长篇回答的新主题。

### 4.5 可能问什么、系统怎么看

问题不固定，但必须落在当前面试官和练习类型允许的目标中。常见目标示例如下：

| 目标 | 可能的问题（英文） | 主要看什么 | 常见追问触发点 |
|---|---|---|---|
| 自我介绍/岗位动机 | “Could you briefly introduce yourself and explain why this role is a strong fit for you?” | 经历是否围绕岗位组织，动机是否具体 | 只复述简历、没有岗位连接 |
| 项目概览 | “Could you give me a concise overview of [project], including the problem, your role, and the goal?” | 背景、目标、个人角色是否清楚 | 只讲团队、不说个人范围 |
| 个人贡献 | “What did you personally do to move [goal] forward, and what did the rest of the team own?” | “我”和“我们”的边界、关键行动 | 只罗列会议或交付物 |
| 方案取舍 | “What alternatives did you consider for [solution], and why did you choose this approach?” | 备选方案、约束、判断依据和风险 | 只给最终做法、没有依据 |
| 结果验证 | “How did you validate the outcome, and what evidence did you get?” | 目标、验证方法和结果是否连通 | 只说“效果很好”或强行编数字 |
| 协作与冲突 | “Tell me about a disagreement with a key partner and how you reached a decision.” | 分歧本身、沟通动作和决策结果 | 把冲突归因于他人能力 |
| 专业能力 | 由 JD 能力项和简历技能生成，例如围绕研究、设计、技术、运营或管理场景 | 是否能将专业知识用于真实问题 | 只背概念、没有实际应用 |
| 反思与成长 | “What would you do differently if you handled the same situation again?” | 是否能识别限制并形成可复用经验 | 只有空泛复盘 |

没有数字时允许用户提供定性、过程或代理证据；系统不得诱导用户编造数据。某项信息在资料和前文中都不存在时，问题应以开放式请求补充，而不是把假设写进题干。

#### 动态串联示例

完整模拟中，产品岗位 JD 强调“用户洞察和优先级判断”，简历包含“创作者后台改版”，面试官为业务负责人：

1. 第一问从匹配点开始：“Could you walk me through the creator dashboard redesign and the user problem you were trying to solve?”
2. 用户说“新用户首次配置成本高”，但没有说个人贡献；控制器将项目背景标记为已覆盖、个人贡献标记为未覆盖。
3. 仍有时间且本题尚未追问，下一问引用真实回答：“What did you personally do to identify and reduce that setup cost?”
4. 用户说“保留新手引导，延后分析面板”；后续进入取舍目标：“Why did you prioritize onboarding over the analytics dashboard, and what evidence supported that choice?”
5. 若此时剩余时间很少，系统不再开启新的协作案例，而用结果验证问题收束；若用户从未提到“分析面板”，系统不得把它写入问题。

### 4.6 实时面试页的逐状态交互

| 状态 | 页面必须显示 | 可用操作 | 禁止操作 |
|---|---|---|---|
| 连接中 | 面试官、“正在连接”、练习类型与剩余时间 | 返回计划 | 录音、重播 |
| 问题生成中 | “面试官正在准备下一个问题” | 取消等待并返回 | 录音 |
| AI 提问中 | 问题文本、播放动画、“面试官正在说话” | 用户自然开口打断、退出 | 手动提交空回答 |
| 用户回答中 | “正在听你回答”、音量反馈、录音时长 | 结束回答、退出 | 重播问题、进入下一问 |
| 回答处理中 | 上传/转录进度的统一文案 | 等待 | 重复提交、进入下一问 |
| 回答无效 | “这次回答过短或未识别，未计入进度” | 重答本题、重播本题 | 跳过本题 |
| Turn 保存成功 | 更新已完成问答数与剩余时间 | 自动判断追问、下一目标或结束 | 覆盖刚保存的回答 |
| 报告生成中 | “本场问答已结束，正在生成报告” | 留在当前页 / 返回计划 | 继续提问 |

页面长期显示的信息只有：面试官、练习类型、已完成问答数、剩余时间、当前状态和退出入口。不在面试中展示评分、证据完整性或改进建议，避免打断回答。

本期默认采用“一问一答”：AI 播放结束后用户回答，回答结束并保存后才生成下一问。页面预留“实时通话”按钮；何时允许切换、切换中的确认、打断和恢复规则留待后续讨论，本稿不将其作为已确认交互。

### 4.7 什么算一次有效回答

一个 Turn 只在以下条件全部成立后计数：

- 用户语音已结束，不是传输中的片段。
- 完整音频已成功保存。
- 转录可用，不是空白或只有无法识别标记。
- 达到用于过滤误触、背景噪声和极短声音的最低有效阈值。

阈值不用于评价回答质量，也不向用户展示内部秒数或词数。产品文案统一使用“回答过短，请再试一次”。

### 4.8 实时通话模式（预留，待细化）

- 对话页预留“实时通话”入口，作为用户熟练后更接近真实面试的模式。
- 实时模式预计需要支持自然打断（Barge-in）、回声消除、误打断处理和说话权切换。
- 这些能力不影响本期已确认的一问一答主流程。
- 在用户确认切换时机、切换方式与退出恢复规则前，不把实时模式写成开发验收项。

### 4.9 退出、提前结束与恢复

| 操作 | 确认文案要点 | 数据结果 | 下次回来 |
|---|---|---|---|
| 退出页面 | “当前未完成的录音将丢失，已完成问答会保留” | 保留有效 Turn，丢弃未完成音频 | 第 9 节显示“继续练习” |
| 提前结束 | “将结束本场面试，无法继续，并根据已完成内容生成阶段性报告” | 保留已完成 Turn，Session 标记“提前结束” | 可查看未完整报告或“再练一场” |
| 浏览器崩溃/刷新 | 无法事前确认 | 只保留已存 Turn | 重连后回到当前未完成问 |

“退出”不等于“提前结束”。前者允许恢复原 Session，后者封闭原 Session，用户若要再练必须创建新场次。

### 4.10 异常恢复矩阵

| 失败点 | 不能做什么 | 保留什么 | 用户从哪里重试 |
|---|---|---|---|
| 语音连接失败 | 不创建 Turn | Session、已完成 Turn | 当前问连接页 |
| 问题生成失败 | 不跳到下一问 | 前序 Turn | 重试生成当前问 |
| 问题文本已保存但播放失败 | 不重新生成另一问 | 已保存问题 | 重播同一问题 |
| 回答中断线 | 不保存不完整 Turn | 已完成 Turn、当前问题 | 重连后重答当前问 |
| 音频上传/保存失败 | 不增加进度 | 若本地完整音频仍可用，可短时保留用于重传 | 重传；无完整音频则重答 |
| 转录失败 | 不将其计为有效 Turn | 已上传音频 | 重试转录；多次失败则重答 |
| 回答过短/空白 | 不增加进度 | 当前问题 | 原题重答 |
| 下一问生成失败 | 不回滚上一 Turn | 已完成 Turn | 重试下一问 |
| 正常结束或提前结束后报告失败 | 不重开面试 | 已完成 Turn 与 Session 结束状态 | 第 9 节点击“重试生成报告” |

所有恢复遵守：**已完成的不重做，未完成的不假装完成**。同一回答的重复请求最终只能产生一个有效 Turn；主问题和追问均受该练习类型的上限约束。

### 4.11 面试结束后的明确去向

1. 控制器判定目标完成、达到时间/问题上限，或用户主动提前结束后，立即停止录音和继续提问。
2. 页面显示“本场问答已结束”与报告生成进度；提前结束时明确标记“未完整完成”。
3. 报告成功时自动进入报告，同时提供“返回面试计划”。
4. 报告按实际完成的 Turn 展示复盘卡，每张卡提供原问题、转录、回听、证据状态、诊断、下次目标和“同题复练”。
5. 报告生成失败时不把 Session 退回“进行中”；用户可返回第 9 节后单独重试报告。

### 4.12 第 10 节验收场景

1. 麦克风检查未通过时不能进入第 1 问，试音不保存。
2. 完整模拟默认按约 15 分钟、最多 6 个主问题控制；专项默认按约 8 分钟、最多 3 个主问题控制，用户开始前不选择参数。
3. 系统先生成隐藏目标提纲和第一问，不一次生成整套固定问题。
4. 回答没有给出个人角色时，下一问可请求补充，不虚构用户担任负责人。
5. 回答提到具体方案时，后续可引用该方案；未提到则使用开放式问法。
6. 结果问题不强迫用户编造数字，可追问定性、过程或代理证据。
7. 空白、过短或转录失败的回答不计数，仍停留在当前问题。
8. 已完成 2 个 Turn，当前回答中断线，重连后保留前两问并重答当前问题。
9. 同一 Turn 保存请求重复到达时，最终只存一条有效 Turn。
10. 达到目标后可以少于问题上限正常完成；时间不足时不得为了凑题数开启新主题。
11. 用户可随时提前结束，已完成问答生成“未完整完成”的阶段性报告。
12. 报告生成失败时，已完成 Turn 不回滚，用户可从计划详情单独重试。
13. 本期一问一答主流程可完整验收；实时通话按钮仅预留入口，不验收尚未确认的切换与打断行为。

## 5. 跨第 7—10 节一致性规则

### 5.1 快照和历史不可变性

```text
点击“生成面试计划”
  -> 锁定计划的岗位、最终 JD、候选人资料、练习重点与示例模式标记

点击“开始面试/再练一场”
  -> 锁定整组面试官配置，并保存该 Session 使用的计划、面试官、完整模拟/专项与环节快照

完成一个 Turn
  -> 锁定实际问题、回答音频与转录

进行同题复练
  -> 新建 RetryAttempt，不覆盖原 Turn 或更早复练
```

### 5.2 建议统一状态名

| 对象 | 建议状态 |
|---|---|
| 简历 | 上传中 / 解析中 / 可用 / 解析失败 |
| 面试计划 | 正在配置面试官 / 配置失败 / 可开始 / 配置已锁定 |
| Session | 待开始问答 / 进行中 / 问答完成 / 未完成 / 已完成 |
| 实时页 | 连接中 / 问题生成中 / AI 提问中 / 用户回答中 / 回答处理中 / 等待重试 |
| 报告 | 生成中 / 可用 / 生成失败 |

不要同时使用“已结束”表示“正常完成”和“提前结束”。

### 5.3 数据隔离

- 一位用户只能访问自己的计划、面试官、Session、Turn、音频、转录和报告。
- 不同面试官的 Session 不共享当前题号、对话上下文、报告或复练版本。
- 回答音频必须通过受保护地址访问；退出登录或账号注销后不得继续读取。

## 6. 交付优先级与依赖

1. **先改信息架构**：使用“面试计划 -> 面试官 -> 完整模拟/该面试官专项”三层结构，历史首页只展示计划卡。
2. **再打通状态与恢复**：计划、面试官、Session、Turn 与报告状态必须可持久化。
3. **再实现动态问题控制**：按时长、问题上限和剩余目标决定追问、换题或结束，并保证问题先保存后播放。
4. **最后接报告与复习闭环**：按实际 Turn 生成复盘卡，回听、同题复练、全部场次和再练一场必须有稳定入口。

### 6.1 必须在实现前完成的技术验证

- 真实设备下的静音结束、最低有效回答与短噪声阈值。
- 实时通话模式启用后，再验证持续说话打断、回声消除与误打断率。
- 回答音频完整上传后的本地重传时窗与清理时机。
- 问题生成、Turn 保存、Session 创建和报告重试的幂等边界。

这些参数应配置化并通过真实设备测试确定，不在 PRD 中伪造精确秒数、分贝或词数。

## 7. 本稿的完成标准

- 产品可从本稿直接拆出第 7—10 节的页面、状态和验收条件。
- 设计可明确画出每个控件点击后的结果、错误状态和返回路径。
- 问题生成方可明确使用 JD、候选人资料、面试官职责、练习类型、练习重点和时间预算，动态决定追问、换题或结束。
- 用户在中断、完成或过几天后回来时，都能从“历史 -> 面试计划 -> 面试官”找到继续、报告、回听、同题复练和再练整场的入口。
