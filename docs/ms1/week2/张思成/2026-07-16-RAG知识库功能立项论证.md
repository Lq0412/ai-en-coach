# [提案] 建设面试知识库与 RAG，让 SpeakUp 的追问和反馈有据可查

> 文档用途：项目立项、中期答辩与评委质询  
> 日期：2026-07-16  
> 结论：建议将“面试知识库 + 轻量 RAG”纳入 MS2；MS1 先建设结构化知识资产和评测集，不为展示技术而匆忙上线向量数据库。

**通用大模型可以用一段提示词扮演面试官，但 SpeakUp 要解决的不是“有人问题”，而是“问对问题、根据回答正确追问、并给出有证据的可复练反馈”。RAG 的价值，是在每次提问和评价前调取经过调研、审核和版本管理的面试知识，让输出可验证、可更新、可评测。**

我们不把“接入 RAG”当作护城河。真正的护城河是：

1. 按岗位、职级、技术栈和面试阶段组织的高质量知识资产；
2. 每道题对应的考察目标、评分 Rubric、关键要点和追问树；
3. 经过用户真实练习数据验证的题目难度和反馈策略；
4. 从“练习—反馈—复练—改善”中持续更新的数据闭环。

## 二、为什么仅靠大模型和人设 Prompt 不够

### 2.1 问题不稳定

同一用户、同一岗位重复进入面试时，纯 Prompt 方案可能产生难度波动、超出岗位范围或无法支撑后续追问的问题。如果没有题目元数据和追问规则，系统很难证明“为什么要问这道题”。

### 2.2 反馈容易正确但空泛

“可以更具体”、“建议使用 STAR”往往没有错，但用户不知道缺少哪个证据、技术取舍或量化结果。只有把用户原回答与明确的 Rubric 和参考要点对齐，反馈才能转化为下一轮复练动作。

### 2.3 模型知识不便于追溯和更新

大模型参数内的知识不能由产品团队直接检查、修正或按岗位版本管理。面试知识库可以记录来源、适用岗位、审核人、有效期和版本，也能在发现错误时更新内容，而无需重新训练基座模型。

### 2.4 纯人设很容易被复制

“你是一名 Java 面试官”这类人设提示词无法形成持续差异化。通用 AI 产品只需新增一个 Agent 模板就能覆盖基础对话。但经过专家审核的知识结构、评分标准、真实回答样本和复练效果数据，需要长期积累才能复制。

## 三、公开研究数据支持

> 说明：下列是公开基准上的外部证据，用于证明 RAG 的技术机制有效；不能直接替代 SpeakUp 在英文技术面试场景的自有对照实验。

| 研究 | 关键数据 | 对 SpeakUp 的启示 |
|---|---|---|
| NeurIPS 2020 原始 RAG 论文 | 在 Natural Questions 上，RAG-Sequence 的 Exact Match 为 **44.5**，而闭卷 T5-11B 为 **34.5**，绝对提升 10 个点，约 **29% 相对提升**。论文还报告 RAG 生成的文本比纯参数基线更具体、更多样且更符合事实。 | 在知识密集型任务中，让模型“开卷回答”可以明显优于只依赖参数记忆。 |
| NAACL 2025 FRAMES | 在需要多步检索和推理的任务中，当时的强模型在无检索时准确率为 **0.40**，多步检索流水线达到 **0.66**，绝对提升 26 个点，相对提升约 **65%**。 | 当回答需要组合多份证据时，不能假设更大的模型自然就会给出可靠答案。 |
| EACL 2024 FACTOR | 研究在 Wiki、新闻和专业领域基准上发现，模型加入检索后的事实性评分提升；作者同时强调，专业领域需要自己的事实性基准。 | SpeakUp 不能只引用通用 QA 成果，必须建立英文技术面试评测集。 |
| Stanford AI Index 2026 | 报告引用的一项新准确性基准中，26 个主流模型的幻觉率范围为 **22%–94%**。该结果强烈依赖具体基准，不代表日常对话的通用幻觉率。 | 产品不应把“使用最新大模型”等同于“事实可靠”，仍需要证据约束和评测。 |

## 四、RAG 在 SpeakUp 中具体解决什么问题

### 4.1 生成更符合岗位和职级的问题

检索时先按岗位、职级、技术栈、面试阶段和难度筛选，再召回相关问题卡。模型负责把结构化题目转换为自然的英文提问，而不是临场随机编题。

### 4.2 根据用户回答进行有目的追问

知识库中的每道题不只存“题干”，还存考察目标、关键要点和追问规则。例如，用户提到“使用 Redis 解决并发问题”却没有说明一致性与失效策略时，系统应优先追问这个缺口，而不是随机切换到新题。

### 4.3 输出有证据的结构化反馈

反馈页不只显示“AI 建议”，而是展示：

- 本题考察的能力；
- 用户原回答中已覆盖的要点；
- 遗漏或证据不足的要点；
- 对应的 Rubric 和参考来源；
- 下一次回答可执行的改进动作。

### 4.4 支持知识快速更新

新增一个岗位或修正一条错误知识时，团队只需更新知识库并重建索引，无需等待基座模型升级或进行全量微调。

## 五、什么才能构成护城河

| 层级 | 通用“面试官人设” Agent | SpeakUp 的目标 |
|---|---|---|
| 提问 | 根据 Prompt 开放生成 | 根据岗位、职级和用户背景检索经审核的题目卡 |
| 追问 | 根据当前对话自由发挥 | 根据缺失要点、追问树和简历事实追问 |
| 评分 | 模型即兴判断 | 用版本化 Rubric 分维度判断，并引用用户原文 |
| 可解释性 | 无法说明依据 | 显示考察点、证据和知识来源 |
| 个性化 | 依赖当前上下文 | 连接已确认的简历快照、历史弱项和复练记录 |
| 进化方式 | 依赖平台升级模型 | 根据用户练习效果调整题目、难度和 Rubric |

**RAG 管线本身可以被复制，但高质量知识、评分标准和用户效果数据会随使用时间累积。**

## 六、知识库应该包含什么

### 6.1 面试题目卡

每条至少包含：

- `question_id`；
- 岗位、职级、技术栈和面试阶段；
- 英文主问题与可接受变体；
- 考察目标和难度；
- 参考要点，不是要求用户背诵的“唯一标准答案”；
- 追问触发条件和追问树；
- 评分 Rubric；
- 来源、审核状态、版本和更新时间。

### 6.2 英文表达知识

包含 STAR 结构、技术取舍、澄清问题、承认不确定性、量化成果等场景化表达范式，并标注适用场景和常见误用。

### 6.3 用户真实事实

简历、项目经历和量化结果必须来自用户确认过的结构化快照。系统可以帮助用户表达真实经历，但不应从公共知识库补全或编造用户没有做过的事情。

## 七、建议的产品与技术方案

```text
用户简历快照 + 目标岗位 + 当前回答
  -> 元数据过滤（岗位 / 职级 / 技术栈 / 场景）
  -> 关键词 + 向量混合检索
  -> 重排并过滤低相关证据
  -> 返回题目卡 / Rubric / 追问规则
  -> 模型生成提问或反馈
  -> 校验输出是否被用户原文和检索证据支持
  -> 展示可追溯结果
```

核心原则：

1. **先结构化过滤，后语义检索**：职级和岗位等硬条件不交给向量相似度猜测。
2. **没有证据就降级**：检索结果低于阈值时，明确说明证据不足，或回退到经审核的通用题，不强行生成专业结论。
3. **区分事实性与表达建议**：技术事实需要来源；行为面试和英文表达属于 Rubric 匹配，不宣称存在唯一正确答案。
4. **知识有版本**：保存内容版本和检索记录，使同一份报告可以复现。
5. **不用 RAG 替代确定性指标**：语速、停顿、填充词和重复等仍由 `SpeechMetricsEngine` 计算，不交给文本模型自由评分。
