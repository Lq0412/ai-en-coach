# [提案] 用户账户与个人中心功能范围

> 关联：[Issue #9](https://github.com/1024XEngineer/XE3-ESL/issues/9) · [最新在线原型](https://speakup-preview.oopwqsnyzxm.chatgpt.site/)

## 动机 / 用户故事

简历、面试记录、报告、错题和角色练习需要归属于同一个用户，否则无法形成跨会话的持续训练。与此同时，账户中心本身不是 SpeakUp 的差异化价值，首期不应把主要时间投入会员、订单和复杂资料体系。

典型用户故事：

- 作为一名多次使用 SpeakUp 的候选人，我希望再次进入后还能看到自己的简历、面试历史、报告和错题。
- 作为一名已完成练习的用户，我希望“我的”页能告诉我累计练习时长和近期进度，并快速进入下一次练习。
- 作为一名账户用户，我希望能修改基础资料、退出登录，并在不再使用时注销账户和删除个人数据。
- 作为一名处于 MS1 演示阶段的评审者，我希望能看到最终个人中心结构，但不会把会员和订单入口误判为本阶段已实现。

一句话概括：

**账户模块只负责身份、数据归属和继续练习；商业化与账户装饰不能阻塞面试主链路。**

## 目标用户

- 会重复使用产品，需要保存简历、面试和复练数据的用户。
- 使用模拟面试或自定义角色练习的同一账户用户。
- MS1 中使用固定演示身份完成全流程的评审者。

## 现有做法及其不足

| 方案 | 优点 | 主要问题 |
|---|---|---|
| 无账户，仅保存本地数据 | 开发快、打开即用 | 换设备或清理数据后无法继续，难支持长期复练 |
| 完整账户与商业化中心 | 功能完整、便于长期运营 | 登录、绑定、支付、订单和权益会显著扩大首期范围 |
| 最小账户与个人中心 | 能保存核心数据，范围可控 | 仍需处理隐私、退出和注销 |

两个月选择第三种方案；MS1 暂时使用固定演示用户。

## 本期范围与明确不做

### 两个月最终范围

#### 身份与数据归属

- 提供一种可用的登录/身份识别方式，具体技术在架构 Proposal 中确定。
- 用户只能访问属于自己的简历、面试、报告、错题和自定义角色数据。
- 退出登录后不继续展示受保护数据。
- 提供注销账户入口，并在确认前说明将删除的个人数据范围。

#### “我的”页

- 展示昵称或默认称呼。
- 展示累计对话时长、练习次数或近期练习进度；统计口径必须可解释。
- 提供个人简历、历史、设置的快捷入口。
- 昵称和头像允许轻量设置，但不作为 MS2 核心交付阻塞项。

#### 设置页

- 展示头像、昵称和已绑定身份的脱敏信息；原型中的手机号仅表示一种可能的登录身份。
- 提供隐私政策、服务条款、关于 SpeakUp、退出登录和注销账户入口。
- 如果最终认证方案不是手机号，页面应随架构方案调整，不保留无效手机号字段。

### Milestone 实现深度

| Milestone | 做到什么程度 |
|---|---|
| MS1 | 使用固定演示用户；原型展示“我的”、简历、设置和统计外壳；不实现真实注册登录 |
| MS2 | 支持一种身份识别方式；用户可跨会话访问 1 份简历、面试历史和报告 |
| MS3 | 保存错题和自定义角色；完成昵称、头像、退出登录和注销入口；补齐可解释的练习统计 |
| MS4 | 完成异常状态、权限校验、数据删除验证、隐私文案和可用性打磨 |

### 明确不做

- 不同时建设手机号、邮箱、第三方登录、账号绑定和找回等多套认证流程。
- 不做公开个人主页、等级、勋章、好友、关注、排行榜或社交关系。
- 不做实际会员订阅、权益计费、订单、支付、优惠券或退款流程。
- 不做企业组织、团队成员、管理员后台或多人协作。
- 不收集与面试和账户安全无关的性别、生日、地区、教育详情或社交账号。
- 不把昵称、头像或统计动画作为面试核心链路的发布门槛。

## 关键决策与依据

### 决策 1：采用一种最小身份方式

两个月内只选择一种认证方式，原因是核心问题是“数据属于谁、下次如何继续”，不是覆盖所有登录偏好。具体选择手机号、邮箱或第三方登录，需要结合部署环境和团队架构再决定；本 Proposal 不提前锁定技术实现。

### 决策 2：保留基础资料，但降低优先级

最新版原型已经展示昵称、头像和脱敏手机号。正式范围保留轻量昵称/头像，以保证个人中心完整；手机号仅随认证方式存在。它们安排在 MS3，不应阻塞 MS2 的面试闭环。

### 决策 3：会员和订单只作为原型外壳

最新版原型展示“权益中心”和“我的订单”，但当前没有完整权益、计费和订单流程。这两项不能被写入两个月研发验收：

- 若继续保留在演示原型，应标注“规划中”或点击后说明未开放。
- 在真实开发版本中，未实现前应隐藏，避免用户误解为可以付费。
- 后续若团队决定商业化，需要单独 Proposal 说明权益、价格、退款、支付和合规，不在本 Issue 扩展。

### 决策 4：练习统计只展示可解释指标

优先展示累计对话时长、完成练习数、待复练项等来自真实记录的指标。原型中的星星或奖励图标如果没有稳定规则，不进入产品验收，避免制造无法解释的成长分数。

## 基本概念与信息结构

| 概念 | 定义 |
|---|---|
| User Account | 唯一标识用户及其数据归属的账户身份 |
| Basic Profile | 昵称、头像，以及随认证方式存在的脱敏身份信息 |
| Practice Summary | 从真实会话聚合的练习次数、时长和待复练项 |
| User-owned Data | 简历、面试上下文、会话、报告、错题和私有角色 |
| Account Deletion | 用户发起并确认删除账户及其拥有数据的流程 |

```text
User Account
  ├─ Basic Profile
  ├─ Resumes
  ├─ Interview Sessions
  ├─ Reports / Retry Items
  └─ Custom Roles / Role Sessions
```

Token、认证协议、数据库表、对象存储权限和删除任务由架构 Proposal 定义。

- 相关页面：“我的”、个人简历、设置。

最新版原型对应关系：

- “未设置昵称”、头像、手机号：保留为基础资料外壳，真实字段随认证方案调整。
- 累计对话和练习数据：保留，但必须使用有定义的统计口径。
- 个人简历：两个月核心入口。
- 隐私政策、服务条款、关于、退出登录、注销账户：两个月最终范围。
- 权益中心、我的订单：不进入两个月研发验收；继续展示时必须标注未开放。

## 验收标准

### Proposal 定稿验收

- [ ] 团队确认采用一种最小身份方式，不并行建设多套登录。
- [ ] 团队确认昵称和头像为低优先级基础资料，手机号取决于认证方案。
- [ ] 团队确认会员、权益中心、订单和支付不进入两个月研发验收。
- [ ] 团队确认统计只使用来自真实练习、口径可解释的指标。
- [ ] 团队确认注销账户需要覆盖所有用户拥有的数据。

### 两个月产品行为验收

1. 已识别用户重新进入后，可以访问自己的简历、历史、报告、错题和私有角色。
2. 未登录或退出登录后，受保护的个人数据不继续展示。
3. “我的”页的累计时长、练习次数等数据能从真实会话记录复核。
4. 用户可以修改允许修改的基础资料，并看到脱敏身份信息。
5. 用户可以发起注销，看到数据删除范围并完成二次确认。
6. 产品中不存在可实际购买的会员、权益或订单流程；未开放入口不会误导用户。
