# 产品概念定义：AI 口语教练

> 状态：v0.2 草案
> 参考：1024XEngineer 产品思维先行原则（`reference/02-product-first.md`）
> 本文档回答"做什么、不做什么、为什么"，不讨论实现细节。

---

## 一、一句话定义

> 一个以**自由语音对话**为交互形式的 AI 口语教练——用自然聊天的方式帮用户在真实场景中开口说英语。首期以**技术面试**为主打场景，但内核不依赖单一场景，支持扩展为自定义 Agent。

**关键名词解释**：

| 词 | 本文档中的含义 |
|---|---|
| 自由语音对话 | 用户可以随时开口、随时打断、自然接话，不按固定脚本走 |
| Agent | 一个有特定角色人设、知识领域、评分标准和对话策略的 AI 教练实例。技术面试官是一个 Agent，留学面试官是另一个 Agent |
| 场景 | = Agent 配置。一个场景定义了一个 Agent 的四要素（角色、知识、评分、策略），不是指"面试"这个生活场景 |
| 可扩展 | 新增一个 Agent 不需要改动对话引擎、评分架构和学习记忆模块。未来支持用户自己定义 Agent 参数 |

## 二、产品定位：不是什么，是什么

### 不是

| 不是 | 为什么 |
|---|---|
| **不是通用聊天机器人** | 不做"跟 AI 随便聊英语"——市面上的豆包、ChatGPT 已经能做了，不构成差异化 |
| **不是题库型面试模拟器** | 不做"放一道题 → 用户答 → 打分 → 下一题"——那个不是面试，是考试。用户差评里 "AI 对话内容机械不灵活" 出现 102 次，题库式 = 机械式 |
| **不是英语课程平台** | 不教单词、不教语法、不做闯关——这些是 Duolingo 和流利说的事，5 人团队 2 个月不可能做出课程体系 |
| **不是串 API 的产品** | ASR → LLM → TTS 谁都会调，不构成壁垒。壁垒在评分诊断 + 学习记忆 |

### 是

| 是 | 含义 |
|---|---|
| **是一个教练，不是一个聊天搭子** | 聊天搭子只负责"聊得开心"，教练负责"让你变强"。每次对话后有诊断、有改进建议、有下一次练习的入口。如果用户只把它当情绪陪伴，它就退化成了豆包 |
| **是一个有目标的练习，不是一段聊天** | 参考 CialloClaw 的 Task-Centric 设计——用户进来有明确目标（"我要练习 STAR 法则回答行为面试题"）。每次练习 = 一个 task，可中断、可恢复、可比较"上次 vs 这次" |
| **是一个可配置的 Agent 引擎，不是一套固定场景** | 面试官是第一个 Agent 实例。产品内核（对话引擎 + 评分诊断 + 学习记忆）不感知当前 Agent 是什么场景 |

## 三、核心产品原则

> 参考 ByteMind 产品原则：原则不是口号，是做功能决策时的指南针。每条原则必须能在具体 trade-off 中给出答案。

| # | 原则 | 含义 | 用它做过的决策 |
|---|---|---|---|
| **1** | **教练优先，陪伴其次** | 用户回访的原动力是"我变强了"，不是"它陪我聊天很舒服" | 用户紧张时，AI 先追问后安慰——教练 push 你，陪伴才哄你 |
| **2** | **进步必须可感知** | 用户每次离开时要知道自己比上次强在哪里。不依赖一个总分来假装进步 | 反馈报告不写"78 分"，写"上次你的 STAR 结构缺失，这次三段式完整了。但数据支撑还弱，这是你下次要突破的" |
| **3** | **诊断不放裸分数** | 不给 p0 用户也看不懂的数字。每次反馈必须定位到具体句子、给出对比和改进方案 | 评分 UI 不出现总分数字。跨 session 对比用"这个维度：上次弱 → 这次中"这样的定性锚点 |
| **4** | **自由对话为形，任务目标为魂** | 用户感觉是自然聊天，但 AI 心里有评估框架。对话的目标是完成练习任务，不是无限聊下去 | 见本文第四章——这是最难的设计点，单独展开 |
| **5** | **Agent 不锁死内核** | 面试官 Agent 的需求不能写死在评分引擎里。评分维度、知识库、对话策略都按 Agent 可配置 | 面试要 STAR 法则评分，将来留学要逻辑完整性评分——两者都是"结构化表达"评分子项的不同 rubric，架构上只认"结构化表达"，不认"STAR" |

### 工程约束（不是产品原则，但同样重要）

| 约束 | 含义 |
|---|---|
| 语音延迟 < 2 秒 | 延迟优化不是性能优化，是功能需求。超过 2 秒体验崩了。技术选型时延迟权重高于准确率 |
| Web 先行，Gateway 预留 | 先做 Web 端，但前后端之间留 Gateway 层（JSON-RPC 2.0），后续加移动端后端不用改 |

## 四、核心设计：AI 怎么在"自由聊天"和"暗中评估"之间切换

> 这是产品最核心的体验风险。自由对话要求 AI 像真人一样自然接话，但教练要求 AI 心里有评分表。两者天然冲突。

### 三层对话状态

AI 教练在对话中始终处于三种状态之一：

```
┌─────────────────────────────────────────────────────┐
│  状态 1：开放对话（用户感觉：自由聊天）               │
│                                                     │
│  AI 像真人面试官一样自然接话、追问、互动               │
│  不打断用户的表达节奏                                │
│  暗中记录：卡点位置、结构完整性、关键表达              │
│                                                     │
│  触发切换 → 用户回答中出现明显结构缺陷                │
│           → 或连续两轮没有给出具体数据支撑             │
│           → 或用户主动说"我不知道怎么说了"            │
├─────────────────────────────────────────────────────┤
│  状态 2：定向追问（用户感觉：被追问了）               │
│                                                     │
│  AI 针对刚才的回答深入追问：                          │
│  "你刚才说这个系统扛住了 10k QPS——具体怎么测的？"     │
│  "你提到了团队协作，你具体负责哪一部分？"             │
│                                                     │
│  目的：验证用户是真正理解还是在背答案                  │
│  追问深度取决于 Agent 配置（技术面试深挖技术，        │
│    留学面试深挖动机和规划）                           │
│                                                     │
│  触发切换 → 追问 2-3 轮后                            │
│           → 或用户明显无法继续                        │
├─────────────────────────────────────────────────────┤
│  状态 3：收束推进（用户感觉：进入下一题了）           │
│                                                     │
│  AI 自然收束当前话题：                               │
│  "好的，我大概了解了。接下来我们聊聊..."              │
│  不给用户"被评判了"的感觉                            │
│  不在此刻给反馈——反馈是对话结束后的事                 │
└─────────────────────────────────────────────────────┘
```

**为什么反馈不在对话中给，而是对话后**：

- 面试中没有面试官会说"你这个 STAR 结构不对"
- 对话中给反馈 = 打断用户心流 = 破坏自由对话感
- 反馈放在诊断报告里，用户可以冷静下来再看

**对话中 AI 唯一会打断用户的情况**：

- 用户卡住超过 5 秒 → AI 给关键词提示（不是给答案）
- 用户明显跑题 → AI 自然拉回（"Sorry, let me refocus — can you tell me more about..."）

## 五、产品架构：内核 + Agent

```
┌──────────────────────────────────────────────┐
│              Agent 层（可扩展）                │
│                                                │
│  每个 Agent 定义四个要素：                       │
│                                                │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐      │
│  │ 技术面试官│ │ 留学面试官│ │ 用户自定义│      │
│  │ (MVP)    │ │ (v2)     │ │ (v2+)    │      │
│  └────┬─────┘ └────┬─────┘ └────┬─────┘      │
│       │             │            │            │
│       └─────────────┼────────────┘            │
│                     │                          │
│    Agent 四要素：    │                          │
│    · 角色人设       │  例：严格的 Google 面试官 │
│    · 知识库         │  例：system design 题库  │
│    · 评分 rubric    │  例：STAR 结构 + 数据支撑 │
│    · 追问策略       │  例：深挖技术细节        │
├─────────────────────┼──────────────────────────┤
│              内核层（通用，不感知 Agent）       │
│                                                │
│  ┌────────┐ ┌──────────┐ ┌────────┐           │
│  │ 出题引擎│ │ 对话引擎  │ │评分诊断 │           │
│  │        │ │          │ │        │           │
│  │· 从知识│ │· ASR     │ │· 语音  │           │
│  │  库选配│ │· LLM 对话│ │  切片  │           │
│  │  题目  │ │  (含3层  │ │· 多维  │           │
│  │· 根据用│ │  状态切换│ │  评估  │           │
│  │  户背景│ │· TTS     │ │· 对比  │           │
│  │  个性化│ │· 打断处理│ │  诊断  │           │
│  └────────┘ └──────────┘ └────────┘           │
│                                                │
│  ┌──────────────────────────────────────┐      │
│  │           学习记忆                    │      │
│  │  · 卡点追踪（哪句话卡住了）            │      │
│  │  · 进步锚点（上次 vs 这次的定性比较）  │      │
│  │  · 复习触发（下次优先安排薄弱项）      │      │
│  │  · 跨 session 进步证据               │      │
│  └──────────────────────────────────────┘      │
│                                                │
│  ┌──────────────────────────────────────┐      │
│  │         会话管理 (Task-Centric)        │      │
│  │  · 每次练习 = 一个 task               │      │
│  │  · 每轮问答 = 一个 step               │      │
│  │  · 可中断、可恢复、可回溯              │      │
│  └──────────────────────────────────────┘      │
└──────────────────────────────────────────────┘
```

### 内核层四个模块的职责边界

| 模块 | 负责 | 不负责 |
|---|---|---|
| **出题引擎** | 根据 Agent 的知识库 + 用户背景信息，选出或生成本次练习的题目和追问方向 | 不负责对话（那是对话引擎的事）、不评分 |
| **对话引擎** | 语音输入输出、LLM 对话生成、打断处理、三层状态切换 | 不负责"问什么题"（那是出题引擎的事）、不评分 |
| **评分诊断** | 会话结束后，根据 Agent 的评分 rubric，定位问题句子、生成对比诊断 | 不在对话进行中运行 |
| **学习记忆** | 记录卡点、追踪跨 session 进步、触发复习 | 不参与实时对话 |

### 这个架构回答的边界问题

- **什么场景能接入**：能用四要素（角色、知识、rubric、追问策略）描述的场景。面试、留学、职场会议、商务谈判都可以
- **什么场景不做**：需要视频、多 NPC 角色扮演、3D 环境的场景——那是另一个产品
- **内核不感知 Agent**：对话引擎不知道"这是面试还是留学"，只知道"当前角色是 X，评分 rubric 是 Y，知识库是 Z"
- **新增 Agent 不碰内核**：加一个"留学面试官"Agent = 新增一套四要素配置，不写一行内核代码

## 六、MVP 边界：什么是 v1，什么不是

### v1（两个月交付）

| 模块 | 范围内 | 范围外 | 为什么这么切 |
|---|---|---|---|
| **Agent** | 1 个：技术面试官（3 个子场景：自我介绍、行为面试、项目深挖） | 留学、职场、用户自定义 | 面试是付费意愿最强的场景（张的方向评估），3 个子场景覆盖 80% 面试内容 |
| **出题引擎** | 从知识库选配题目（不是 AI 生成）。题库 30-45 道人工整理的核心题 | AI 动态生成新题 | AI 生成题目质量不可控，MVP 先用人工整理保证质量。生成是 v2 能力 |
| **用户背景** | 用户选择：目标公司 + 岗位 + 经验年限 | 简历上传和解析 | 简历解析是独立功能模块，MVP 不做。3 个选择题已经够出题引擎个性化选配 |
| **交互** | Web 端自由语音对话 + 打断 | 移动端、微信 | 先做一个端验证，架构留 Gateway 后续加端 |
| **对话状态** | 完整三层状态切换（开放对话 → 定向追问 → 收束推进） | — | 这是核心体验，MVP 必须实现，不能简化 |
| **评分** | 结构化表达 + 语言质量，定位到句子 | 音素级发音、多裁判评分 | 结构化表达是面试最相关的维度（覃的 T1 结论）；音素纠错是 ELSA 的战场 |
| **诊断报告** | 对话后出。定性定位 + 对比示例 + 下次练习建议 | 实时反馈（对话中给） | 基于第四章的设计：对话中不打断，反馈放报告里 |
| **记忆** | 本次对话内上下文 + 卡点记录 + **两次练习间的进步锚点** | 跨 Agent 学习路径、长期统计面板 | 进步锚点（"上次→这次"对比）是最小闭环，必须做。跨 Agent 记忆等有多 Agent 再说 |
| **会话** | Task-Centric：中断恢复、练习历史列表 | — | 来自 `05-task-centric.md`，用户随时能回来继续 |

### 明确不做（v1 砍掉）

| 不做 | 理由 |
|---|---|
| ❌ 视频生成 | 交互形式是语音对话，不是视频。视频是另一个产品形态 |
| ❌ 音素级发音评分 | 面试官不在意 /θ/ 音标，在意你说的内容。发音纠错是 ELSA 的战场 |
| ❌ 多人角色扮演 | 面试是 1v1 |
| ❌ 移动端 App | Web 先验证，移动端是分发问题不是产品问题 |
| ❌ 自定义 Agent 编辑器 | 1 个 Agent 没东西可编辑，v2 有多个 Agent 后再做 |
| ❌ B 端企业版 | C 端还没验证，同时打两场仗必死 |
| ❌ 简历上传解析 | 解析质量不可控，3 个选择题够用了 |
| ❌ AI 实时生成题目 | 质量不可控。用人工整理题库 |

### 但要为 v2 预留的抽象（MVP 写代码时就要考虑）

| 预留 | 为什么现在就要考虑 |
|---|---|
| ✅ Agent 四要素可配置化 | 加一个新 Agent 应该是新增配置，不是改代码 |
| ✅ 评分 rubric 可替换 | 面试用 STAR，留学用逻辑完整性——代码里不写死维度名称 |
| ✅ 知识库 schema 通用化 | 题库结构不假定"一定是面试题"，字段抽象到"题目、参考答案要点、追问方向" |
| ✅ Gateway 协议层 | 前端和后端之间走 JSON-RPC 2.0，后续加移动端后端不用改 |

## 七、用户流程（MVP）

### 首次使用

```
进入首页
  │
  ├→ 新用户引导（简短的体验说明，< 30 秒能读完）
  │   "这是一个模拟真实面试的口语练习工具。
  │    你会面对一个 AI 面试官，用语音自然对话。
  │    练完后会有一份诊断报告，告诉你哪里需要改进。
  │    不用担心说错——说错才是练习的意义。"
  │
  ├→ 选择场景（v1 只有"技术面试"）
  │
  ├→ 快速配置（3 个选择题，不是表单）：
  │   · 目标公司类型：大厂（Google/Meta/...）/ 国内出海 / 创业公司 / 不限
  │   · 目标岗位：后端 / 前端 / 全栈 / 数据 / 其他
  │   · 经验年限：应届 / 1-3年 / 3-5年 / 5年+
  │
  └→ 系统从题库中选配 3-5 道核心题 + 追问策略
        └→ 进入练习
```

### 练习流程（含中断恢复）

```
练习开始
  │
  ├→ AI 面试官语音提问（第 1 题）
  │
  ├→ 用户语音回答
  │   ├── 正常回答 → AI 自然追问（状态 2，2-3 轮）
  │   ├── 用户卡住（>5 秒）→ AI 给关键词提示
  │   └── 用户跑题 → AI 自然拉回
  │
  ├→ AI 收束推进（状态 3）→ 进入第 2 题
  │
  ├→ ... 重复，直到全部题目完成 ...
  │
  ├→ [中断] 用户关闭页面 / 超时
  │   └→ 自动保存当前进度（第 X 题第 Y 轮追问）
  │     下次回来 → 练习历史 → 点"继续"→ 从中断处恢复
  │
  └→ 全部完成 → 诊断报告
```

### 诊断报告

```
不是一段文字，而是一个结构化反馈：

┌─────────────────────────────────────────┐
│  本次练习：技术面试（行为面试题 ×3）       │
│                                           │
│  📌 你最大的问题                           │
│  回答缺少具体数据支撑。3 道题中有 2 道用了  │
│  模糊表述（"我们做得挺好的""效果不错"）。   │
│                                           │
│  📍 具体定位                               │
│  第 1 题 "最骄傲的项目"                     │
│    你说："We improved the performance a    │
│           lot, it was very fast..."        │
│    问题：没有数字支撑。面试官不知道"a lot"   │
│          是多少、为什么技术决策是合理的。    │
│    建议："We reduced p99 latency from      │
│           200ms to 45ms by switching to..." │
│                                           │
│  第 2 题 "团队冲突"                         │
│    你说：STAR 结构完整 ✅                    │
│    问题：Situation 太长（45 秒），Action    │
│          太短（10 秒）。面试官想听你做了什么，│
│          而不是背景有多复杂。               │
│    建议：S 压缩到 2 句话，A 展开到具体行动   │
│                                           │
│  📈 和上次对比                             │
│    上次练习（7/8）：STAR 结构不完整          │
│    本次练习（7/9）：3 道题中 2 道 STAR 完整  │
│    → 结构力在进步。下次突破重点：数据支撑     │
│                                           │
│  🎯 下次练习建议                           │
│    系统将优先安排需要"数据支撑"的题目         │
└─────────────────────────────────────────┘
```

**为什么诊断报告这样设计**：

- "你最大的问题"——用户先看到结论，不需要读完才知道自己差在哪
- "具体定位"——引用用户原话 vs 建议版本，不是抽象的"语法需要改进"
- "和上次对比"——来自原则 2"进步必须可感知"。只用定性锚点，不给裸分
- "下次建议"——给用户回来的理由。报告不是终点，是下一次练习的起点

---

## 八、进步衡量：不给总分，怎么知道变强了

### 为什么不用总分

- 总分给人的感觉是"被评判了"而不是"知道怎么改"
- 78 分 vs 82 分的差异用户感受不到，但"上次 STAR 缺失，这次完整了"能感受到
- 黄的评论分析：136 条"评分机制不合理"差评——用户不信任一个看不懂的数字

### 用什么代替

```
不是横向对比（你和别人比多少分）
而是纵向锚点（你和你上次比，哪里变了）

锚点类型：
  维度锚点：STAR 结构 → 上次：缺失 │ 本次：2/3 题完整
  卡点锚点：上次卡在"不知道怎么用数据支撑观点"
           本次能说出 1 个具体数字
  行为锚点：上次追问到第 2 轮就崩了 │ 本次扛到第 3 轮

为什么这样设计：
  · 用户不需要理解"STAR 结构评分算法"
  · 用户只需要知道自己"上次不行 → 这次行了"
  · 每次只比上一次，不制造排行榜焦虑
```
