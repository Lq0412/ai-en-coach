# 用 Claude Code Skills 辅助团队研发规范落地——一个 5 人团队第一周的实践记录

每个团队都有开发规范。CONTRIBUTING.md、代码规范、分支策略、Issue 模板……写得再完善，也很难在开发过程中被持续遵循。

我们在参加七牛云 XEngineer 实训营时想了一个办法：把规范写成 Claude Code Skills——不是让人读的文档，而是 AI 在开发者主动调用时参考执行的规则。本文是我们的初步尝试记录。

## 1. 问题：规范文档为什么难以落地？

三个原因：

1. **时机不对。**写代码时不会专门翻 CONTRIBUTING.md，写完才想起来格式不对。
2. **记不住。**10 条规则，写到第 3 条就忘了前两条。
3. **没有后果。**没有约束。违反规范也不会影响开发流程，规范最终形同虚设。

核心矛盾是：规范存在文档里，但工作在编辑器里。两者不在同一个时空。

## 2. 尝试：把规范写成 Skill

Claude Code 的 Skill 机制允许我们把团队约定写成结构化提示词，放进 .claude/skills/ 目录。每个 Skill 同时定义了触发场景——开发者在对话中说"帮我创建个 Issue"，issue-standard 自动匹配触发；说"帮我提交代码"，pr-commit 自动生效。不需要刻意记命令，规范在干活的时候自然就带上了。

7 个流程 Skill——对应研发流程的关键节点：

3 个辅助 Skill——不对应特定流程节点：

|---|---|---|
| `ai-design-smell` | 识别 AI 生成 UI 的“模板味”（蓝紫渐变、卡片套娃等）并给出修复方向 | 评审或生成界面时 |
| `product-killshot` | 竞品深度分析：读代码 → 搜竞品 → 双视角分析 → 输出优化报告 | 产品决策前 |

## 3. 流程 Skill 的边界：能管什么、不能管什么

一套 Skill 不是一条自动流水线。它在一些环节有效，在另一些环节无能为力。我们把每个环节的边界交代清楚：

> 原文配图（请在微信公众号查看）

Skill 管的是格式、流程提醒和常见模式检查，不管业务判断。它把机械性的规范检查负担从人身上卸掉，但决策始终是人来做。

另一个需要交代的边界：Skill 不是自动化 hooks。它不会在 git commit 时自动执行、不会在 push 时自动检查——触发的前提是开发者在对话中提到了相关场景（或者显式输入 /skill-name），AI 再按 Skill 规则响应。

这意味着如果开发者全程不用 AI，规范就不会生效。真要把规范"长"在流程里，需要配合 git hooks、CI 检查等机制——Skill 只是其中一环。

## 4. 流程 Skill 的设计思路

以下按逻辑顺序展开：先讲全局框架（milestone-manager），再按研发流程从头到尾讲各节点 Skill。

### 1. milestone-manager：8 周 4 个里程碑（设计框架）

我们把 8 周实训营规划为 4 个里程碑——这是事前设计，目前只跑完了第一周：

```text
MS1 (W1-2)：方向成立吗？   → 产品决策 + 可运行骨架
MS2 (W3-4)：核心路径通吗？  → 核心功能可演示
MS3 (W5-6)：MVP 做完了吗？  → 全部功能可试用
MS4 (W7-8)：交付真实吗？    → 功能冻结 + 文档 + Tag
```

每个 MS 一个核心问题 + 一个可打 tag 的交付物。进度跟踪设置了预警规则：MS 过半但 Issue 完成率 <30% → 检查 scope 是否膨胀；成员 Issue 全部卡住 → 排查是否被阻塞。

实际效果待后续验证。

### 2. issue-standard：写好 Issue 比写好代码更重要

这是 issue-standard skill 自己的定位——Issue 写不清楚，代码写得再好也没人知道做完了没有。

我们的团队规则是：一个 Issue 一个人、一天能做完。Skill 把这个规则变成了拆分条件——当描述里出现两个以上技术域、或者预估超过 1 人天时，Skill 会提示拆分。同时 Skill 强制要求四段正文（背景/要做的事/验收标准/关联）和至少 2 个标签，没有挂 Milestone 的 Issue 会被标记为"没有排期"。

### 3. code-discipline：不猜、不兜底、不过度设计

AI 写代码有三个常见毛病：不清楚的字段名瞎猜一个、API 调失败静默返回假数据、需求没提的功能自己加戏。code-discipline 定了三条禁止线：不写猜测性代码（不知道就查源码或问人）、不写兜底代码（失败就明确报错）、不写冗余代码（不复制已有逻辑、不加多余的 try/catch）。

在此基础上，用"决策阶梯"拦住过度设计——每一级够用就停：

### 4. pr-commit：先对齐，再套模板

不强行统一格式，而是先对齐项目已有习惯——读 git log、读现有 PR、读 CONTRIBUTING.md 等项目约定文件。无约定时才用默认模板。这避免了一个常见问题：新规范跟项目历史风格打架。

默认规则包括：Conventional Commits 格式（feat(scope): subject）、分支命名（feat/xxx）、PR 四段描述（功能描述、实现思路、测试方式、相关 Issue）。

### 5. branch-commit：分支生命周期和提 PR 时机

branch-commit 与 pr-commit 分工明确：pr-commit 管 commit/PR 的格式，branch-commit 管什么时候提 PR、提之前检查什么。

Skill 定义了几个 AI 的触发行为：检测到未推送的 commit 时提醒开发者推送；开发者说"提 PR"时检查分支粒度（一个分支只做一个 Issue）、逐条过自检清单（commit 数量是否合理、跟 dev 有没有冲突、调试代码清理了没有、Issue 号写了没有），通过后创建 PR；合入后自动删除功能分支。

至于每天推送一次、分支从 dev 切出、WIP 规范这些——是 Skill 里写的团队约定，AI 只能提醒，不能强制。

### 6. pr-review：不只"看一遍代码"

pr-review 定义了五个必须逐条过的维度：技术正确性（边界处理、错误处理、对已有功能的副作用）、常见安全问题（硬编码密钥、输入校验、授权检查）、架构合理性（模块划分、接口边界、数据兼容性）、可维护性（新人 5 分钟能看懂吗）、AI 代码专项（是否有兜底数据、猜测字段、冗余异常处理）。

其中 AI 代码专项检查是针对性设计——询问提交者能否讲清每一段 AI 生成代码的逻辑、检查是否有 AI 常见糊弄行为。

Review 意见也定了规范：指出位置 + 说清问题 + 给方向不代写。用 [阻塞]/[建议]/[疑问] 前缀区分严重度。

合入条件五条：至少 1 人 Approved、自动化验证通过、已 rebase 无冲突、意见全部处理、改动范围与 Issue 一致。

### 7. release-tag：每个 MS 一个 tag

MS 结束时，dev 分支合入 main，随即在 main 上打 annotated tag。Tag 不可删除或移动——错了就补打新 tag（如 v0.2.1）。

Release Notes 必须包含四段：主要成果（写决策和产出不写过程）、已关闭 Issue、已知限制（必须写——知道什么还没做好比假装做好了更重要）、下个 MS 计划。

## 5. 一个设计细节：操作和数据的解耦

写 milestone-manager 时我们遇到一个问题：MS 的起止时间该写在哪？

一开始我们把它写在了 SKILL.md 里——「MS1：第 1-2 周，截止 7/19」。但仔细一想不对：起止时间是数据，不是操作规则。换一个项目、换一个时间安排，skill 的逻辑完全不用变，但日期全得改。

所以我们把两者分开了：

- SKILL.md 只管操作——怎么创建 MS、怎么关联 Issue、怎么跟踪进度。它是稳定的，跨项目复用。

- 数据文件（如 ms-config.json）管参数——MS 数量、每段起止时间、人数、Issue 配额。它随项目变。

```text
milestone-manager/
├── SKILL.md            ← 操作逻辑（稳定，可复用）
└── data/
    └── ms-config.json  ← 项目参数（随项目变）
```

这个分离带来的好处很直接：同一个 skill 可以原样搬到下一个项目，只换一份数据文件就行。skill 里不再出现「8 周」「7/19」「5 人」这种具体数字——它们都是参数。

推而广之，这个原则适用于所有流程 skill：skill 定义怎么做（how），数据文件定义做什么（what）。issue-standard 的标签体系、release-tag 的版本号规则，都存在类似的分离机会。

## 7. 现阶段的认识

把规范写成 Skill 解决了一个具体问题：规范不再需要人专门去记、去翻文档。开发者在终端里调 AI 帮忙时，规范作为 AI 的内置规则自然生效——前提是人主动调用了 AI。

但这远不是「规范自动执行」。Skill 的机制决定了它只在 AI 被调用时生效，而 git hooks 和 CI 检查才是在未调用 AI 时也能拦住问题的防线。我们后续的方向是把 Skill 和 CI 结合起来，让同一套规范在 AI 对话和自动化检查两个渠道都生效。

我们也还没验证这套体系能否撑过 8 周——当任务复杂度上升、交付压力变大时，规范还愿不愿意遵守，才是真正的考验。
