## 前端

- **React 18 + Vite**：团队零前端经验但 React 生态最大，几乎所有问题都有现成答案；后续上移动端 React → React Native 是唯一有积累的路径。Vite 开发服务器秒启动，CRA 已停止维护，Next.js 的 SSR 对 SPA 是额外负担。
- **Tailwind CSS**：零前端经验团队最大障碍是组织 CSS（BEM、命名、拆文件），Tailwind 直接消除这个问题。

## 后端

- **FastAPI**：核心链路"收音频→调AI API→分析→返回"每步都是 I/O 等待，同步框架会卡 worker。原生 async/await、WebSocket 一行装饰器、Pydantic 自动校验、`/docs` 即 API 文档。

## 通信协议选型：WebSocket

### 核心结论

**WebSocket 对当前阶段完全够用，不必焦虑**。原因有三：

1. **绝大多数 ASR/TTS 厂商的云 API 只支持 WebSocket**，你换不了更快的协议，除非自建
2. **传输层延迟差异很小**（WebSocket vs WebRTC < 200ms），而 LLM 推理延迟是 500ms-4s，**传输层只占总延迟的 < 5%**
3. **WebRTC 的优势在客户端侧**（降噪、回声消除、弱网自适应），不在"更快"，如果走 S2S API，厂商端已经帮你处理了协议

### 分阶段建议

| 链路 | 当前选择 | 建议 |
|---|---|---|
| 浏览器 ↔ FastAPI 后端 | **WebSocket** | ✅ 保持，MVP 不需要改 |
| FastAPI ↔ AI 厂商 API | 取决于厂商 | OpenAI→WebRTC; Qwen/豆包/GLM→WebSocket |

**MVP（Web 端）**：前后端保持 WebSocket，FastAPI 原生支持，实现成本最低。Web 用户通常在 WiFi 环境，TCP 队头阻塞问题不突出。

**产品化（移动端）**：如果发现弱网体验问题，引入 LiveKit 做 WebRTC 网关，不直接换协议。

### 延迟优化优先级

如果真要追求极致低延迟，按这个优先级优化：

| 优先级 | 优化方向 | 预期收益 | 实现成本 |
|---|---|---|---|
| **P0** | 选 S2S 端到端方案，跳过 ASR→LLM→TTS 串行 | -500~1000ms | 低 |
| **P1** | LLM 流式输出 + TTS 并行生成 | -300~600ms | 中 |
| **P2** | 客户端预处理（VAD提前截断静音、降噪） | -100~200ms | 低 |
| **P3** | 部署边缘节点，缩短网络 RTT | -50~150ms | 高 |
| **P4** | 前后端之间换 WebRTC | -50~100ms | 中 |

> **核心原则：先把架构选对，再抠传输层。** 选 S2S 比换协议重要 10 倍。

## AI 模型层

### 国产化实时语音方案（UniSpeaking）

**一句话方案**：用 **Qwen-Omni S2S** 做主链路（开转写拿文本），**MiniMax TTS** 做备选发音，Web MVP 用 WebSocket 直连，移动端阶段引入 LiveKit 做 WebRTC 桥接。

### 为什么 Qwen-Omni 是首选

国产 S2S 对比：

| 国产 S2S 候选 | 延迟 | 能拿文本吗 | API 成熟度 | 结论 |
|---|---|---|---|---|
| **Qwen-Omni (3.5 Flash Realtime)** | <150ms | ✅ `enable_input_audio_transcription=True` 即可 | GA，Python SDK 完善 | **首选** |
| 豆包 Seeduplex | 300-500ms | 未确认 | 2026年6月才开邀测 | 需要申请，稳定性存疑 |
| GLM-4-Voice | ~300ms | 未确认 | 半双工 | 体验不如全双工 |
| Step-Audio 2 | ~80ms | 开源自建 | 需要自己部署 | 延迟最优但要运维 |

**Qwen-Omni 的关键能力**：开启转写后在同一个 WebSocket 连接里同时拿到：
- 用户语音 → 自动转写文本（用于课后分析）
- AI 回复 → 同时输出语音流 + 文本（用于反馈展示）

一个 WebSocket 连接，同时得到：**低延迟语音 + 用户说了什么文本 + AI 回复了什么文本**。这解决了 S2S "黑盒拿不到文本" 的问题。

### 架构对比：为什么选 S2S 不选纯管线

管线架构可控性更强，但国产方案存在一个具体问题：**LLM 和 TTS 之间多一次网络往返**。

```
管线：ASR(WS) → 等出文本 → LLM(REST SSE) → 等出token → TTS(WS) → 等出音频
      一次WS建连   一次HTTP建连   一次WS建连
      三次网络往返，三次建连开销
      
Qwen-Omni S2S：一次 WS 建连，音频进，音频+文本出
```

在国内外网环境下，这个差异会被放大——每多一次建连就多一次 DNS 解析 + TCP 握手 + TLS，可能多 50-150ms。

### 端到端语音大模型：Qwen-Omni 3.5 Flash Realtime（林锵）

用于**自由对话场景**（面试前后的闲聊暖场、开放式英语对话练习），端到端模型原生支持全双工、随时打断、保留语气情绪，交互更自然。

| 模型 | 英文 | 延迟 | 全双工 | 获取文本 | 可访问性 |
|------|:---:|------|:---:|:---:|------|
| **Qwen-Omni 3.5 Flash Realtime** | 4星 | <150ms（厂商宣称） | 全双工 | ✅ 同时输出 | 直接开通 |
| GLM-4-Voice | 4星 | ~300ms | 半双工 | 未确认 | 直接开通 |
| GPT-Realtime-2 | 5星 | 232-320ms | 全双工 | ✅ | 海外，需要魔法 |
| Gemini Flash Live | 5星 | 300-500ms | 全双工 | ✅ | 海外，需要魔法 |
| 豆包 Seeduplex | 4星 | 300-500ms | 全双工 | 未确认 | 需要邀测申请 |

**结论**：Qwen-Omni 是目前国产 S2S 中**唯一同时满足低延迟 + 输出文本 + 直接可用**的方案，作为首选。GLM-4-Voice 备选。

### 级联架构大模型：Qwen3.5-Flash（智铭威）— 结构化面试对话

用于**结构化面试场景**（自我介绍、行为面试、项目深挖、技术问答），ASR 转写后由 LLM 驱动追问策略和结构化反馈。

| 维度 | Qwen3.5-Flash | DeepSeek V3 | 豆包2.1-Turbo | Kimi K2 |
|------|:---:|:---:|:---:|:---:|
| TTFT | 300-500ms | 300-500ms | 200-400ms | 400-600ms |
| MMLU | **89.7%** | — | — | 88.2% |
| 单次成本(30min) | **14.4元（恒定）** | 14-28元（峰谷） | ~39元 | ~150元 |
| 上下文 | 256K | 1M | 128-256K | 256K |
| Function Calling | 强 | 强 | 强 | 强 |
| API稳定性 | **阿里云企业级** | 中（预览版） | 火山引擎 | 中 |
| 合规 | 已备案 | 已备案 | 已备案 | 已备案 |

**结论**：延迟、英文能力、定价、Function Calling、稳定性、合规六个维度均领先。DeepSeek 峰谷定价不确定，Kimi 成本过高。

### ASR：Qwen 3.5 ASR Flash Realtime（黄天宇）

调研 30+ 款模型。结论：国产 SOTA，中英混说强，先 API 接入验证再逐步私有化。发音评测不能依赖普通 ASR，后续需单独评估讯飞/驰声/Azure。

作为**管线备选链路**的 ASR 环节：当需要更精细的文本控制（如特定教学策略）时，切换到管线架构。

### TTS：MiniMax speech-2.8-turbo（覃迦迎）

调研 15+ 款方案。优先国内：

| 优先级 | 方案 | 定位 |
|:---:|------|------|
| 首选 | **MiniMax speech-2.8-turbo** | MVP 主方案，流式低延迟，价格稳定 |
| 备选1 | MiMo TTS | 限时免费，性价比高 |
| 备选2 | CosyVoice / 百炼 | 中英混合+声音克隆 |
| 备选3 | Azure Speech | 稳定国际 fallback |
| 储备 | Inworld / ElevenLabs | 海外/精品音色阶段 |

## 延迟预期

### S2S 主链路（Qwen-Omni）

```
用户开始说话
    │
    ├─ 说话中...（Qwen-Omni 已在接收，simultaneous processing）
    │
用户说完最后一个词
    │
    ├─ VAD 判停（enable_turn_detection）       ~100-200ms
    ├─ Omni 模型推理 + 生成语音                 ~100-200ms（阿里宣称<150ms）
    ├─ 网络往返                                  ~30-50ms（国内阿里云）
    │
    ▼
用户听到 AI 首句语音：                          ~250-450ms ✅
```

### 管线备选链路

```
用户说完 → ASR finalize(100ms) → LLM首token(300ms) → TTS首块(150ms) + 网络(50ms)
= ~600ms
```

### 竞品对比

| 方案 | 延迟 | 备注 |
|---|---|---|
| Speak (OpenAI S2S) | ~300-500ms | 海外用户，Cloudflare边缘节点 |
| 可栗口语 (豆包) | ~300-500ms | 国内用户，火山引擎 |
| **UniSpeaking (Qwen-Omni)** | **~250-450ms** | 国内用户，阿里云，同区域 |
| UniSpeaking (管线备选) | ~500-800ms | DeepSeek + MiniMax |

考虑到国内网络环境（服务器在阿里云/火山引擎同区域部署），实际延迟会比海外方案更低。

## 移动端阶段：LiveKit WebRTC 网关

### 触发条件

MVP 阶段 Web 端用 WebSocket 够用。出现以下任一信号时，引入 WebRTC 网关：
- 移动端 App 开始开发
- Web 端用户反馈"地铁上/外面用的时候经常卡"
- 用户量上来了，发现在非 WiFi 环境下的会话完成率显著低于 WiFi

### 架构

```
移动端 App                           阿里云 ECS
─────────                          ─────────────
│         │  WebRTC (UDP/Opus)    │           │  WebSocket   ┌──────────┐
│ LiveKit │ ───────────────────→ │  LiveKit  │ ──────────→ │ Qwen-Omni│
│  SDK    │ ←─────────────────── │  Server   │ ←────────── │ (阿里云)  │
│         │                       │           │             └──────────┘
└─────────┘                       └──────────┘
                                   Opus ↔ PCM 转码在这里做
```

**为什么选 LiveKit**：
- 开源，Go 写的，一个二进制文件部署
- 自带 WebRTC 的 STUN/TURN、房间管理、录音
- 有 React Native SDK，移动端直接接入
- 国内 ECS 部署延迟可控
- 比自建 Janus/mediasoup 省 1-2 周开发时间

**成本**：一台 4C8G 阿里云 ECS 跑 LiveKit，约 ¥300-500/月，支撑几百并发没问题。

## 数据层

- **PostgreSQL + pgvector**：关系+JSON+向量三合一。AI 结果先存 `jsonb`，后续拆字段。pgvector 预留 RAG 空间。
- **单主库策略**：PostgreSQL 做主存储，对象存储放音频/图片/报告。
- **会话记录**：结构化主表（sessions/turns）+ JSON 扩展字段（asr_raw / pronunciation_detail / llm_feedback）。

## 架构决策：双轨并行

MVP 阶段**级联与端到端同时接入**，按场景分工：

| 场景 | 架构 | 模型链路 | 特点 |
|------|------|------|------|
| **结构化面试对话** | 级联 ASR+LLM+TTS | Qwen 3.5 ASR → Qwen3.5-Flash → MiniMax TTS | 可控性高、可接入发音评测、追问策略精确 |
| **自由通话** | 端到端语音大模型 | Qwen-Omni S2S | 全双工、可打断、保留语气情绪、交互自然 |

### 为什么不用一套架构覆盖所有场景

| 维度 | 级联 | 端到端 |
|------|------|------|
| 延迟 | 500-800ms | **250-450ms** |
| 交互 | 半双工 | **全双工，可打断** |
| 副语言 | 文本中转丢失 | **原生保留语气情绪** |
| 可控性 | **高，每步独立调试** | 低，黑盒 |
| 发音评测 | **可接入专业引擎** | 需额外集成 |
| 适合场景 | **结构化面试（需精确追问+评分）** | **自由对话（需自然流畅）** |

**结构化面试**需要精确的追问策略、结构化评分和发音评测，级联架构每一步可独立调试，更适合这个场景。**自由通话**追求自然流畅的对话体验，端到端模型的全双工和情绪保留优势明显。两条链路在后端统一封装，共享会话上下文和用户数据。

## MVP 开发路径（4 周）

```
Week 1: 搭通 Qwen-Omni 主链路
  □ FastAPI WebSocket endpoint
  □ Qwen-Omni Realtime Python SDK 接入（百炼）
  □ 配置 enable_input_audio_transcription
  □ 前端 WebSocket 音频收发
  □ 验证：浏览器说英语 → AI 英语回复 → 浏览器播放

Week 2: 对话体验打通
  □ System prompt 调教（回复长度、难度、追问策略、隐性纠错）
  □ 前端状态机：录音中 / 识别中 / 思考中 / 播放中
  □ 打断逻辑：前端音量检测 → interrupt 信号 → 停止播放
  □ 用户转写文本展示（可选显示）
  □ 验证：3-5 分钟自由对话无卡顿

Week 3: 课后反馈 + 管线备选
  □ 对话记录存储（转写文本 + AI 回复文本）
  □ 基础课后反馈：开口时长、说了多少轮、关键表达提示
  □ 搭建管线备选链路（Qwen ASR → DeepSeek → MiniMax）
  □ 策略路由：默认 S2S，特定场景切管线
  □ 验证：结束对话后看到反馈

Week 4: 打磨 + 内测
  □ 前端 UI 打磨（通话式界面）
  □ 延迟压测（P50/P95）
  □ 打断成功率测试
  □ 内部 3-5 人真实使用
  □ 验证：北极星指标"有效开口时长"可测量
```

## 成本估算

### MVP 阶段（假设 10 人内测，每人每天 10 分钟）

| 项目 | 月消耗 | 月成本 |
|---|---|---|
| Qwen-Omni API（音频输入） | ~500 万 token | ~¥400 |
| Qwen-Omni API（音频输出） | ~300 万 token | ~¥250 |
| 阿里云 ECS（2C4G） | 1 台 | ~¥200 |
| **合计** | | **~¥850/月** |

### 上线后（假设 100 DAU，每人 10 分钟）

| 项目 | 月消耗 | 月成本 |
|---|---|---|
| Qwen-Omni API | ~3 亿 token | ~¥2-3 万 |
| 备选管线（DeepSeek + MiniMax） | 溢出流量 | ~¥0.2-0.5 万 |
| 阿里云 ECS × 2 | 2 台 | ~¥500 |
| LiveKit Server（移动端后） | 1 台 ECS | ~¥400 |
| **合计** | | **~¥2.5-3.5 万/月** |

### 降本路径

- Qwen-Omni 模型也在持续降价——阿里云的通义系列历史上降价幅度很大
- 如果后续用户量大，可考虑自建 Qwen3-ASR + Qwen LLM + CosyVoice TTS，全部开源免费
- MiniMax speech-2.8-turbo 比 HD 便宜近一半（¥2 vs ¥3.5/万字符），实时对话场景 Turbo 够用

## 最终一句话决策

> **主链路：Qwen-Omni S2S（开转写拿文本）→ WebSocket → 浏览器。备选管线：Qwen ASR + DeepSeek + MiniMax TTS。移动端阶段：加一层 LiveKit WebRTC 网关。国产全链路，延迟 250-450ms，文本可控，成本可控。**
