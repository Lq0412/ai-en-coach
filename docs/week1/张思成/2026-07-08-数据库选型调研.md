## 一、调研目标

本调研需要回答 4 个问题：

1. 这个项目的主数据库应该选什么？
2. 会话记录、评分结果、课后总结这类数据应该怎么存？
3. 是否需要提前考虑向量检索与 RAG？
4. 两个月 MVP 阶段，怎样选型最稳、最省事、最容易上线？

先给结论：

> **推荐方案：PostgreSQL 作为主数据库，启用 `pgvector` 作为向量扩展；音频、场景图、报告文件放对象存储；Redis 作为可选缓存，不作为主库。**

---

## 二、项目的数据特点

AI 口语陪练不是一个只有用户表和订单表的普通应用，它的数据结构比较特殊，主要包括以下几类：

### 2.1 强关系型数据

- 用户信息
- 课程 / 场景
- 一次训练会话（session）
- 一次会话中的多轮对话（turn）
- 会员订阅、配额、权限

这类数据有明确主键、外键、时间顺序和统计需求，适合关系型数据库。

### 2.2 半结构化数据

- ASR 原始识别结果
- 发音评分细项
- 表达纠错结果
- LLM 反馈 JSON
- 场景配置
- NPC 人设与任务配置

这类数据字段不完全稳定，后续很可能迭代变化，适合存为 JSON，但又需要保留可查询能力。

### 2.3 检索型数据

- 面试题库 / 场景话术库
- 用户历史错误聚类
- 个性化复习推荐
- RAG 知识块与 embedding

这类数据未来很可能需要向量检索，因此数据库最好不要把“向量能力”完全排除在外。

### 2.4 大文件数据

- 用户语音片段
- AI 回复音频
- 场景图片 / 全景图
- 训练报告导出文件

这类数据不适合直接进数据库，应该放对象存储，数据库只保存元数据和 URL。

## 三、选型原则

### 3.1 MVP 阶段优先级

对于 5 人团队、2 个月周期的项目，数据库选型优先级应该是：

1. **能稳妥支撑核心业务**
2. **学习成本和运维成本低**
3. **后续增加 RAG / 推荐 / 统计时不容易推倒重来**
4. **先用一套主库跑通，不轻易拆多套系统**

### 3.2 不追求“最强”，追求“最合适”

我们现在不需要为了未来几百万用户做过度设计，也不适合一上来就上：

- 独立向量数据库
- 独立图数据库
- 复杂分库分表
- 事件溯源架构

MVP 阶段更适合**一套关系型主库 + 少量扩展能力**。

## 四、候选方案对比

### 4.1 方案一：MySQL

#### 优点

- 团队常见，学习门槛低
- 运维资料多，部署简单
- 做传统业务表没有问题

#### 缺点

- 对半结构化数据支持不如 PostgreSQL 灵活
- 如果后续要做向量检索，路线不如 PostgreSQL 自然
- 复杂统计、文本检索、扩展能力整体偏保守

#### 适合度判断

- 如果项目只是“用户 + 订单 + 课程”这类标准后台，MySQL 足够
- 但对于 AI 口语陪练这种**同时有关系数据、JSON 数据和向量数据**的项目，MySQL 不是最优解

### 4.2 方案二：PostgreSQL

#### 优点

- 关系型能力强，事务、约束、统计都成熟
- 官方文档明确推荐大多数应用优先使用 `jsonb`，更适合存半结构化结果
- 扩展生态成熟，后续可以直接启用 `pgvector`
- 适合在一套主库里同时管理业务数据、JSON 数据和知识库元数据

#### 缺点

- 团队如果之前只用过 MySQL，初期需要一点适应成本
- 索引和查询优化需要基本的 PostgreSQL 经验

#### 适合度判断

- 对本项目非常合适，是目前最平衡的主库方案

### 4.3 方案三：MongoDB

#### 优点

- 存 JSON 文档很自然
- 场景配置、对话原始结果这类数据看起来很好塞进去

#### 缺点

- 对复杂关系和统计查询不如关系型数据库顺手
- session / turn / 用户 / 订阅 / 报表这类核心业务最终还是强关系型
- 如果后面又补一套关系库，系统复杂度会提高

#### 适合度判断

- 适合内容管理、日志类、文档类业务
- 不适合作为这个项目的唯一主库

### 4.4 方案四：Firebase / 纯 BaaS 文档库

#### 优点

- 上手快
- 实时同步体验好

#### 缺点

- 复杂查询、统计报表、关系建模不舒服
- 后续做知识库、向量检索、复杂权限时不自然
- 容易在 MVP 后期暴露结构性限制

#### 适合度判断

- 适合轻社交、轻协作、小工具
- 不适合作为 AI 口语训练产品的长期主库

## 五、为什么推荐 PostgreSQL

### 5.1 同时满足三类核心需求

PostgreSQL 最大的好处是，它能在一套库里兼顾：

- 关系型业务数据
- 半结构化 JSON 数据
- 向量检索扩展

这对 AI 口语陪练项目特别重要。

### 5.2 `jsonb` 很适合装 AI 结果

PostgreSQL 官方文档中对 JSON 类型的说明很明确：`json` 保存原始输入文本，`jsonb` 会转换为二进制格式，处理更快、支持索引，而且**大多数应用应该优先使用 `jsonb`**。

这对我们非常有用，因为很多 AI 结果不适合早早拆成 30 个字段，例如：

- 词级发音评分
- ASR 候选结果
- 语法纠错细项
- 课后总结结构体

这些内容可以先存 `jsonb`，后续稳定后再逐步拆字段。

### 5.3 `pgvector` 让后续 RAG 更顺

`pgvector` 是 PostgreSQL 上非常成熟的向量扩展。它支持：

- 精确向量检索
- `HNSW` 近似索引
- `IVFFlat` 近似索引

这意味着你们后面如果要做：

- 面试题库检索
- 场景话术召回
- 用户相似错误聚类
- 个性化复习推荐

都可以先直接在 PostgreSQL 内完成，不必第一阶段就引入独立向量数据库。

### 5.4 最适合 MVP 的“单主库策略”

MVP 阶段最怕的是：

- 主库用 MySQL
- 向量检索再上单独向量库
- 文件再上对象存储
- 状态再上 Redis

这样一来，5 人团队会在两个月里同时维护四套系统，风险很大。

更合理的策略是：

- 主库：PostgreSQL
- 向量：pgvector
- 文件：对象存储
- Redis：有明显需求时再加

这样主干最清晰。

## 六、推荐的数据库架构

### 6.1 主体架构

```text
App / RN 前端
    ↓
FastAPI 服务
    ↓
PostgreSQL（主库）
    ├─ 用户 / 订阅 / 场景 / 会话 / 对话轮次
    ├─ 评分结果 / 纠错结果 / 课后总结
    ├─ jsonb 原始 AI 输出
    └─ pgvector 知识块向量

对象存储
    ├─ 用户音频
    ├─ AI 音频
    ├─ 场景图片
    └─ 导出报告

Redis（可选）
    ├─ 会话状态缓存
    ├─ 限流
    └─ 异步任务队列
```

### 6.2 哪些东西不应该进数据库

不建议直接把以下内容存进主库大字段：

- 原始音频二进制
- 场景图片二进制
- 长时缓存的 TTS 音频
- 大体积导出文件

原因很简单：

- 占空间快
- 备份慢
- 查询慢
- 数据库和文件生命周期不一致

正确做法是：

- 文件存对象存储
- 数据库存文件 ID、URL、时长、大小、状态

### 6.3 会话记录怎么存

推荐拆成两层：

#### 第一层：结构化主表

- `conversation_sessions`
- `conversation_turns`

这样便于：

- 统计轮次
- 统计时长
- 生成学习记录
- 做分页与查询

#### 第二层：原始结果 JSON

在 `conversation_turns` 或附属表中增加：

- `asr_raw jsonb`
- `pronunciation_detail jsonb`
- `grammar_feedback jsonb`
- `llm_feedback jsonb`

这样既灵活又不容易把表结构设计死。

## 七、是否需要向量检索

### 7.1 结论

**不是 Day 1 必须，但强烈建议主库提前选择能兼容向量检索的路线。**

### 7.2 为什么

如果产品后续有以下任何一个功能，就会很适合 embedding / 向量检索：

- 面试题库或场景知识库检索
- 根据用户发音问题推荐相似训练内容
- 根据简历 / 场景召回最相关问题
- 根据历史错误做个性化复盘

这些功能不一定第一周就做，但非常可能在 MVP 的第二阶段出现。

### 7.3 MVP 阶段建议

- 主库直接选 PostgreSQL
- 开启 `pgvector`
- 第一阶段即使先不用向量检索，也把路线留出来

这样后面想加 RAG，不用迁库。

---

## 八、Redis 要不要上

### 8.1 结论

**Redis 不是一开始必须，但建议预留。**

### 8.2 适合 Redis 的场景

- 用户会话状态缓存
- WebSocket 房间 / 瞬时状态
- 限流
- 热点场景缓存
- 异步任务队列

### 8.3 不适合 Redis 的场景

- 永久业务数据
- 会话历史主存储
- 训练记录归档

所以 Redis 的定位应该是：

> **辅助组件，不是主数据库。**

---

## 九、托管方案建议

### 9.1 首选：托管 PostgreSQL

MVP 阶段建议优先使用托管 PostgreSQL，而不是自己从第一天开始维护数据库服务器。

可选方向：

- Supabase Postgres
- Neon Postgres
- 云厂商托管 PostgreSQL

### 9.2 为什么托管更合适

- 节省运维时间
- 默认有备份能力
- 更适合学生团队和短周期项目
- 容易快速上线 Demo 和 MVP

如果后面你们需要结合对象存储，也可以直接配合云存储服务使用。

---

## 十、对 MVP 的影响

### 如果选 PostgreSQL

你们在两个月内能更稳地完成：

- 用户体系
- 场景训练记录
- 会话与对话历史
- 评分与复盘
- 后续轻量 RAG

并且中途不太需要推翻数据库架构。

### 如果继续选 MySQL

短期也能做，但可能很快遇到这些问题：

- JSON 数据存得不够舒服
- 想加向量检索时要补第二套系统
- 查询和统计逻辑开始变得别扭

所以 MySQL 不是不能用，而是**没有 PostgreSQL 顺手**。

---

## 十一、最终推荐

### 主推荐

**PostgreSQL + pgvector + 对象存储**

### 备选

**PostgreSQL + 对象存储**

说明：

- 如果第一阶段确认完全不做 RAG，可以先只上 PostgreSQL
- 但仍建议保留后续启用 `pgvector` 的空间

### 不推荐作为主线方案

- MySQL 作为长期主库
- MongoDB 作为唯一主库
- Firebase / 纯文档库方案
- 一开始就拆独立向量数据库

---

## 十二、建议的后续动作

1. 技术栈文档中的数据库默认值从 `MySQL` 调整为 `PostgreSQL`
2. 数据库设计时，先按“结构化主表 + jsonb 扩展字段”建模
3. 音频、图片、报告文件统一走对象存储，不进主库
4. 如果 Week 2 明确要做题库 / 场景知识库，就同步把 `pgvector` 纳入开发计划
5. Redis 先不强制纳入 MVP，等实时链路和异步任务明确后再补
