# 国内外端到端语音大模型调研报告——AI 英语口语陪练项目选型

> 部分内容由豆包生成

## 1. 调研背景与目标

随着大语言模型与语音技术的快速融合，端到端语音大模型正在取代传统"ASR + LLM + TTS"三级管道架构，成为 AI 语音交互的新一代技术底座。本报告针对 AI 英语口语陪练项目的实际需求，系统调研国内外主流端到端语音大模型的技术能力、产品成熟度、API 可用性与成本，为项目技术选型提供决策依据。

- **调研范围**：覆盖 OpenAI、Google、微软等海外厂商，以及字节跳动、智谱 AI、阿里巴巴、科大讯飞、小米、阶跃星辰等国内厂商的端到端语音大模型产品，包含闭源 API 与开源模型两类。
- **核心评估场景**：AI 英语口语陪练，重点考察英文语音识别准确率、语音自然度、对话延迟、打断能力、多场景适配性与综合成本。

## 2. 端到端语音大模型技术概述

### 2.1 传统架构与端到端架构对比

传统语音对话系统采用三级管道架构——ASR 将语音转为文本 → LLM 生成文本回复 → TTS 将文本转为语音。这种架构存在三大固有问题：一是多级处理累积延迟高，通常在 800ms 以上；二是文本中转过程丢失语气、停顿、情绪等副语言信息；三是模块间误差传递，ASR 识别错误会直接影响 LLM 理解质量。

端到端语音大模型采用统一的神经网络直接处理音频输入并生成音频输出，中间不经过显式的文本转写环节。模型通过语音 tokenizer 将连续音频离散化为 token 序列，再由语言模型进行理解与生成，最后通过解码器还原为语音。这一架构实现了声学特征与语义理解的联合建模，从根本上解决了传统管道的信息损失与延迟问题。

| 对比维度 | 传统 ASR+LLM+TTS | 端到端语音大模型 |
|:---------|:-----------------|:-----------------|
| 处理链路 | 三级独立模块串联 | 单一模型端到端处理 |
| 典型延迟 | 800ms–1500ms | 200ms–500ms |
| 副语言信息 | 文本中转后大量丢失 | 原生保留语气、情绪、停顿 |
| 错误传播 | ASR 错误向下游传递 | 联合建模，误差整体优化 |
| 交互模式 | 半双工，轮次式对话 | 支持全双工，可随时打断 |
| 部署复杂度 | 需维护三套系统 | 单一模型，运维简化 |

### 2.2 核心技术特征

**语音 Tokenizer 技术**。端到端模型的核心组件是语音 tokenizer，负责将连续音频波形转换为离散的 token 序列。主流方案采用残差矢量量化（RVQ）技术，以 12.5Hz–25Hz 的频率提取音频 token，在压缩率与信息保留之间取得平衡。例如 GLM-4-Voice 使用 12.5Hz 帧率的单码本监督语音 tokenizer，小米 MiMo-Audio 则采用 8 层 RVQ 实现 25Hz 高精度 token 生成。

**全双工交互架构**。2026 年的新一代端到端语音模型普遍升级为全双工架构，支持模型"边听边说"，用户可以随时打断 AI，系统能够自然承接上下文而不出现逻辑断层。代表产品包括 OpenAI 的 GPT-Live、字节跳动的 Seeduplex 全双工语音大模型。全双工架构大幅提升了对话的自然感，使 AI 交互更接近真人对话。

**情感与风格表达**。端到端模型直接在语音空间建模，能够生成更自然的语调、节奏与情感表达。多数模型支持通过指令调整语速、语调、情感倾向，部分模型还支持多语言无缝切换与方言识别，为英语口语陪练场景提供了丰富的表现力。

### 2.3 发展现状与趋势

2024 年至 2026 年是端到端语音大模型的快速爆发期。OpenAI 于 2024 年 5 月发布 GPT-4o，首次将端到端语音能力带入主流视野；2026 年以来，技术迭代明显加速，全双工架构成为旗舰产品标配，开源模型也快速追赶，小米 MiMo-Audio、Step-Audio 2 等开源模型在部分评测中已接近闭源产品水平。

从应用趋势看，端到端语音大模型正在从"能用"向"好用"演进，核心竞争点从单纯的识别准确率转向对话自然度、交互流畅度与场景化能力。语音 Agent 化是另一重要趋势，模型开始支持工具调用、函数执行等能力，使语音交互从简单问答升级为任务执行。

## 3. 国外主流端到端语音大模型

### 3.1 OpenAI GPT-4o Realtime / GPT-Live

OpenAI 是端到端语音大模型的开创者，2024 年 5 月发布的 GPT-4o 首次实现了文本、视觉、音频的统一端到端建模。语音模式下最低响应延迟 232 毫秒，平均约 320 毫秒，达到人类对话级别的响应速度。

2026 年推出的 GPT-Live 是重要升级，采用架构解耦的全双工设计，支持连续交互与自然插话。模型不再处理独立的语音轮次，而是持续监听音频流，能够识别用户的插话意图并自然切换话轮，大幅提升对话真实感。GPT-Live 还支持实时同声传译功能，可在对话过程中同步完成跨语言翻译。

开发者层面，OpenAI 提供 Realtime API，基于 WebSocket 协议，支持低延迟的"语音进、语音出"对话交互。API 支持文本与音频混合输入输出，可灵活适配不同应用场景。

价格方面，GPT-Realtime-2 的音频输入定价为 32 美元/百万 token，音频输出 64 美元/百万 token；文本输入 4 美元/百万 token，文本输出 24 美元/百万 token。整体定位高端，适合对对话质量要求极高的场景。

### 3.2 Google Gemini Audio 系列

Google 的 Gemini Audio 产品线包含多个面向不同场景的端到端语音模型。Gemini 3.1 Flash Live 是专为实时对话设计的高质量低延迟音频模型，采用原生 Audio-to-Audio 架构，音频直接输入模型、直接输出音频，中间无文字中转媒介，实现了延迟与语音质量的双重优化。

Gemini 3.5 Live Translate 则主打实时语音翻译能力，支持 70 种以上语言的语音到语音翻译，能够保留说话人的自然语气与节奏。该模型在多语言场景下表现突出，适合跨语言交流场景。

此外，Gemini 3.1 Flash TTS 提供精细化的语音合成控制，通过直观的音频标签可以精确调整语调、语速与语气风格，满足个性化语音输出需求。

开发者通过 Gemini Live API 接入，音频输入定价为 3 美元/百万 token（付费层级），显著低于 OpenAI。需要注意的是，原生音频模型仅支持 AUDIO 回答模态，如需文本形式响应需使用输出音频转写功能。

### 3.3 Microsoft VibeVoice（开源）

VibeVoice 是微软研究院于 2025 年 8 月开源的前沿语音 AI 模型家族，采用 MIT 许可证，GitHub 获得 46.2k 以上 Stars。项目包含 VibeVoice-ASR 语音识别与 VibeVoice-TTS 语音合成两大核心组件，核心定位是突破长音频处理的性能极限。

VibeVoice 最突出的能力是超长音频生成，可一次性生成长达 90 分钟的连续语音，支持最多 4 人对话场景与中英文无缝切换。技术上采用创新的连续语音 tokenizer 与 next-token 扩散框架，在长序列处理上相比传统方案有数量级提升。

需要注意的是，VibeVoice 主要聚焦 TTS 与 ASR 组件，并非完整的端到端对话模型，用于口语陪练场景需要与 LLM 组合使用。但其高质量的语音合成能力可作为传统架构的升级选项。

## 4. 国内主流端到端语音大模型

### 4.1 字节跳动豆包端到端语音大模型（Seeduplex）

字节跳动是国内端到端语音大模型的领先厂商，通过火山引擎提供豆包端到端实时语音大模型 API。2026 年 4 月发布的 Seeduplex 全双工语音大模型是其旗舰产品，基于"边听边说"的全新框架设计，相比上一代半双工模型，交互自然感与顺畅度大幅提升。

Seeduplex 具备**懂倾听**与**抗干扰**两大核心能力，能够准确识别用户的插话意图，在背景噪声环境下仍保持稳定的对话效果。豆包实时语音模型 3.0 API 还支持自定义工具调用，可在实时对话中直接调用外部工具完成任务，为语音 Agent 场景提供基础能力。

API 层面，豆包端到端实时语音大模型支持 WebSocket 协议双向流式交互，覆盖中文与英文两大语种。同时提供 iOS/Android SDK，方便移动端快速集成。从实际产品案例看，可栗口语等应用采用豆包融合技术，实现 300–500 毫秒级端到端响应，支持 Barge-in 打断与自然话轮切换，对话体验接近真人外教。

### 4.2 智谱 GLM-4-Voice

GLM-4-Voice 是智谱 AI 推出的端到端情感语音模型，基于 GLM-4-9B 基座进行语音预训练与对齐。模型采用单码本监督语音 tokenizer，以 12.5Hz 帧率高效表示语音，配合流匹配语音解码器生成自然语音。通过 1 万亿 token 规模的语音-文本预训练，有效打通了文本与语音模态的鸿沟。

该模型支持中英文实时语音对话，平均响应延迟控制在 300 毫秒以内。核心特色是**情感语音表达**能力，可根据用户指令灵活调整语音的情感、语调、语速与方言，使语音交互更具生动性。模型还支持随时打断，交互体验接近真人对话。

API 定价为 80 元/百万 token，输入模态支持音频与文本，输出模态为音频，上下文窗口 8K，最大输出 4K token。智谱同时提供开源版本的 GLM-4-Voice，支持本地化部署，适合对数据隐私有要求的场景。

### 4.3 阿里通义千问 Qwen-Omni

阿里通义千问的多模态语音能力主要通过 Qwen-Omni 系列提供，其中 Qwen3.5-Omni-Plus 是旗舰级多模态模型，支持音频输入与音频输出。模型在语音理解与生成方面均有较强表现，依托阿里云的基础设施提供稳定的 API 服务。

价格方面，Qwen3.5-Omni-Plus 的音频输入定价为 82.44 元/百万 token，音频输出 329.74 元/百万 token，整体定位中高端。此外，阿里云还提供独立的 Qwen-TTS 语音合成服务，可与文本大模型组合使用。

通义千问的优势在于阿里云生态的完整性与价格竞争力，纯文本模型价格已降至极低水平。对于已有阿里云技术栈的团队，接入成本相对较低。

### 4.4 科大讯飞星火语音大模型

科大讯飞作为传统语音技术龙头，在星火大模型体系中构建了专门的语音大模型，涵盖大模型语音识别与超拟人语音合成两大方向。讯飞采用三阶段层次化语音建模框架，在语音自然度与中文识别准确率方面保持行业领先。

在**声音复刻技术**方面，讯飞星火依托自监督预训练编码器，可从极短音频样本中提取声纹特征，实现高质量的个性化语音生成。这一能力对于打造个性化英语陪练角色具有实用价值。

需要注意的是，讯飞的语音能力更多以 ASR 与 TTS 独立模块的形式提供，完整的端到端语音对话一体化方案相对其他厂商曝光较少，需进一步验证其端到端架构的成熟度。

### 4.5 小米 MiMo-Audio（开源）

MiMo-Audio 是小米于 2025 年 9 月开源的原生端到端语音大模型，采用 12 亿参数 Tokenizer 与 70 亿参数主体模型的协同架构，通过 8 层残差矢量量化技术实现 25Hz 高精度音频 token 生成。模型训练数据超过 1 亿小时，在通用语音理解与对话等多项评测中大幅超越同参数量级的开源模型。

MiMo-Audio 原生支持语音转文字、文字转语音、语音转语音、语音风格迁移、多语言互转等 **5 种模态组合任务**。其独创的"补丁编解码"机制将 4 个连续音频 token 聚合成单个语义补丁单元，显著提升语言模型处理效率。

该模型支持多语言与多种方言，包括四川话、闽南语等，在噪音环境下表现出色。对于英语口语陪练场景，开源可本地化部署是核心优势，且模型本身具备较强的英文处理能力。

### 4.6 阶跃星辰 Step-Audio 2（开源）

Step-Audio 2 是阶跃星辰推出的真端到端语音大模型，采用连续输入与离散输出范式，避免了特征提取中的信息损失。2025 年 10 月开源的 Step-Audio 2 mini 仅以 2 亿参数，在 15 项国际评测中超越 GPT-4o Audio 与 Kimi-Audio。

识别准确率方面，Step-Audio 2 在英文测试集上平均词错误率（WER）为 3.14%，中文测试集平均字错误率（CER）为 3.08%，在阿拉伯语与日语识别上与 GPT-4o Transcribe 相当，粤语识别与 Qwen-Omni 持平。

延迟表现突出，传统架构约 300 毫秒的时延被压缩至 **80 毫秒**，同时保留 85% 的副语言信息。对于追求极致响应速度的口语陪练场景，Step-Audio 2 是值得关注的开源选项。

### 5.3 各维度详细分析

**英文语音识别与理解能力**。海外模型凭借更大规模的英文训练数据，在纯英文识别准确率与理解深度上整体占优。GPT-4o 与 Gemini 在复杂句式、口音适应与上下文理解方面表现最佳。国内模型中，GLM-4-Voice 与 MiMo-Audio 的英文能力已达到较强水平，能够满足日常口语对话需求；豆包与通义千问的英文能力也在快速提升，但相对中文仍有差距。对于中式英语口音的适应性，国内模型可能反而具备一定优势，因为训练数据中包含更多非母语英语样本。

**语音自然度与发音标准度**。端到端模型的语音自然度普遍优于传统 TTS，尤其是在语调起伏与情感表达方面。Google 与 OpenAI 的英文语音合成质量处于第一梯队，发音标准且富有表现力。国内模型中，GLM-4-Voice 的情感语音能力突出，讯飞星火在语音拟真度上有深厚积累。对于英语口语教学场景，标准清晰的发音比情感丰富度更为重要，这一点上主流模型均能满足基本要求，差异主要体现在自然流畅度上。

**延迟与交互流畅度**。延迟是语音对话体验的核心指标。Step-Audio 2 以 80 毫秒的延迟位居榜首，但作为开源小模型，对话能力相对有限。闭源产品中，GPT-4o 的 232 毫秒最低延迟保持领先，豆包 Seeduplex、GLM-4-Voice、Gemini Flash Live 均在 300–500 毫秒区间，能够支撑流畅的自然对话。全双工能力方面，GPT-Live 与 Seeduplex 是目前明确支持完整全双工交互的产品，其他模型多为半双工模式，支持打断但对话切换的自然度略逊。

**对话与教学能力**。英语口语陪练不仅是语音对话，还需要模型具备英语教学能力，包括语法纠错、发音指导、场景化对话引导等。这一能力主要取决于基座大模型的语言理解与生成水平。GPT 系列与 Gemini 在英文语言能力上最强，能够提供高质量的语法纠正与地道表达建议；国内模型中智谱与通义的英文能力相对突出。此外，是否支持 Function Call 与工具调用，决定了能否集成发音评分、单词查询等教学辅助功能，这方面豆包实时语音 3.0 与 OpenAI Realtime API 均已支持。

**成本分析**。闭源 API 按 token 计费，语音 token 的消耗速率远高于文本。粗略估算，一分钟语音对话约消耗 15–25 万音频 token。按此计算：

| 模型 | 每分钟对话成本（估算） |
|:-----|:----------------------|
| GPT-Realtime-2 | $10–15 |
| Gemini Flash Live | $1–2 |
| GLM-4-Voice | ¥12–20 |
| Qwen-Omni | ¥50–80 |

开源模型虽然 API 免费，但需要承担 GPU 服务器成本，单卡 A100 部署 7B 级模型的月成本约数千元，适合日活较大的场景摊薄成本。

## 6. 选型建议与落地路径

### 6.1 不同阶段选型推荐

**原型验证阶段（MVP）**。建议优先选择国内闭源 API 快速验证产品形态。豆包端到端语音 API 或 GLM-4-Voice 是首选——中文文档完善、接入便捷、国内访问延迟稳定，且英文能力足以支撑原型验证。开发周期短，可在 1–2 周内完成基础对话功能搭建，快速验证用户需求与产品方向。如果团队已有海外云资源，Gemini Flash Live 也是高性价比的选择，英文质量更优且价格显著低于 OpenAI。

**产品化阶段（规模化运营）**。根据目标用户群体分化选择：
- **高端用户**（追求极致对话体验）→ 接入 GPT-Realtime-2 或 Gemini 3.5 Live Translate，通过海外节点部署保障访问质量
- **大众用户**（成本敏感）→ 以豆包或 GLM-4-Voice 为主力模型，同时探索开源模型的混合部署方案

此阶段需要重点评估并发承载能力与稳定性，建议进行压力测试后再确定主选模型。

**成熟阶段（成本优化与差异化）**。当用户量达到一定规模后，引入开源模型进行混合部署是降低成本的关键路径。MiMo-Audio（7B）或 Step-Audio 2 可部署为基础层处理常规对话，复杂场景再路由到闭源大模型，预计可降低 50% 以上的 API 成本。同时，基于开源模型进行领域微调，针对英语口语教学场景优化识别与表达，可构建差异化的技术壁垒。

### 6.2 技术落地架构建议

完整的 AI 英语口语陪练系统建议采用"**端到端语音大模型 + 教学业务层**"的分层架构：

- **语音交互链路**：直接使用端到端语音大模型的 Realtime API，保障对话的自然流畅度
- **教学业务层**：在应用层实现场景管理、发音评估、学习记录、错误追踪等功能，通过 Function Call 或外部服务与语音模型联动

具体实现上，前端通过 WebSocket 与语音模型建立双向流式连接，实时传输用户音频并接收 AI 回复音频；后端服务负责用户身份认证、会话管理、学习数据持久化与教学策略调度。对于发音纠错等需要精确音素级评估的功能，可额外集成专业的语音评测引擎，与端到端模型形成能力互补。

### 6.3 成本估算参考

以**日均 1000 名活跃用户、人均每日对话 10 分钟**为例：

| 方案 | 月度成本估算 |
|:-----|:------------|
| Gemini Flash Live | ¥3–6 万 |
| GLM-4-Voice | ¥3.6–6 万 |
| GPT-Realtime-2 | ¥30–50 万 |
| 开源自部署（4×A100） | ¥2–4 万（硬件+带宽） |

若采用开源模型自部署方案，按 4 张 A100 GPU 服务器估算，硬件与带宽月成本约 2–4 万元，可支撑日均数千用户规模，用户量越大边际成本优势越明显。但需要投入额外的算法工程团队进行部署、优化与运维，适合技术能力较强的团队。

## 7. 总结

端到端语音大模型已经进入实用化阶段，为 AI 英语口语陪练产品提供了远超传统方案的对话体验：

- **技术成熟度**：海外厂商仍保持领先优势，国内厂商追赶速度很快，在中文场景与成本控制上更具竞争力；开源生态也在快速成熟
- **推荐技术路线**：国内闭源 API 快速起步 → 开源模型逐步替代。早期聚焦产品验证与用户体验打磨，随着用户规模增长逐步引入开源模型降低成本
- **核心建议**：模型选型不是一劳永逸的决策，建议建立多模型路由机制，根据不同场景与用户分层动态选择最优模型，在体验与成本之间取得最佳平衡
