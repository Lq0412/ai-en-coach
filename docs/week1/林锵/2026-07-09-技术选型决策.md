## 一、前端框架：React 18 + Vite

| 候选 | 优势 | 劣势 |
|---|---|---|
| **React 18** ✅ | 生态最大，教程和组件库最多；后续上 React Native 有积累；Vite 加持下开发体验好 | 团队零经验，需要学习曲线 |
| Vue 3 | 中文社区好，上手比 React 略快 | 生态规模不如 React；迁移 RN 路径不存在 |
| Svelte | 代码量最少，性能好 | 生态太小，5 人团队踩坑没人救 |
| Streamlit / Gradio | 纯 Python，零学习成本 | UI 交互受限，做不到自定义音频控件和非标准对话界面 |

### 决策理由

- 团队虽然零前端经验，但 React 的生态规模意味着**几乎所有问题都有现成答案**，学习曲线的实际成本比小众框架低
- 产品概念文档写明后续可能上移动端，React → React Native 是唯一有积累价值的路径
- 1024XEngineer 参考项目（ByteMind Web UI、CialloClaw）均使用 React

### 构建工具选 Vite

| 候选 | 优势 | 劣势 |
|---|---|---|
| **Vite** ✅ | 开发服务器秒启动，HMR 极快，React + Tailwind 模板开箱即用 | — |
| Create React App | 曾经是标准 | 已停止维护，启动慢 |
| Next.js | SSR/SSG 能力强 | 口语教练是 SPA，不需要 SSR，Next.js 的复杂度是额外负担 |

---

## 二、样式方案：Tailwind CSS

| 候选 | 优势 | 劣势 |
|---|---|---|
| **Tailwind CSS** ✅ | 直接在 HTML 里写样式类名，不需要学 CSS 架构（BEM、命名规范），开发速度最快 | HTML 可读性略差（长 class 列表） |
| CSS Modules | 传统写法，灵活，与原生 CSS 一致 | 需要团队成员有 CSS 基础，命名和组织成本高 |
| styled-components | CSS-in-JS，组件化思维 | 额外运行时开销，学习成本 |

### 决策理由

零前端经验的团队最大障碍不是写 CSS 语法，而是**组织 CSS**——什么时候用 BEM、什么时候拆文件、命名怎么不冲突。Tailwind 直接消除了这个问题。1024XEngineer 参考项目也用它做标配。

---

## 三、后端框架：FastAPI

| 候选 | 优势 | 劣势 |
|---|---|---|
| **FastAPI** ✅ | 原生 async/await，WebSocket 开箱即用，Pydantic 做请求/响应校验，自动生成 OpenAPI 文档，Python 生态性能最高 | 学习曲线比 Flask 略陡 |
| Flask | 最简单，灵活 | WebSocket 需 Flask-SocketIO 插件，异步能力弱，无自动数据校验 |
| Django | ORM、Admin、Auth 全套自带 | 太重，2 个月项目是杀鸡用牛刀 |

### 决策理由

- **async 是刚需**——核心链路"收音频 → 调 AI API → 分析 → 返回"每一步都是 I/O 等待，同步框架会让一个请求卡住整个 worker
- **WebSocket 原生支持**——不需要额外插件，一行装饰器搞定
- **Pydantic 校验**——LLM 返回的评分 JSON 结构复杂且不稳定，Pydantic 自动校验和序列化，Flask 需手写校验
- **自动 OpenAPI 文档**——`/docs` 即 API 文档，前后端并行开发时省去手写和同步成本

---

## 四、通信协议：WebSocket

| 候选 | 优势 | 劣势 |
|---|---|---|
| **WebSocket** ✅ | 全双工，服务端可主动推送；一连接多轮对话，上下文不中断；天然适配语音交互（流式 ASR 结果、打断信号、TTS 流式播放） | 调试比 REST 麻烦，需处理断线重连和心跳 |
| REST | 最简单，调试方便 | 单向请求-响应，每轮对话需新建连接；不支持服务端推送 |
| JSON-RPC 2.0 over WebSocket | 方法/参数/错误码 schema 化，1024XEngineer 标准 | 多一层抽象，MVP 可以先不用 |

### 决策理由

- 虽然 MVP 是回合制，但 WebSocket 连接**跨多轮对话保持**，不需要每轮重新建连
- v2 要上实时打断和流式 TTS 时，WebSocket 是必需的——与其 v1 REST → v2 重写通信层，不如一开始就上 WebSocket
- FastAPI 原生支持，实现成本不高
- v1 直接在 WebSocket message 里约定 `{method, params, id}` 格式即可，不引入独立协议包
