### 3.1 Python 框架选型 — 智

#### 调研问题

1. FastAPI vs Django vs Flask：哪个最适合**实时语音对话**场景？
2. WebSocket 支持成熟度（实时语音流传输）
3. 异步能力（语音流 + LLM 流式输出并发处理）
4. 社区生态、学习成本、部署复杂度

#### 关注要点

- 语音对话是 **长连接 + 流式处理** 场景，不是传统 HTTP 请求-响应
- 框架对 WebSocket 的原生支持程度很重要
- 后续可能接入 WebRTC，需要考虑扩展性

### 3.2 数据库选型 — 张

#### 调研问题

1. MySQL 能否满足 MVP 需求？瓶颈在哪？
2. 会话记录 / 对话历史怎么存？（JSON / 关系表 / 独立存储？）

- 语音对话的**数据结构特殊**：每句话 + 时间戳 + 发音评分 + 纠错结果

### 3.3 ASR 选型 — 黄

#### 调研问题

1. 主要候选：**Whisper**（本地/API）、**Deepgram**、**Azure STT**、**Google STT**
2. 关键指标（按优先级）：
   - **端到端延迟**（直接影响对话流畅度，P0）
   - 中文 + 英文混合识别准确率
   - 是否支持**流式识别**（边说边出，不等说完）
   - 价格（API 调用成本 / 自建服务器成本）
3. 竞品 Speak / ELSA / 流利说 各用的什么 ASR？（从 T2 竞品调研交叉验证）

#### 关注要点

- ASR 延迟是口语产品的**生命线**。延迟 > 500ms 用户感知明显
- 流式识别 vs 分段识别的选择直接影响打断体验
- 中文口音 + 英文混合场景是识别难点

### 3.4 TTS 选型 — 覃

#### 调研问题

1. 主要候选：**ElevenLabs**、**Azure TTS**、**OpenAI TTS**、**Coqui TTS**（开源）
2. 关键指标（按优先级）：
   - **自然度 / 真人感**（用户访谈中会收集对「声音自然度」的期望）
   - **首字延迟**（TTFB，直接影响对话节奏）
   - 语速控制、停顿、情感表达
   - 声音定制（是否支持克隆/定制 voice）
   - 价格
3. 结合 T1 访谈了解用户：更在意「声音好听」还是「反应快」？

#### 关注要点

- TTS 延迟与 ASR 延迟共同决定**对话轮转速度**
- 「高压/冲突」场景需要 TTS 能表达**不耐烦、严肃、挑衅**等情感（方向 D 的独特需求）
- 是否需要**打断当前播报**（TTS 流式截断）？各方案的 API 是否支持？

## 八、附录：技术风险预判

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| ASR 端到端延迟 > 1s | 对话不自然，产品价值坍塌 | 优先测延迟；备选方案需提前确认 API 可用性 |
| 中英混合识别率低 | 纠错不准确，用户不信任 | 准备中英分离策略或二次纠错 |
| TTS 不支持打断截断 | 打断体验实现困难 | 评估流式 TTS 的截断机制 |
| Python 框架选错导致性能瓶颈 | 重构成本高 | 优先选异步能力最强的框架 |
| 团队无人有实时语音系统经验 | 踩坑周期长 | 本周先确认是否有现成 SDK 或样板项目 |
