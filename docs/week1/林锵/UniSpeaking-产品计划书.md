# UniSpeaking 产品计划书-竞争对手

---

# 一、用户问题

## 1.1 一句话定位

UniSpeaking 是一款面向成人英语学习者的 AI 实时口语陪练产品，通过端到端语音对话和学习者友好的对话策略，帮助用户以低压力、低成本、高频率完成英语开口练习。

## 1.2 要解决的核心问题

英语口语是大多数中国英语学习者最薄弱的环节。核心矛盾不是「没有学习工具」，而是**现有方式无法让用户稳定、持续、低压力地开口说英语**。

具体表现为三个层次的问题：

**第一层：没有随时可用的练习对象。** 真人外教需要预约和付费，语言交换需要匹配，英语角受时间地点限制。即使用户有 5-10 分钟碎片时间想练口语，也找不到能立即开始的对话对象。

**第二层：开口存在心理障碍。** 许多学习者并非完全不会表达，而是在面对真人时害怕出错、害怕被评价，导致长期不敢开口。「不敢说」比「不会说」更先成为口语提升的瓶颈。

**第三层：现有 AI 能聊天，但不会陪练。** 通用 AI 助手（如 ChatGPT、豆包）可以用英语对话，但它们的目标是回答问题或提供信息，而不是帮助学习者开口。常见问题包括：AI 单次回复过长，用户听得多说得少；语言难度与用户水平不匹配；用户卡壳时缺少恰当引导；对话缺少学习目标，无法沉淀为长期提升路径。

## 1.3 目标用户

### 首批核心用户

产品第一阶段优先服务的不是「所有想学英语的人」，而是**已经具备基础表达能力、并愿意重复开口的成人学习者**。

核心画像：
- 已掌握一定词汇和语法（约四级水平），能说简单句但不流利
- 有明确的口语提升意愿，愿意每周练习 3 次以上
- 曾尝试过跟读 App、英语角或真人外教，但没有形成稳定练习习惯
- 更需要一个「可以立即开始、不怕出错」的开口环境

选择这类用户的原因：他们能直接验证产品的核心假设——**低压力的 AI 实时对话能否让用户形成持续的口语练习习惯**。

### 后续扩展用户

- 有明确场景目标的用户：准备旅行、留学、外企面试、IELTS 等
- 希望利用碎片时间保持英语语感的职场人群

### 明确不覆盖的用户

- 零基础用户（需要语法/发音教学，无法进入自由对话）
- 只需要背单词、刷题的用户（核心需求不是口语交流）
- 儿童用户（需要完全不同的内容和安全设计）

## 1.4 关键使用时刻

用户最可能在以下时刻打开产品：
- 通勤、午休或睡前有 5-15 分钟空闲
- 意识到当天还没有开口说过英语
- 学过一组表达，想立刻在对话中试用
- 想练口语但不愿预约真人或参加英语角

产品应让用户在产生练习意愿后 **10 秒内进入对话**，减少一切前置步骤。

# 二、竞品分析与差异化

## 2.1 口语练习产品的交互方式分类

当前市场上的英语口语练习产品，按核心交互方式可分为四类：

### 第一类：跟读评分式

**代表产品**：流利说、ELSA Speak、多邻国

**交互流程**：系统播放示范音频 → 用户点击录音按钮 → 用户朗读指定句子 → 系统评分并反馈发音准确度

**优势**：评分维度明确（发音、语调、流利度），用户有清晰的进步感知；适合发音矫正和句式训练。

**局限**：
- 本质是「复读机」而非「对话」，用户不需要组织语言，只需模仿
- 无法训练临场表达能力——用户知道要说什么，不需要自己想
- 练习与真实交流场景脱节，用户「跟读满分」但开口仍然困难
- 每句话之间是独立的，缺少对话的连续性和上下文

### 第二类：点击后录音式

**代表产品**：部分 AI 口语应用、海螺 AI 的口语练习模式

**交互流程**：AI 提出问题（文字或语音）→ 用户按住录音按钮 → 松开后 AI 处理 → AI 回复

**优势**：有对话感，用户需要自己组织语言；比跟读更接近真实交流。

**局限**：
- 「按住-松开」的操作打断了对话的自然节奏
- 用户无法在 AI 说话过程中打断，必须等 AI 说完才能操作
- 每轮对话之间有明显的「等待处理」间隙，缺少连续流动感
- 用户容易形成「听完 → 想好 → 按钮 → 说」的答题心态，而不是自然交流

### 第三类：通用 AI 实时语音对话

**代表产品**：GPT-4o 语音模式、豆包 AI 语音

**交互流程**：用户直接开口说话 → AI 实时理解并以语音回复 → 支持打断和连续对话

**优势**：最接近真实对话体验，延迟低、支持打断、对话连续。

**局限**：
- 作为通用助手设计，不针对英语学习者优化
- AI 倾向于给出完整、详细的回答，用户在对话中的输出比例低
- 语言难度不可控，可能远高于或低于用户实际水平
- 用户卡壳时 AI 不会降低难度或提供表达提示
- 不提供学习路径、反馈报告和持续进步追踪
- 对话结束后没有任何学习沉淀

### 第四类：学习者友好的实时语音对话（UniSpeaking）

**交互流程**：用户打开即进入语音通话 → AI 主动打招呼引导话题 → 双方自然交流 → AI 根据用户水平动态调整难度和回复方式 → 用户可随时打断 → 结束后无压力反馈

**设计差异**：
- AI 控制回复长度（1-3 句），将更多对话时间留给用户
- 根据用户水平动态调整词汇、语速和句式难度
- 用户卡壳时主动降级并提供表达提示，而不是等用户想出来
- 不随意纠错，优先理解用户表达意图并推进对话
- 对话结束后不展示评分，保持低压力体验

## 2.3 UniSpeaking 的差异化定位

UniSpeaking 的核心交互不是跟读评分，不是课程教学，也不是通用聊天——它专注于一件事：

> **让 AI 知道怎样陪一个英语学习者说话，并让用户在每次对话中获得更多有效开口时间。**

需要说明的是，UniSpeaking 的场景训练中包含跟读环节（「学-读-说」流程中的「读」），但跟读在这里是模拟对话前的**准备步骤**，而不是产品的核心交互。区别在于：流利说等产品的整个体验围绕「跟读→评分」构建，UniSpeaking 的跟读只是帮用户熟悉表达，最终目的是让用户在模拟对话中能用出来。

差异化的核心不是「能和 AI 用英语聊天」（这一点 GPT-4o 和豆包已经做到），而是：

1. **用户说得更多**：AI 控制回复长度，主动追问，将对话时间留给用户
2. **难度更合适**：根据用户水平和当次表现实时调整
3. **卡壳有人接**：用户停顿或表达困难时，AI 主动降级并提供帮助
4. **压力更低**：不纠错、不评分、不考试，先接住表达再推进对话
5. **练习可升级**：从自由对话到场景训练，形成从「敢说」到「会说」的成长路径

# 三、核心业务

## 3.1 产品能力四层结构

UniSpeaking 的产品能力分为四层，由下向上逐层构建：

```
┌─────────────────────────────────────────────────────┐
│  第四层：专业场景训练                                   │
│  IELTS口语 / 外企面试 / 职场英语                        │
│  专用题库 · 专用评分 · 结合用户资料                      │
├─────────────────────────────────────────────────────┤
│  第三层：场景化训练（场景广场）                           │
│  用户输入场景 → 学词汇 → 跟读 → 模拟对话 → 反馈          │
│  「学-读-说」闭环                                      │
├─────────────────────────────────────────────────────┤
│  第二层：学习者对话策略                                  │
│  难度适配 · 回复控制 · 追问引导 · 卡壳帮助 · 开口保护     │
│  ★ 核心差异层                                          │
├─────────────────────────────────────────────────────┤
│  第一层：实时语音底座                                    │
│  端到端语音对话 · 低延迟 · 打断 · 轮次管理 · 上下文连续    │
└─────────────────────────────────────────────────────┘
```

**第一层**是基础设施，决定用户能否流畅交流；**第二层**是核心差异，决定用户为什么选择本产品而不是通用 AI；**第三层**承接具体学习目标；**第四层**承接高价值专业需求和商业变现。

## 3.2 三大核心模块

### 模块一：自由对话（核心入口）

产品的第一入口和核心体验。用户打开产品即进入类似电话通话的实时英语对话，AI 主动打招呼、引导话题，用户随时开口、随时打断、随时结束。

**核心价值**：降低开口门槛，让用户建立「每天说几分钟英语」的习惯。

**设计原则**：不评分、不纠错、不限定话题，先接住用户表达，再推进对话继续。

### 模块二：场景广场（结构化训练）

用户输入想练习的具体场景（如「在酒店前台申请延迟退房」），系统生成结构化学习路径：

1. **学**：生成该场景的核心词汇、短语和常用表达
2. **读**：用户跟读关键表达，熟悉发音和句式
3. **说**：进入带角色和任务目标的模拟对话，AI 扮演对应角色进行追问

练习结束后系统给出基础反馈，帮助用户知道哪里可以优化。

**核心价值**：将「自由聊天」升级为「有目标的练习」，满足用户对具体场景的准备需求。

### 模块三：个人主页（学习沉淀）

承接用户的学习记录和反馈，提供复练入口：
- 基础学习概况（累计练习次数、时长、场景）
- 练习记录列表（自由对话 / 场景练习）
- 反馈记录（来自场景模拟，不来自自由对话）
- 历史场景复练入口

**核心价值**：让每次练习留下痕迹，形成「练习 → 记录 → 复练 → 进步」的闭环。

## 3.3 模块关系

```
用户打开产品
    │
    ├── 自由对话（低压力开口，建立习惯）
    │       │
    │       └── 产生练习意愿 ──────────┐
    │                                  │
    ├── 场景广场（有目标训练）◄──────────┘
    │       │
    │       ├── 学 → 读 → 说（闭环）
    │       │
    │       └── 反馈 & 复练
    │               │
    └── 个人主页（沉淀记录）◄──────────┘
```

自由对话负责**拉新和习惯养成**，场景广场负责**学习深度和留存**，个人主页负责**记录沉淀和复练驱动**。

# 四、核心体验

## 4.1 自由对话体验设计

### 4.1.1 交互形态

自由对话采用**语音通话式界面**，而非传统聊天对话框。用户打开页面后的体验应接近「和一个朋友打电话」，而不是「和一个 AI 文字聊天」。

| 设计决策 | 选择 | 理由 |
|---------|------|------|
| 界面形态 | 通话式，非聊天气泡式 | 降低用户对文字的依赖，鼓励纯听说 |
| 文字显示 | 默认不显示，用户可按需展开 | 避免用户习惯性看字幕而不练听力 |
| AI 语音 | 优先输出语音，不等文字生成完成 | 保证对话的即时性和连续感 |
| 打断能力 | 用户开口即中断 AI 播报 | 还原真实对话的自然节奏 |
| 结束方式 | 用户主动结束，无评分弹窗 | 保持低压力体验 |


### 4.1.2 对话策略

**AI 主动开场**：用户进入后 AI 先打招呼，不要求用户先完成任何任务。

> Hi! I'm happy to chat with you. What would you like to talk about today?

**用户不知道聊什么时**：AI 给出轻量建议。

> We can talk about your day, movies, food, travel, or anything you like.

**AI 回复原则**：
- 句子短、表达自然、不使用过难词汇
- 每次回复 1-3 句，不长篇大论
- 多追问，引导用户继续说
- 不频繁纠错，优先理解用户意图

**隐性纠错示例**：
- 用户：I go work today very tired.
- AI：Sounds like a long day. What made you feel so tired today?
- （不直接纠正 "I went to work"，而是在回复中自然使用正确形式）

### 4.1.3 难度适配

**初始难度**：新用户默认四级左右（常用词汇、中短句、适中语速、生活化话题）。

**短期降级信号**：
- 用户长时间停顿（>5 秒）
- 频繁使用中文或中英夹杂
- 连续短回答（yes/no/I don't know）
- 多次请求翻译

**降级方式**：不直接切换等级，而是逐步降低表达难度。

| 场景 | AI 处理方式 |
|------|-----------|
| 用户沉默 | "No worries. You can say anything simple. How was your day?" |
| 用户用中文 | 先理解，再引导："I understand. You can try saying it in English like this..." |
| 用户说不知道 | 给选择题："Do you want to talk about food, movies, or your day?" |
| 用户表达很卡 | 降低要求："Take your time. Just say one simple sentence." |
| 识别失败 | 轻量提示："Sorry, I didn't catch that. Could you say it again?" |

**长期调整**：结合多次对话的平均回复长度、英文占比、停顿时长、翻译频率等指标，逐步调整用户等级。

## 4.2 场景广场体验设计

### 4.2.1 核心流程

```
用户输入场景（文本）
    │
    ▼
系统生成学习资料
    │
    ├── 「学」：核心词汇、短语、常用表达
    │
    ├── 「读」：跟读关键表达
    │
    └── 「说」：带角色的模拟对话
            │
            ▼
        基础反馈
```

### 4.2.2 示例：「在酒店前台申请延迟退房」

**学**：
- late checkout / extra charge / available room
- Could I extend my checkout time?
- Is there an extra fee for late checkout?

**读**：用户跟读以上关键表达，熟悉发音和句式。

**说**：AI 扮演酒店前台，围绕房间号、退房时间、额外费用等展开对话。

**反馈**：
- 是否完成了场景任务
- 哪些表达比较自然
- 建议复练的 1-2 个表达

### 4.2.3 场景类型

**普通自定义场景**：用户自由输入，如咖啡店点单、机场转机、向同事解释项目延期等。由大模型根据输入自动生成学习资料和模拟对话。

**专业场景（独立设计）**：IELTS 口语、外企面试等不应按普通场景处理，而需要单独设计题型、评分维度和训练流程（详见第八章扩展业务）。

### 4.2.4 与自由对话的核心区别

| 维度 | 自由对话 | 场景广场 |
|------|---------|---------|
| 目标 | 低压力开口 | 完成具体场景练习 |
| 话题 | 无限制 | 围绕用户输入的场景 |
| 反馈 | 默认无结构化反馈 | 有任务完成度和表达优化建议 |
| 学习路径 | 无 | 学 → 读 → 说 |
| 心理定位 | 「随便聊聊」 | 「为真实场景做准备」 |

### 4.2.5 设计取舍

**为什么不只做自由对话**：自由对话可以降低开口门槛，但训练目标容易发散。用户可能聊了很久，却不知道自己是否掌握了某个真实场景需要的表达。场景广场通过「学-读-说」路径让每次练习都有明确目标。

**为什么采用「学-读-说」**：如果用户输入场景后直接进入模拟对话，低中水平用户容易因为缺少表达储备而卡住。先学后读再说，更符合从准备到表达的学习过程。

**为什么专业场景不交给自定义生成**：IELTS 涉及考试结构和评分维度，英文面试涉及用户资料和岗位匹配。它们的训练价值来自更专业的设计，不能完全依赖普通场景生成。

## 4.3 学习者对话策略（核心差异层）

这一层是 UniSpeaking 区别于通用 AI 助手的关键。以下策略需要贯穿自由对话和场景模拟：

| 策略 | 具体做法 | 解决什么问题 |
|------|---------|-------------|
| 回复长度控制 | AI 每次回复 1-3 句 | 避免 AI 占用过多对话时间 |
| 难度适配 | 根据用户水平调整词汇、句式和语速 | 避免过难挫败或过易无效 |
| 主动追问 | 每 1-3 轮主动抛出问题 | 避免对话中断 |
| 卡壳帮助 | 提供关键词、半句提示或更简单的表达方式 | 用户停顿时有人接住 |
| 隐性纠错 | 先接住表达，再在回复中自然使用正确形式 | 避免打断对话流畅度 |
| 话题连续 | 保持上下文，不突然转换或重复提问 | 还原真实对话体验 |
| 开口保护 | 不因小错误频繁打断 | 保护用户开口意愿 |

# 六、可行性分析

## 6.1 技术可行性

### 6.1.1 端到端语音方案（Speech-to-Speech）

UniSpeaking 选择 **Speech-to-Speech 端到端语音方案**，而非传统的 ASR → LLM → TTS 管线式方案。

**选择理由**：

| 维度 | 管线式（ASR→LLM→TTS） | 端到端（S2S） |
|------|----------------------|--------------|
| 延迟 | 三个环节串行，累积延迟高 | 单模型端到端处理，延迟更低 |
| 打断 | 需要额外实现 VAD + 状态管理 | 原生支持打断和轮次管理 |
| 自然度 | 合成语音生硬 | 语音表达更自然、有情感 |
| 可控性 | 可以精确控制中间文本 | 文本不一定作为中间产物，控制能力取决于 API |
| 成本 | 三个 API 分别计费 | 单一 API 计费，但单价可能更高 |

**待调研确认项**：

具体 API 供应商需要从以下候选方案中调研选择：

| 候选方案 | 关注点 |
|---------|--------|
| 字节豆包（火山引擎 Doubao Realtime API） | 延迟、中文支持、学习者场景适配能力 |
| 阿里通义千问（Qwen Audio 实时语音） | 延迟、打断支持、多轮上下文能力 |


**调研标准**：
1. 端到端延迟是否能控制在 1.5s 以内
2. 是否支持实时打断和 VAD
3. 是否可通过 system prompt 控制 AI 的语速、难度和回复长度
4. API 成本结构（按分钟 vs 按 token）
5. Web 端 WebRTC 接入的成熟度

### 6.1.2 客户端技术方案

第一阶段采用 **Web 应用**，后续扩展至移动端。

| 技术选项 | 方案 |
|---------|------|
| 客户端 | Web（优先），后续移动端App |
| 音频传输 | WebRTC |
| 前端框架 | React Native |

Web 端的优势是开发迭代快、无需应用商店审核，适合第一阶段快速验证。

## 6.2 对话策略可行性

学习者友好的对话策略是产品的核心差异点，其实现路径需要分阶段验证：

| 策略 | 实现方式 | 可行性评估 |
|------|---------|-----------|
| 回复长度控制 | System prompt 指令 + 后端截断 | 高（prompt 指令通常有效） |
| 难度适配 | Function Call | 中高（需测试不同模型的遵从度） |
| 主动追问 | System prompt 角色设定 | 高 |
| 卡壳帮助 | Prompt 指令 + 后端检测沉默时长 | 中（依赖 VAD 和超时检测） |
| 隐性纠错 | Prompt 指令 | 中（模型可能仍然直接纠错） |

**风险**：如果端到端模型对 system prompt 的遵从度不够（特别是回复长度和难度控制），可能需要回退到管线式方案，在 LLM 层增加工程化控制逻辑。

**缓解策略**：第一阶段以 prompt 工程为主，同时保留管线式方案作为备选。

## 7.2 核心数据流（自由对话）

```
用户开口说话
    │
    ▼
客户端采集音频 → WebRTC 发送至后端
    │
    ▼
对话编排引擎 → 转发至 S2S API（携带 system prompt + 上下文）
    │
    ▼
S2S API 实时返回语音流
    │
    ▼
后端转发语音流 → 客户端实时播放
    │
    ├── 同时：后台并行生成文本（用于用户按需查看）
    │
    └── 用户打断 → 客户端发送 interrupt 信号 → 停止播放
```

| 第一阶段客户端 | Web | 迭代快，无需审核 |

# 八、扩展业务

## 8.1 专业场景训练

普通自定义场景之外，产品需要为高价值专业场景提供独立的训练设计。专业场景与普通场景的核心区别在于：它们有特定的题型结构、评分标准和专业内容，不能简单复用通用生成。

### IELTS 口语训练

- 支持选择 Part 1、Part 2、Part 3 或完整模拟
- 生成符合题型的题目和追问
- 按流利度、词汇、语法、发音等维度提供反馈
- 明确反馈仅作为练习参考，不等同于官方成绩

### 外企面试训练

- 用户输入简历、岗位 JD 或项目经历
- 系统生成面试 Brief，用户确认后进入模拟
- AI 围绕用户资料进行行为面试追问
- 反馈包括回答结构、表达质量和岗位匹配建议
- 系统不编造用户未提供的经历或数据

## 8.2 商业模式

### 8.2.1 商业化逻辑

产品的商业化不建立在「无限 AI 聊天」之上——通用 AI 已经能提供免费聊天，用户很难为此付费。付费价值应来自**更专业的场景、更清晰的训练路径、更具体的反馈标准**。

### 8.2.2 免费与付费边界

| 能力 | 定位 | 理由 |
|------|------|------|
| 自由对话（限时） | 免费 | 核心入口，负责拉新和建立信任 |
| 普通自定义场景（限次） | 免费/基础 | 证明产品不只是聊天工具 |
| 更长对话时长 + 更多场景 | 基础会员 | 日常练习用户的付费点 |
| IELTS / 面试等专业场景 | 高级会员/专项包 | 高价值、强目标场景 |
| 深度测评报告 | 单次付费 | 结果可视化，用户容易理解付费价值 |

### 8.2.3 参考定价

| 层级 | 建议价格 | 核心权益 |
|------|---------|---------|
| 免费版 | ¥0 | 每天 5-10 分钟自由对话 + 每天 1 次普通场景 |
| 基础会员 | ¥29.9/月 | 更长时长 + 更多场景 + 错题本 + 基础反馈 |
| 高级会员 | ¥49.9/月 | 含基础权益 + 专业场景 + 深度反馈 |
| 专项训练包 | ¥29.9-69.9/个 | 外企面试 10 题、IELTS 高频题等单一目标 |
| 单次深度测评 | ¥9.9-29.9/次 | CEFR 估算、IELTS 口语预测分、面试报告 |

### 8.2.4 商业化承接位置

本期不实现完整付费体系，但在产品结构中预留承接位置：
- **场景广场**：用户选择专业场景时，提示高级训练入口
- **个人主页**：历史记录和反馈自然引导用户继续训练
- **练习结束页**：用户看到反馈后，推荐更系统的训练路径

## 8.3 明确不做的边界

以下内容不是开发顺序问题，而是产品的**长期定位边界**：

| 不做 | 理由 |
|------|------|
| 完整英语课程平台 | 产品定位是「口语练习入口」，不是课程工具 |
| 自由对话中的评分和纠错 | 自由对话长期保持低压力定位 |
| 英语社交社区 | 核心关系是「用户与 AI」，不是用户之间 |
| 真人外教/陪练平台 | AI 的价值是随时可用和低压力 |
| 通用 AI 助手 | 不扩展写作、问答等与口语无关的能力 |
| 替代官方考试评分 | 评分定位是练习参考，不是认证结论 |
