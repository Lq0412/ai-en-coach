# 外企英文面试陪练 MVP Proposal

## 动机 / 用户故事

希望进入外企、出海团队或国际化技术岗位的求职者，常见问题不是不会技术，而是无法在英文面试里自然、完整、专业地表达自己的技术能力。用户能阅读英文文档，也能提前写好自我介绍，但一旦进入真实追问，就容易卡词、逻辑散、回答过短，或者不知道自己到底是发音、表达、结构还是面试思路出了问题。

典型用户故事：

- 作为一名准备外企面试的开发者，我希望在正式面试前围绕自我介绍、行为题和项目经历进行英文语音练习，这样我可以把真实经历答顺、答清楚。
- 作为一名能看懂英文资料但口语表达不足的候选人，我希望系统像面试官一样追问，而不是只让我背固定答案，这样我可以提前暴露临场卡壳点。
- 作为一名时间有限的求职者，我希望每次练习后获得具体反馈和下一轮复练建议，而不是只得到泛泛的鼓励或一个看不懂的分数。

一句话概括共性需求：

**用户需要一个懂技术面试语境、能进行多轮追问、并能输出可复练反馈的英文面试教练。**

## 目标用户

首期目标用户是准备外企、出海公司、海外岗位或国际化团队面试的技术相关候选人，优先覆盖：

- 程序员、测试、产品、技术运营等技术相关岗位候选人。
- 计算机相关专业学生、应届生和 0-5 年经验工程师。
- 有一定英文阅读和基础表达能力，但在英文面试中容易紧张、卡词、结构混乱的成人学习者。
- 明确以“通过英文面试”或“提升英文技术表达”为目标的用户。

首期不面向完全零基础英语学习者，也不面向纯考试提分、儿童英语、泛娱乐陪聊或高级商务谈判用户。

## 现有做法及其不足

用户当前通常会通过以下方式准备英文面试：

- 背英文自我介绍、项目介绍和常见问答模板。
- 使用通用 AI 或语音助手做英文对话。
- 使用通用英语学习 App 练发音、语法或日常口语。
- 找朋友、老师或真人外教做少量模拟面试。

这些方式仍有明显不足：

- 背稿只能覆盖固定问题，一旦追问方式变化就容易失效。
- 通用 AI 能陪聊，但往往不够懂技术面试语境，追问浅，反馈泛。
- 通用英语 App 更偏发音、课程或日常表达，难以覆盖项目深挖、行为题结构和技术表达。
- 真人陪练成本高、频次低，不适合高密度复练。
- 很多工具能指出“说得不好”，但不能告诉用户下一轮应该具体练什么。

因此，本期不把目标定义为“再做一个 AI 口语 App”，而是聚焦一个更窄的结果导向场景：**帮助候选人在英文技术面试中表达得更自然、更专业、更有结构。**

## 本期范围与明确不做

### 本期范围

本期聚焦”外企英文面试陪练”单一主方向，跑通最小可演示闭环：

1. 用户选择一个面试练习场景。
2. 用户完成背景设置（支持简历上传 PDF 全文进 LLM 上下文；未上传简历的用户用 3 个选择题兜底）。
3. AI 以面试官身份发起英文问题。
4. 用户用语音回答。
5. AI 根据用户回答进行追问或自然推进。
6. 用户完成 3-5 轮问答。
7. 系统在练习结束后输出结构化反馈。
8. 用户获得下一轮复练建议。

首期候选场景包括：

- 英文自我介绍。
- 行为面试。
- 项目经历深挖。
- 高频基础技术问答。

为了控制范围，Demo 优先跑通其中 1 个主场景。若开发资源收紧，优先级按“自我介绍、行为面试、项目经历深挖”排序，基础技术问答可作为候补场景。

首期结构化反馈至少包含：

- 回答结构问题，例如是否有 STAR 结构、是否缺少具体行动或结果。
- 语言表达问题，例如不自然、不专业、过于口语化或表达重复。
- 可理解度问题，例如明显影响理解的发音或流利度问题。
- 原句与优化表达对照。
- 本轮总结和下一轮复练建议。

### 明确不做

本期明确不做：

- 不做泛英语学习平台。
- 不做旅游、点餐、社交等生活口语大场景。
- 不做完整求职社区、岗位投递、简历优化平台或商业化会员系统。
- 不做完整课程体系、闯关体系、打卡排行榜。
- 不做视频生成、数字人面试官或多角色扮演。
- 不做音素级发音评分，不把产品做成发音课。
- 不自研 ASR、TTS、LLM 等底层模型能力。
- 不做简历结构化解析（复杂字段提取、格式化），简历上传后全文进 LLM 上下文即可，不做字段级拆解。
- 不依赖 AI 实时开放生成无限题目，优先使用轻量题库或知识库约束问题质量。

## 关键决策与依据

### 决策 1：选择“外企英文面试陪练”作为主方向

早期讨论过多个方向，包括泛英语口语陪练、高压场景口语模拟、沉浸式外语环境模拟和英文面试陪练。综合用户目标、付费理由、MVP 边界和两个月落地难度，本期选择外企英文面试陪练。

对比结论：

| 候选方向 | 优点 | 主要问题 | 本期判断 |
|---|---|---|---|
| 泛英语口语陪练 | 市场大，用户广 | 范围太宽，竞品成熟，2 个月内难做出清晰差异化 | 不作为主方向 |
| 高压场景口语模拟 | 记忆点强，展示效果好 | 场景低频，长期付费理由偏弱 | 可作为后续特色模块 |
| 沉浸式外语环境模拟 | 产品感强，适合展示 | 研发和内容成本较高，容易偏离学习价值 | 暂不作为本期主线 |
| 外企英文面试陪练 | 用户目标明确，结果导向强，边界可控 | 需要做好追问质量和反馈可信度 | 作为本期主方向 |

该决策的核心理由是：英文面试具有明确结果压力，用户不是泛泛“想学英语”，而是要在具体面试场景中把自己已有的技术能力讲清楚。这个切口更容易解释价值，也更容易在 MS1 和后续 MVP 中做出可演示闭环。

### 决策 2：做“教练”，不是“聊天搭子”

通用 AI 已经可以陪用户聊天，单纯语音对话不构成差异化。本产品的关键不是让用户聊得久，而是让用户练完后知道自己哪里变强、哪里还要练。

因此，产品体验采用“自然对话 + 事后诊断”的方式：

- 对话中 AI 扮演面试官，保持追问和推进，不逐句打断纠错。
- 用户卡住时，AI 可以提供最小提示，例如关键词、句首或简短表达。
- 练习结束后，系统再输出结构化反馈。
- 反馈必须定位到具体回答或句子，而不是只给抽象评价。

这能避免把练习过程变成考试，也能避免用户每说一句都被打断，导致更紧张、更不敢说。

### 决策 3：首期用轻量题库 / 知识库约束问题质量

如果完全让 AI 开放生成面试问题，容易出现问题不稳定、追问不贴合、偏离面试语境或反馈泛化的问题。英文技术面试有相对稳定的高频场景和追问套路，首期更适合先人工整理轻量题库或知识库。

首期题库不追求大而全，只需要支撑核心演示链路：

- 常见自我介绍问题。
- 行为面试高频问题。
- 项目经历追问套路。
- 基础技术表达题。
- 参考回答要点和追问方向。

AI 的作用不是随机出题，而是在受控材料范围内进行自然追问和结构化反馈。

### 决策 4：反馈以“可复练”为核心，不以裸分数为核心

很多口语产品会给用户一个总分，但用户很难从总分知道下一轮怎么改。本期反馈优先采用“问题定位 + 对照改写 + 下次建议”的形式。

示例：

```text
你说：
I solved the problem very fast.

问题：
表达过于笼统，没有说明具体行动和结果。

可以改成：
I identified the root cause quickly and coordinated with the backend team to deliver the fix before the deadline.

下次重点：
回答项目问题时补充具体行动、数据或结果。
```

如果需要展示综合状态，可以用定性标签或维度状态，不把“78 分”作为用户理解产品价值的核心。

## 基本概念与信息结构

本期产品围绕以下概念组织：

| 概念 | 定义 |
|---|---|
| 面试场景 | 一次练习所属的主题，例如自我介绍、行为面试、项目深挖、基础技术问答 |
| 用户背景 | 用户为本次练习提供的轻量信息，例如目标公司类型、岗位方向、经验年限 |
| 练习会话 | 用户在一个场景下完成的一次完整模拟面试练习 |
| 面试轮次 | AI 提问 / 追问与用户回答构成的一轮交互 |
| 追问策略 | AI 根据用户回答继续深挖、澄清或推进下一题的规则 |
| 结构化反馈 | 练习结束后对用户回答的诊断，包含结构、表达、可理解度、示例和建议 |
| 复练建议 | 指向下一次练习重点的可执行建议 |

建议的信息流如下：

```text
场景选择
  -> 面试背景设置
  -> AI 面试官提问
  -> 用户语音回答
  -> AI 追问 / 收束推进
  -> 练习结束
  -> 结构化反馈
  -> Retry this question / Start another scenario
```

首期页面结构建议控制在 4 个页面：

1. 场景选择页：让用户直接选择要练习的面试类型。
2. 面试设置页：只保留少量背景选项，不要求上传简历。
3. 语音练习页：展示当前问题、用户转写、追问和语音输入入口。
4. 反馈报告页：展示问题定位、优化表达、总结和复练入口。

## 原型 / Demo

MS1 原型不需要覆盖完整产品，只需要验证“场景训练 + 反馈复练”的主链路。

建议 Demo 默认路径：

```text
Behavioral Interview
  -> 选择目标岗位和难度
  -> AI 提问：Tell me about a time you solved a difficult problem under pressure.
  -> 用户语音回答
  -> AI 追问：What was your specific contribution?
  -> 用户继续回答
  -> 完成 3 轮左右
  -> 展示反馈报告
  -> 点击 Retry this question
```

Demo 中必须体现的体验：

- 用户不是进入一个空白聊天框，而是先选择明确练习场景。
- AI 不是简单问答机器，而是能根据用户回答继续追问。
- 反馈不是一段泛文本，而是结构化定位问题。
- 反馈页必须给出下一步复练入口。

Demo 可以使用 mock 数据或固定脚本串联主流程，不要求在 MS1 完成所有真实模型调用。重点是让评审能看清产品判断和体验闭环。

## 验收标准

本 Proposal 在 MS1 阶段的验收标准如下：

1. 用户可以从至少 1 个英文面试场景进入练习，页面上能看出后续可扩展到其他场景。
2. 用户可以完成至少 3 轮英文面试问答，流程包含 AI 提问、用户回答和 AI 追问。
3. AI 在演示中保持面试官角色，不退化成普通陪聊或泛英语老师。
4. 至少 1 次追问需要基于用户上一轮回答，而不是固定模板下一题。
5. 练习结束后必须展示结构化反馈，不能只展示一段总结文字。
6. 结构化反馈至少包含回答结构、语言表达、可理解度、原句与优化表达、下一轮建议。
7. 发音相关反馈只提示影响理解的明显问题，不做音素级评分。
8. 用户看完反馈后，能明确知道下一轮优先练什么，并能进入 Retry 或下一场景。
9. Demo 可以使用 mock 或固定脚本，但必须清楚区分“已实现演示链路”和“后续真实模型能力”。
10. 产品边界中明确不做的内容不得在演示或 Issue 中被包装成本期目标。

## 核心假设与后续验证

本期 Proposal 依赖以下待验证假设：

- 目标用户会认为“英文技术面试表达”比泛英语口语更值得专项练习。
- 3-5 轮语音模拟足以暴露用户的主要表达和结构问题。
- 结构化反馈比单纯分数或鼓励更能驱动用户复练。
- 轻量题库 / 知识库能显著提升追问和反馈的稳定性。
- 用户愿意接受先不上传简历，仅通过轻量背景选项开始练习。

这些假设需要在后续通过用户访谈、原型试用和真实语音样本继续验证。MS1 不要求证明所有假设成立，但必须把假设讲清楚，并在后续 Issue 中拆出验证任务。
