# 语音 AI 传输协议调研：WebSocket 够不够快？

> 日期：2026-07-10
> 背景：团队已确定 WebSocket 为前后端通信协议（见[技术选型决策](./2026-07-09-技术选型决策.md)），但需要确认 WebSocket 是否足够快，以及各 ASR/TTS/S2S 厂商是否支持更快的协议
> 结论：WebSocket 不是瓶颈，真正需要关注的是架构选型（S2S vs 管线）和厂商 API 协议

---

## 一、核心结论（先说结论）

**WebSocket 对 UniSpeaking 当前阶段完全够用，不必焦虑。** 原因有三：

1. **绝大多数 ASR/TTS 厂商的云 API 只支持 WebSocket**，你换不了更快的协议，除非自建
2. **传输层延迟差异很小**（WebSocket vs WebRTC < 200ms），而 LLM 推理延迟是 500ms-4s，**传输层只占总延迟的 < 5%**
3. **WebRTC 的优势在客户端侧**（降噪、回声消除、弱网自适应），不在"更快"，如果走 S2S API（如 OpenAI Realtime），厂商端已经帮你处理了 WebRTC

**但有一个重要的区分**：这里讨论的是两层不同的"协议"——

| 层 | 是什么 | 你控制什么 |
|---|---|---|
| **你的前后端之间** | 浏览器 ↔ 你的 Go 后端 | 你选 WebSocket / WebRTC |
| **你的后端 ↔ AI 厂商 API** | 你的服务器 ↔ OpenAI/Deepgram/ElevenLabs 等 | 厂商定协议，你只能适配 |

**第一层**（前后端之间），WebSocket 目前选得对。**第二层**（调用厂商 API），取决于你选哪个厂商——下面有详细对比。

## 二、各厂商 API 协议全景

### 2.1 S2S 端到端语音（直接竞争对手参考）

| 厂商/模型 | 协议 | 延迟 | 备注 |
|---|---|---|---|
| **OpenAI GPT-Realtime-2** | **WebRTC** (GA) + WebSocket 备选 | 232-320ms | 2026年5月GA，WebRTC是主力，WebSocket是降级路径 |
| **Google Gemini Live** | **WebSocket** (原生) | 300-500ms | 无原生WebRTC，需通过LiveKit/Pipecat等第三方桥接 |
| **Azure Voice Live API** | **WebRTC** + WebSocket | 对标OpenAI | 基于GPT Realtime，包装了Azure Speech的降噪/VAD |
| **豆包 Seeduplex (火山引擎)** | **WebSocket** + 火山RTC(WebRTC) | 300-500ms | 2026年6月API邀测，WebSocket主力，WebRTC通过火山RTC实现 |
| **Qwen-Omni (阿里百炼)** | **WebSocket** | <150ms 语音响应 | API风格兼容OpenAI Realtime，支持VAD+打断 |
| **GLM-4-Voice (智谱)** | **WebSocket** | ~300ms | 半双工，支持打断 |
| **Step-Audio 2 (阶跃星辰)** | 开源，自部署 | ~80ms | 开源小模型，延迟极低但对话能力有限 |

**关键发现**：S2S 领域正在从 WebSocket 向 WebRTC 迁移，但 WebSocket 仍是绝大多数厂商的标配。OpenAI 是第一个把 WebRTC 作为主力协议的，Azure 跟进。但国内厂商（豆包、Qwen、GLM）目前都是 WebSocket。

### 2.2 ASR 厂商

| 厂商/模型 | 协议 | 延迟 | 备注 |
|---|---|---|---|
| **Deepgram Nova-3 / Flux** | **WebSocket** only | 实时流式 | 行业标杆，但只有WebSocket |
| **AssemblyAI Universal-Streaming** | **WebSocket** only | 实时流式 | 只有WebSocket |
| **Azure Speech Recognition** | **WebSocket** (Speech SDK) | 实时流式 | 传统SDK基于WebSocket；新Voice Live API支持WebRTC |
| **Google Chirp / STT V2** | **gRPC** + REST | 流式 | Google是少数用gRPC做流式ASR的厂商 |
| **AWS Transcribe** | **WebSocket** (boto3) | 流式 | WebSocket streaming |
| **ElevenLabs Scribe v2** | **WebSocket** | 实时 | WebSocket |
| **NVIDIA Riva** | **gRPC** streaming | 100-300ms | 自部署方案，gRPC双向流，延迟最低 |
| **豆包流式ASR (火山引擎)** | **WebSocket** 双向流式 | 实时 | 自定义二进制协议 |
| **腾讯实时ASR** | **WebSocket** | 实时 | 标准WebSocket |

**关键发现**：ASR 厂商几乎清一色 WebSocket。Google 和 NVIDIA 是少数用 gRPC 的，但 Google Cloud STT 的 gRPC 也是云端 API 形态。**在 ASR 层，你没有 WebRTC 可选。**

### 2.3 TTS 厂商

| 厂商/模型 | 协议 | 延迟 | 备注 |
|---|---|---|---|
| **ElevenLabs Flash v2.5** | **WebSocket** 双向流式 | ~75ms推理 + 网络 | auto_mode优化延迟，WebSocket唯一实时协议 |
| **Azure Speech TTS** | **WebSocket** (Speech SDK) | 流式 | 传统SDK；Voice Live API支持WebRTC |
| **MiniMax TTS (speech-2.8)** | **WebSocket** (t2a_v2) | 流式 | 事件驱动协议，task_start→task_continue→task_finish |
| **豆包语音合成 (火山引擎)** | **WebSocket** 双向流式 | 首包<300ms | 双流接口最优 |
| **小米 MiMo TTS** | 开源自部署 | 取决于部署 | 可自选协议 |

**关键发现**：TTS 厂商同样是 WebSocket 一统天下。ElevenLabs 和 MiniMax 都没有 WebRTC 接口。

## 三、WebRTC vs WebSocket：延迟到底差多少？

### 3.1 理论对比

| 维度 | WebRTC (UDP) | WebSocket (TCP) |
|---|---|---|
| **传输层** | UDP (SRTP) | TCP |
| **典型延迟** | 50-200ms | 100-500ms |
| **队头阻塞** | 无（UDP） | 有（TCP丢包重传阻塞后续所有包） |
| **连接建立** | 4-8 RTT（ICE/DTLS/SDP） | 2-3 RTT |
| **音频编解码** | Opus 内置 (~20-32kbps) | 需自实现，通常PCM/base64 (~512kbps) |
| **降噪/AEC** | 内置 | 需自实现 |
| **弱网适应** | 自适应码率+jitter buffer | 无 |
| **浏览器支持** | 所有浏览器 | 所有浏览器 |

### 3.2 但真实的端到端延迟是这样拆分的

```
用户说话结束
    │
    ├─ 客户端音频采集 + 缓冲          ~20-50ms
    ├─ 网络传输（WebSocket/WebRTC）    ~30-100ms  ← 传输层在这里
    ├─ VAD + 端点检测                  ~100-200ms
    ├─ ASR 转写（或S2S理解）           ~200-500ms
    ├─ LLM 生成回复文本                ~500-4000ms  ← 真正的瓶颈！
    ├─ TTS 合成语音（或S2S生成）       ~75-300ms
    ├─ 网络传输（WebSocket/WebRTC）    ~30-100ms  ← 传输层在这里
    ├─ 客户端缓冲 + 播放               ~20-50ms
    │
    ▼
用户听到回复：总延迟 1000-5000ms
```

**传输层（WebSocket vs WebRTC）的差异在最理想情况下约 100ms，最差情况（网络拥塞）约 300-500ms。而 LLM 推理延迟是 500ms-4000ms——前者在后者面前几乎可以忽略。**

### 3.3 一个真实的生产数据

2025 年某企业级语音系统实测：
- WebSocket 路径：中位响应延迟 **~1,920ms**
- WebRTC 路径：中位响应延迟 **~2,060ms**
- **几乎一样**，因为 LLM 推理时间主导了总延迟

> 来源：[Zoice.ai - Network Transport in Conversational AI](https://zoice.ai/blog/network-transport-in-conversational-ai/)

## 四、竞争对手用什么协议？

### 4.1 Speak（最强直接竞品）

Speak 是 AI 口语陪练的标杆产品（10M+ 下载，4.8 评分，OpenAI 投资）。

从公开信息和技术栈推断：
- **核心对话链路**：基于 OpenAI Realtime API（Speak 是 OpenAI 官方投资并深度合作的公司）
- **协议**：WebRTC（OpenAI Realtime API 的主力协议）
- **架构**：客户端 → WebRTC → OpenAI Realtime → WebRTC → 客户端播放
- **已知性能**：能够支撑"能一直聊"的实时口语对话，支持打断

**借鉴意义**：Speak 走的是 S2S 端到端路线，协议是 OpenAI 帮你选好的（WebRTC）。如果你也选 OpenAI Realtime API，协议层的选择是被动的——厂商已经选了 WebRTC。

### 4.2 ELSA Speak

- 更偏发音诊断，实时性要求低于 Speak
- 推测：ASR（采集后分析）→ 评分引擎 → TTS 反馈
- 不依赖极低延迟的实时对话，WebSocket 足够

### 4.3 Character.AI

- 2025 年开始推语音通话功能
- 架构未公开，推测也是 S2S 或 WebSocket 管线

### 4.4 可栗口语（国内直接竞品）

- 采用豆包融合技术
- 端到端响应 300-500ms
- 支持 Barge-in 打断与自然话轮切换
- 协议：豆包 API → WebSocket

## 五、那 WebSocket 真正的风险是什么？

**不是"慢"，而是另外三个问题**：

### 5.1 TCP 队头阻塞（Head-of-Line Blocking）

WebSocket 跑在 TCP 上，一个丢包会阻塞后续所有数据包。在弱网/WiFi 不稳定/移动网络下，用户感知的卡顿不是"延迟高"而是"突然卡一下"。

**缓解方案**：
- 前端做音频分片发送，不要一个大的音频块
- 后端做 jitter buffer
- MVP 阶段 Web 端用户通常在 WiFi 环境，问题不大
- 移动端上线前需要专项测试

### 5.2 无内置音频处理

WebSocket 只传数据，不处理音频。你需要自己实现：
- 回声消除（AEC）——用户听到 AI 语音时，麦克风不应把 AI 的声音录回去
- 降噪
- VAD

**但这些不是协议问题**——即使你用 WebRTC 传数据，如果你的 AI API 不吃 Opus 只吃 PCM，你还是得自己转码。

### 5.3 厂商 API 才是真正的约束

你的后端调用厂商 API 时，**厂商用什么协议你就得用什么**：

- 如果你选 OpenAI Realtime API → 厂商端是 WebRTC（你控制不了，也不需要控制）
- 如果你选 Deepgram ASR → 厂商端是 WebSocket
- 如果你选 ElevenLabs TTS → 厂商端是 WebSocket

**你的前后端之间用 WebSocket 完全没问题，因为真正"慢"的环节是 AI 推理，不是你跟服务器之间的网络。**

## 六、对 UniSpeaking 的具体建议

### 6.1 协议层决策矩阵

| 链路层 | 当前选择 | 评估 | 建议 |
|---|---|---|---|
| 浏览器 ↔ Go 后端 | WebSocket | ✅ 正确 | 保持，v1 不需要改 |
| Go 后端 ↔ S2S API（如果走端到端） | 取决于厂商 | 厂商决定 | OpenAI→WebRTC; 豆包/Qwen/GLM→WebSocket |
| Go 后端 ↔ ASR API（如果走管线） | 取决于厂商 | WebSocket为主 | Deepgram/AssemblyAI/豆包都是WebSocket |
| Go 后端 ↔ TTS API（如果走管线） | 取决于厂商 | WebSocket为主 | ElevenLabs/MiniMax/豆包都是WebSocket |

### 6.2 分阶段建议

**第一阶段（MVP，Web端）**：
- 前后端之间：**保持 WebSocket**，Go 后端实现成本最低
- 后端到 AI：选 OpenAI Realtime（WebRTC）或 Qwen-Omni/豆包（WebSocket）
- 不需要引入 WebRTC 到你的前后端——让厂商 API 处理音频协议

**第二阶段（产品化）**：
- 如果在移动端发现弱网体验问题，考虑在前端引入 WebRTC
- 可以选择 LiveKit / Pipecat 等框架做客户端侧的 WebRTC → 后端转 WebSocket → AI API
- 这不是"WebSocket 太慢"的问题，而是"移动端弱网需要 WebRTC 的媒体处理能力"

**第三阶段（规模化）**：
- 如果走管线架构（ASR→LLM→TTS），各模块之间的 WebSocket 延迟不是瓶颈
- 瓶颈在 LLM 推理，优化方向应该是：流式 LLM 输出 + TTS 并行生成，而不是换协议
- 如果自建 NVIDIA Riva → 用 gRPC，可以比 WebSocket 再省 ~50-100ms

### 6.3 一个务实的架构建议

```
┌─────────────────────────────────────────────────────────┐
│  浏览器 (Web)                                            │
│  MediaRecorder API / AudioContext                        │
│  → PCM 16kHz 音频分片 → WebSocket →                     │
└─────────────────────────────────────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────────┐
│  Go 后端 (WebSocket endpoint)                            │
│  → 接收音频 → VAD → 转发至 AI API                        │
│  → 接收 AI 回复 → 转发回浏览器                           │
└─────────────────────────────────────────────────────────┘
                            │
              ┌─────────────┼─────────────┐
              ▼             ▼             ▼
        ┌──────────┐ ┌──────────┐ ┌──────────┐
        │ S2S API  │ │ ASR API  │ │ TTS API  │
        │ (WebRTC  │ │(WebSocket│ │(WebSocket│
        │  或 WS)  │ │  only)   │ │  only)   │
        └──────────┘ └──────────┘ └──────────┘
```

**核心原则：前后端用 WebSocket 不变；后端到 AI API 的协议由厂商决定；如果要优化延迟，优先换 S2S 方案而非换协议。**

### 6.4 还要把“接入难度”纳入评估

协议快不快只是一部分，真正落地时还要看 API 调用和 SDK 封装的接入成本。不同厂商即便协议相同，开发速度也会不一样，常见差异包括：
- 是否需要自己拼事件流、状态机和断线重连
- SDK 是否把鉴权、心跳、转写回调、打断事件都封装好了
- 文档和示例是否能一次跑通，还是要反复试错

**建议测试口径统一成四项**：首包延迟、首轮对话接通时间、集成代码量、调试轮次。这样最后选型不只看“协议名义上的快”，也看“接入到底快不快”。

## 七、如果真的要追求极致低延迟

如果你的产品定位需要比 Speak 更好的体验（如竞品杀手锏报告中提到的"更低延迟"目标），以下是真正有效的优化路径，按优先级排列：

| 优先级 | 优化方向 | 预期收益 | 实现成本 |
|---|---|---|---|
| **P0** | 选 S2S 端到端方案（OpenAI Realtime / Qwen-Omni），跳过 ASR→LLM→TTS 串行 | -500~1000ms | 低（换API） |
| **P1** | LLM 流式输出 + TTS 并行生成（边生成边合成） | -300~600ms | 中 |
| **P2** | 客户端预处理（VAD提前截断静音、降噪） | -100~200ms | 低 |
| **P3** | 部署边缘节点，缩短网络 RTT | -50~150ms | 高（运维） |
| **P4** | 前后端之间换 WebRTC | -50~100ms | 中 |
| **P5** | 自建 NVIDIA Riva（ASR gRPC）+ 自部署 TTS | -100~200ms | 高（GPU+运维） |

**注意：P4（前后端换 WebRTC）排在 P0-P3 之后。先把架构选对，再抠传输层。**

## 八、总结

1. **WebSocket 不是瓶颈**——传输层延迟占总延迟 <5%，LLM 推理才是大头
2. **ASR/TTS 厂商几乎全是 WebSocket**——Deepgram、AssemblyAI、ElevenLabs、MiniMax 无一例外
3. **WebRTC 正在 S2S 层兴起**——OpenAI 和 Azure 率先迁移，国内厂商还是 WebSocket
4. **Speak 用 OpenAI Realtime (WebRTC)**——但协议是厂商端的事，你不需要在自己前后端之间引入 WebRTC
5. **真正应该关注的**：选 S2S 还是管线架构？选哪个 S2S 厂商？—— 这比协议选择重要 10 倍

**一句话**：别纠结 WebSocket 快不快。把精力放在选对 S2S 厂商上——OpenAI Realtime（WebRTC，英文最好但贵）、Qwen-Omni（WebSocket，国产性价比）、还是豆包 Seeduplex（WebSocket，全双工最先进）。
