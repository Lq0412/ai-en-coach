# UniSpeaking 国产化实时语音方案

> 日期：2026-07-10  
> 约束：国内可用，国产厂商为主  
> 目标平台：Web 端（MVP）→ 移动端（后续）  
> 核心取舍：既要实时对话流畅，又要拿到文本做教学反馈

---

## 一、一句话方案

**用 Qwen-Omni S2S 做主链路（开转写拿文本），MiniMax TTS 做备选发音，Web MVP 用 WebSocket 直连，移动端阶段引入 LiveKit 做 WebRTC 桥接。**

---

## 二、为什么是这个架构

### 2.1 国产 S2S 里，Qwen-Omni 是唯一同时满足"低延迟 + 给文本"的

| 国产 S2S 候选 | 延迟 | 能拿文本吗 | API 成熟度 | 结论 |
|---|---|---|---|---|
| **Qwen-Omni (3.5 Flash Realtime)** | <150ms | ✅ `enable_input_audio_transcription=True` 即可 | GA，Go 侧需做一层适配 | **首选** |
| 豆包 Seeduplex | 300-500ms | 未确认 | 2026年6月才开邀测 | 稳定性存疑，备选 |
| GLM-4-Voice | ~300ms | 未确认 | 半双工 | 体验不如全双工 |
| Step-Audio 2 | ~80ms | 开源自建 | 需要自己部署 | 延迟最优但要运维 |

**Qwen-Omni 的关键能力**——开启转写后在同一个 WebSocket 连接里同时拿到：

```python
# 用户说的话 → 自动转写为文本
{"type": "conversation.item.input_audio_transcription.completed",
 "transcript": "I think learning English is really fun"}

# AI 的回复 → 同时出语音流和文本
{"type": "response.audio_transcript.delta", "delta": "That's great! "}
{"type": "response.audio.delta", "audio": "<base64 pcm>"}
```

**一个 WebSocket 连接，同时得到：低延迟语音 + 用户说了什么文本 + AI 回复了什么文本。** 这解决了 S2S "黑盒拿不到文本" 的问题。

### 2.2 为什么不走纯管线（ASR→LLM→TTS）

管线架构可控性更强，但国产方案存在一个具体问题：**LLM 和 TTS 之间多一次网络往返**。

国产 LLM API（DeepSeek/Qwen/豆包）都是 REST streaming，不是 WebSocket。这意味着：

```
管线：ASR(WS) → 等出文本 → LLM(REST SSE) → 等出token → TTS(WS) → 等出音频
      一次WS建连   一次HTTP建连   一次WS建连
      三次网络往返，三次建连开销
      
Qwen-Omni S2S：一次 WS 建连，音频进，音频+文本出
```

在国内外网环境下，这个差异会被放大——每多一次建连就多一次 DNS 解析 + TCP 握手 + TLS，在国内网络下可能多 50-150ms。

## 三、具体技术方案

### 3.1 MVP 阶段（Web 端，WebSocket）

```
┌──────────────────────────────────────────────────────────┐
│  浏览器 (Web)                                              │
│                                                          │
│  getUserMedia({echoCancellation: true, noiseSuppression}) │
│  → PCM 16kHz mono → MediaRecorder/AudioWorklet 分片       │
│  → WebSocket (wss://your-server/ws/conversation)          │
│                                                          │
│  播放：AudioContext → 接收 PCM 音频块 → 排队播放            │
│  打断：用户开口 → 检测音量 → 发送 interrupt → 停止播放      │
└──────────────────────────┬───────────────────────────────┘
                           │  WebSocket (PCM 16kHz)
                           ▼
┌──────────────────────────────────────────────────────────┐
│  Go 对话编排层                                              │
│                                                          │
│  /ws/conversation                                        │
│    │                                                     │
│    ├─ 音频队列 → Qwen-Omni Realtime WS                    │
│    │   wss://{workspace}.cn-beijing.maas.aliyuncs.com/   │
│    │   api-ws/v1/realtime                                │
│    │                                                     │
│    ├─ 接收: response.audio.delta → 转发射频到浏览器        │
│    ├─ 接收: response.audio_transcript.delta → 存文本      │
│    ├─ 接收: input_audio_transcription.completed → 存转写   │
│    │                                                     │
│    └─ 课后：文本 + 转写 → 生成反馈报告                      │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

**关键代码结构**：

```python
# Go 侧（示意）

from dashscope.omni_realtime import OmniRealtimeConversation

@router.websocket("/ws/conversation")
async def conversation(ws: WebSocket):
    await ws.accept()
    
    # 建立到 Qwen-Omni 的 WebSocket
    qwen = OmniRealtimeConversation(
        model="qwen3.5-omni-flash-realtime",
        url="wss://{workspace}.cn-beijing.maas.aliyuncs.com/api-ws/v1/realtime",
    )
    
    # 关键配置：同时输出音频和文本
    qwen.update_session(
        modalities=["TEXT", "AUDIO"],
        voice="Cherry",  # 英文学音色，有49种可选
        enable_input_audio_transcription=True,
        input_audio_transcription_model="gummy-realtime-v1",
        enable_turn_detection=True,  # 自动检测用户说完
        instructions="""You are an English speaking coach..."""
    )
    
    transcript_buffer = []
    
    async for audio_chunk in ws.iter_bytes():
        # 喂音频给 Qwen-Omni
        qwen.send_audio(audio_chunk)
        
        # 收 Qwen-Omni 的回调
        async for event in qwen.receive():
            if event.type == "response.audio.delta":
                # 音频块直转浏览器
                await ws.send_bytes(event.audio)
            
            elif event.type == "response.audio_transcript.delta":
                # AI 说了什么文本 → 存起来
                transcript_buffer.append(event.delta)
            
            elif event.type == "input_audio_transcription.completed":
                # 用户说了什么 → 存起来做课后分析
                save_user_utterance(event.transcript)
```

### 3.2 备选链路（当 S2S 控制力不够时）

有些教学策略（如隐性纠错、难度适配、expression 替换建议）可能需要更精细的文本层控制。这时候切管线：

```
┌─────────┐     ┌───────────┐     ┌──────────┐
│ Qwen3   │     │ DeepSeek  │     │ MiniMax  │
│ ASR     │ ──→ │ V3 / Qwen │ ──→ │ TTS      │
│ Flash   │     │ 3-Max     │     │ 2.8 turbo│
│Realtime │     │ (REST SSE)│     │  (WS)    │
└─────────┘     └───────────┘     └──────────┘
 WebSocket        REST 流式         WebSocket

延迟: 100ms   +   300-500ms    +   100-200ms  =  500-800ms
```

**切换条件**：当需要精确控制 LLM 的 prompt（比如"用户说了 X，这轮对话的策略应该是降低难度到 Y 级别"）时，走管线。S2S 走 prompt 控制不是每次都灵。

### 3.3 厂商选型速查

| 环节 | 首选 | 备选 | 价格参考 | 协议 |
|---|---|---|---|---|
| **S2S 主链路** | 阿里 Qwen3.5-Omni-Flash-Realtime | 豆包 Seeduplex（等GA） | 阿里: ¥82/百万 token 输入 | WebSocket |
| **ASR（管线备选）** | 阿里 Qwen3-ASR-Flash-Realtime | 腾讯实时 ASR | Qwen ASR: 待确认; 腾讯: ¥1-3/小时 | WebSocket |
| **LLM（管线备选）** | DeepSeek-V3 | 阿里 Qwen3-Max | DeepSeek: ¥1/百万 token 入 | REST SSE |
| **TTS（管线备选）** | MiniMax speech-2.8-turbo | 豆包 TTS 2.0 | MiniMax: ¥2/万字符; 豆包: 150元/年/音色 | WebSocket |
| **发音评测（后期）** | 科大讯飞 口语评测 | 驰声 Chivox | 商务询价 | REST |
| **WebRTC 网关（移动端）** | LiveKit (自建) | Janus SFU | 开源免费，服务器成本 | WebRTC↔WS |

**补充一个实际落地维度：接入难度要单独测。** 同样是“能用”，不同厂商的 API/SDK 封装程度、鉴权方式、事件模型、错误码和流式回调差异很大，最终会影响：
- 首次接通时间（从拿到 key 到跑通首轮对话）
- 代码复杂度（是直接 API 调用，还是要封装 SDK 事件循环）
- 调试成本（日志、断线重连、打断、转写回调是否好排查）

**建议把它和延迟一起做 PoC 对比**：同一台机器、同一段录音、同一套前端，分别测“首包延迟 / 首轮对话跑通时间 / 一小时内稳定性 / 接入代码行数”。

## 四、延迟预期

### 4.1 S2S 主链路（Qwen-Omni）

```
用户开始说话
    │
    ├─ 说话中...（Qwen-Omni 已在接收，simultaneous processing）
    │
用户说完最后一个词
    │
    ├─ VAD 判停（enable_turn_detection）       ~100-200ms
    ├─ Omni 模型推理 + 生成语音                 ~100-200ms（阿里宣称<150ms）
    ├─ 网络往返                                  ~30-50ms（国内阿里云）
    │
    ▼
用户听到 AI 首句语音：                          ~250-450ms ✅
```

### 4.2 管线备选链路

```
用户说完 → ASR finalize(100ms) → LLM首token(300ms) → TTS首块(150ms) + 网络(50ms)
= ~600ms
```

**考虑到国内网络环境（服务器在阿里云/火山引擎同区域部署），实际延迟会比海外方案更低。**

### 4.3 对比竞品

| 方案 | 延迟 | 备注 |
|---|---|---|
| Speak (OpenAI S2S) | ~300-500ms | 海外用户，走Cloudflare边缘节点 |
| 可栗口语 (豆包) | ~300-500ms | 国内用户，火山引擎 |
| **UniSpeaking (Qwen-Omni)** | **~250-450ms** | 国内用户，阿里云，同区域 |
| UniSpeaking (管线备选) | ~500-800ms | DeepSeek + MiniMax |

## 五、移动端阶段：什么时候引入 WebRTC

### 5.1 现状

Web MVP 阶段，用户用浏览器在 WiFi 下使用，WebSocket 够用。浏览器 `getUserMedia({echoCancellation: true})` 做基础回声消除。

### 5.2 触发条件

出现以下任一信号时，引入 WebRTC 网关：
- 移动端 App 开始开发
- Web 端用户反馈"地铁上/外面用的时候经常卡"
- 用户量上来了，发现在非 WiFi 环境下的会话完成率显著低于 WiFi

### 5.3 引入方式

**推荐 LiveKit**（开源，Go 实现，国内可部署）：

```
移动端 App                           阿里云 ECS
─────────                          ─────────────
│         │  WebRTC (UDP/Opus)    │           │  WebSocket   ┌──────────┐
│ LiveKit │ ───────────────────→ │  LiveKit  │ ──────────→ │ Qwen-Omni│
│  SDK    │ ←─────────────────── │  Server   │ ←────────── │ (阿里云)  │
│         │                       │           │             └──────────┘
└─────────┘                       └──────────┘
                                   Opus ↔ PCM 转码在这里做
```

**为什么是 LiveKit 而不是自建 SFU**：
- 开源，Go 写的，一个二进制文件部署
- 自带 WebRTC 的 STUN/TURN、房间管理、录音
- 有 React Native SDK，移动端直接接入
- 国内 ECS 部署延迟可控
- 比自建 Janus/mediasoup 省 1-2 周开发时间

**成本**：一台 4C8G 阿里云 ECS 跑 LiveKit，约 ¥300-500/月，支撑几百并发没问题。

### 5.4 如果不引入 WebRTC 会怎样

移动端用 WebSocket 裸跑 PCM 音频：
- WiFi 环境：没问题
- 4G/5G 不稳定时：偶发卡顿（TCP 重传），但不会崩
- 地铁/电梯/切换基站：丢包率上去后，用户体验显著下降

**务实判断**：MVP 阶段不引入 WebRTC，先把产品跑起来。移动端开发启动时，花一周搭 LiveKit。

## 六、MVP 开发路径（4 周）

```
Week 1: 搭通 Qwen-Omni 主链路
  □ Go WebSocket endpoint
  □ Qwen-Omni Realtime WebSocket 接入（百炼）
  □ 配置 enable_input_audio_transcription
  □ 前端 WebSocket 音频收发
  □ 验证：浏览器说英语 → AI 英语回复 → 浏览器播放

Week 2: 对话体验打通
  □ System prompt 调教（回复长度、难度、追问策略、隐性纠错）
  □ 前端状态机：录音中 / 识别中 / 思考中 / 播放中
  □ 打断逻辑：前端音量检测 → interrupt 信号 → 停止播放
  □ 用户转写文本展示（可选显示）
  □ 验证：3-5 分钟自由对话无卡顿

Week 3: 课后反馈 + 管线备选
  □ 对话记录存储（转写文本 + AI 回复文本）
  □ 基础课后反馈：开口时长、说了多少轮、关键表达提示
  □ 搭建管线备选链路（Qwen ASR → DeepSeek → MiniMax）
  □ 策略路由：默认 S2S，特定场景切管线
  □ 验证：结束对话后看到反馈

Week 4: 打磨 + 内测
  □ 前端 UI 打磨（通话式界面）
  □ 延迟压测（P50/P95）
  □ 打断成功率测试
  □ 内部 3-5 人真实使用
  □ 验证：北极星指标"有效开口时长"可测量
```

## 七、成本估算

### 7.1 MVP 阶段（假设 10 人内测，每人每天 10 分钟）

| 项目 | 月消耗 | 月成本 |
|---|---|---|
| Qwen-Omni API（音频输入） | ~500 万 token | ~¥400 |
| Qwen-Omni API（音频输出） | ~300 万 token | ~¥250 |
| 阿里云 ECS（2C4G） | 1 台 | ~¥200 |
| **合计** | | **~¥850/月** |

### 7.2 上线后（假设 100 DAU，每人 10 分钟）

| 项目 | 月消耗 | 月成本 |
|---|---|---|
| Qwen-Omni API | ~3 亿 token | ~¥2-3 万 |
| 备选管线（DeepSeek + MiniMax） | 溢出流量 | ~¥0.2-0.5 万 |
| 阿里云 ECS × 2 | 2 台 | ~¥500 |
| LiveKit Server（移动端后） | 1 台 ECS | ~¥400 |
| **合计** | | **~¥2.5-3.5 万/月** |

### 7.3 降本路径

- Qwen-Omni 模型也在持续降价——阿里云的通义系列历史上降价幅度很大
- 如果后续用户量大，可考虑自建 Qwen3-ASR + Qwen LLM + CosyVoice TTS，全部开源免费
- MiniMax speech-2.8-turbo 比 HD 便宜近一半（¥2 vs ¥3.5/万字符），实时对话场景 Turbo 够用

## 八、一句话决策

> **主链路：Qwen-Omni S2S（开转写拿文本）→ WebSocket → 浏览器。备选管线：Qwen ASR + DeepSeek + MiniMax TTS。移动端阶段：加一层 LiveKit WebRTC 网关。国产全链路，延迟 250-450ms，文本可控，成本可控。**
