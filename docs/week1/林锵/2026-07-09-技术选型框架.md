## 三、新增选型详解

### 3.1 视频生成（Pivot 后新增，优先级最高）

这是产品差异化的核心——面试官视频的逼真度直接决定用户体验。

| 候选 | 逼真度 | API | 中文支持 | 成本（估算） | 备注 |
|---|---|---|---|---|---|
| **HeyGen** | ⭐⭐⭐⭐⭐ | ✅ | ✅ 优秀 | ~$2-5/min | 行业标杆，口型同步最好 |
| **Synthesia** | ⭐⭐⭐⭐ | ✅ | ✅ | ~$1-3/min | 企业级，模板丰富 |
| **D-ID** | ⭐⭐⭐ | ✅ | ✅ | ~$0.5-1/min | 成本低，逼真度略逊 |
| **自研开源** | ⭐⭐ | — | 需调优 | 人力成本高 | 不推荐 MVP 阶段做 |

**MVP 推荐**：HeyGen API 批量生成核心题库视频，验证体验后再优化成本。

**关键决策点**：
- API 调用方式（同步/异步、批量生成时间）
- 是否支持"输入脚本 → 生成视频"的自动化管线
- 视频存储与 CDN 成本

### 3.2 评分引擎架构（Pivot 后新增，差异化核心）

三选一或混合：

```
方案 A：纯 LLM 评分
  用户录音 → ASR → transcript → LLM(rubric) → 分数+评语
  优势: 简单，语义理解强
  劣势: 一致性差，无法精确定位"哪句话有问题"

方案 B：语音切片 + 维度评分
  用户录音 → ASR → 按句子切段 → 每段独立评分(发音/流利/语法)
           → LLM 语义评分(结构/说服力) → 融合报告
  优势: 定位精准，可追溯
  劣势: 切片策略需要调优

方案 C：对比诊断
  用户录音 → ASR → LLM 生成"理想版本" → 逐段对比差异 → 诊断报告
  优势: 反馈最具体，"你说X → 更好的说法是Y"
  劣势: 依赖 LLM 生成质量，成本较高
```

**MVP 推荐**：**B + C 混合**。切片定位问题句子，LLM 生成对比诊断。不对用户说"78 分"，而是说"第 3 句缺少 STAR 结构，建议改成..."

**本周要回答的技术问题**：
- 切片策略怎么定？（静音检测 vs 语义切分 vs 固定时间窗）
- 同一段话 LLM 评两次，分数波动多大？（一致性测试）
- 评分延迟能接受多少？（影响实时反馈 vs 事后报告的选择）

## 四、1024XEngineer 可复用的模式

| 模式 | 来源 | 怎么用 |
|---|---|---|
| **分层架构** | ByteMind `internal/` 组织方式 | 后端按 `engine/scoring/video/session/` 拆模块 |
| **多 Provider 路由** | ByteMind `provider/` 包 | LLM 选型用统一路由，方便切换和对比 |
| **Gateway 中继** | NeoCode / AnyClaw | 如果后续做移动端/微信端，JSON-RPC Gateway 不改后端 |
| **Task-Centric 设计** | CialloClaw | 面试 = Task，每轮回答 = Step，可恢复、可追踪 |
| **CLAUDE.md 行为规范** | ByteMind | 直接复用，控制 AI 辅助开发的质量 |
| **PRD + 架构先于代码** | ByteMind + CialloClaw | 技术选型之前，先把产品边界和架构分层写清楚 |
| **Evals 驱动开发** | ByteMind | 评分引擎用 eval 集驱动——先定义"什么叫评得好"，再调代码 |
| **JSON-RPC 2.0 协议** | CialloClaw `packages/protocol/` | 前后端共享协议定义，接口稳定 |
