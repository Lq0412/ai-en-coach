# [提案] 简历与面试背景管理功能范围

> 关联：[Issue #9](https://github.com/1024XEngineer/XE3-ESL/issues/9) · [最新在线原型](https://speakup-product-prototype.wendymcdonald606998.chatgpt.site/)

## 动机 / 用户故事

高价值面试问题需要建立在目标岗位和真实经历上。只填写一个岗位名称容易得到泛化问题，强制用户先整理并上传简历又会拖慢开始练习。因此，产品需要让用户快速输入岗位，再逐步补充岗位描述、简历亮点、考察范围和其他要求。

典型用户故事：

- 作为一名候选人，我希望直接说或输入“我要面试某个岗位”，然后再检查系统生成的面试方案。
- 作为一名有简历的用户，我希望选择已保存简历，让系统围绕真实项目和职责提问。
- 作为一名暂时没有可上传简历的用户，我希望通过目标岗位、岗位描述和训练重点继续创建面试。
- 作为一名准备多个岗位的用户，我希望保存少量简历、设置默认简历，并在本次面试中切换。
- 作为一名文件解析失败的用户，我希望看到明确原因，并能换文件或继续手动编辑背景。

一句话概括：

**面试背景决定问题相关性；简历用于增强个性化，但创建流程必须允许用户先从岗位开始。**

## 目标用户

- 准备英文技术相关岗位面试、希望快速生成方案的求职者。
- 有 PDF、DOC 或 DOCX 简历，希望使用真实经历进行追问的用户。
- 没有可上传简历，但能够填写岗位描述、经历亮点或训练重点的用户。
- 同时准备少量岗位，需要切换简历的用户。

## 现有做法及其不足

| 做法 | 优点 | 主要不足 |
|---|---|---|
| 只选择岗位名称 | 开始最快 | 问题容易泛化，无法围绕真实经历深挖 |
| 强制上传并完整解析简历 | 个性化强 | 阻塞无简历或不愿上传的用户，失败路径复杂 |
| 先输入岗位，再编辑背景并可选简历 | 门槛低，信息可逐步补充 | 需要明确字段来源和冲突处理 |

选择第三种方式，与最新版原型“输入职位 → 生成方案 → 编辑确认”的路径一致。

## 本期范围与明确不做

### 两个月最终范围

#### 创建面试的背景字段

- 第一步接受自然语言或文本输入，至少识别目标岗位。
- 生成方案后允许编辑：岗位名称、岗位描述、简历优势亮点、考察范围、补充要求。
- 所有生成字段都必须由用户确认后再用于生成面试官和问题。
- 未选择简历时，用户仍可通过岗位名称以及可选的岗位描述、优势亮点和考察范围继续。
- 如果简历内容与用户手动编辑冲突，以本次确认后的字段为准。

#### 简历管理

- 支持上传、查看文件名/大小/更新时间、设为默认、切换和删除。
- 两个月最终支持 PDF、DOC、DOCX，单文件最大 10 MB；MS2 先交付 PDF，MS3 补 DOC/DOCX。
- 每个用户最多保存 3 份简历；达到上限时提示先删除或替换。
- 状态至少包括：上传中、解析中、可用、解析失败。
- 解析失败后允许重新上传、换文件或继续手动编辑面试背景。
- 简历用于生成相关问题和反馈，不生成“是否能通过面试”的结论。

### Milestone 实现深度

| Milestone | 做到什么程度 |
|---|---|
| MS1 | 展示完整创建与简历管理原型；使用一份预置简历或手动背景，文件解析可 Mock |
| MS2 | 真实支持 1 份文本型 PDF 的上传、解析、选择和删除；无简历时可继续创建 |
| MS3 | 支持最多 3 份简历、默认/切换，以及 DOC/DOCX；补齐解析失败与替换流程 |
| MS4 | 完成大小/格式限制、异常提示、历史快照、删除验证、隐私说明和兼容性测试 |

### 明确不做

- 不做扫描件 OCR；无法提取文本时给出失败提示并允许手动继续。
- 不做复杂字段级简历编辑和人工校正界面。
- 不做简历模板、润色、翻译、版本对比、导出或公开分享。
- 不做 ATS 打分、职位匹配、岗位搜索和自动投递。
- 不建立职位收藏库或招聘信息聚合；岗位描述只服务本次面试。
- 不承诺支持 PDF/DOC/DOCX 之外的文件，也不承诺解析所有复杂排版。

## 关键决策与依据

### 决策 1：目标岗位必填，其余背景逐步增强

“先用一句话开始”能降低创建门槛；生成后的结构化方案又让用户在开始面试前纠正系统理解。目标岗位是唯一必填信息，岗位描述、简历亮点、考察范围和补充要求可以由系统生成后再编辑。

### 决策 2：最终支持三种格式，按 Milestone 分阶段交付

最新版原型明确展示 PDF、DOC、DOCX 和 10 MB 限制，因此正式两个月范围与原型保持一致。为控制实现风险：

| 阶段 | 支持格式 | 原因 |
|---|---|---|
| MS2 | PDF | 先验证文件上下文能否提升问题相关性 |
| MS3 | DOC、DOCX | 在核心闭环可用后补齐原型承诺 |

开发版必须如实展示当前支持格式，不能在 MS2 声称三种格式已经可用。

### 决策 3：最多保存 3 份简历

三份足以覆盖用户针对少量岗位的版本切换，同时限制文件存储和状态管理复杂度。此上限属于产品规则，需要在达到上限时明确提示，而不是静默失败。

### 决策 4：简历文本只作为面试上下文

系统可以提取项目、职责、技能和结果等线索，但不建设结构化招聘档案，也不评价简历质量。如何摘要、截断、检索和保护文本属于架构决策，不能简单承诺每次将全文无条件发送给模型。

## 基本概念与信息结构

| 概念 | 定义 |
|---|---|
| Resume File | 用户上传的 PDF、DOC 或 DOCX 原文件 |
| Resume Text | 从文件提取、可作为面试上下文的文本 |
| Resume Status | 上传中、解析中、可用或解析失败 |
| Default Resume | 创建面试时默认选中的一份可用简历 |
| Interview Context | 本次确认后的岗位名称、岗位描述、简历亮点、考察范围和补充要求 |
| Context Source | 字段来自用户输入、简历提取或系统生成的信息标记 |

```text
User
  └─ Resume 0..3

Interview Context
  ├─ Target Role（必填）
  ├─ Position Description（可编辑）
  ├─ Resume Highlights（可编辑）
  ├─ Assessment Scope（可编辑）
  ├─ Extra Requirement（可编辑）
  └─ Selected Resume（可选）
```

历史面试应保存本次确认后的上下文快照，避免后续修改或删除简历导致历史报告失去依据；具体存储方式由架构 Proposal 决定。

## 验收标准

### Proposal 定稿验收

- [ ] 团队确认目标岗位是唯一必填背景，其他字段可生成后编辑。
- [ ] 团队确认两个月最终支持 PDF、DOC、DOCX，最大 10 MB，并按 MS2/MS3 分阶段实现。
- [ ] 团队确认每个用户最多保存 3 份简历。
- [ ] 团队确认简历只服务面试上下文，不建设简历编辑、ATS 或招聘工具。
- [ ] 团队确认历史面试保存已确认背景快照。

### 两个月产品行为验收

1. 用户输入非演示岗位后，可以生成并编辑与该岗位一致的面试背景。
2. 用户未选择简历时仍可确认背景并继续创建面试。
3. 用户上传合法文件后能看到状态；解析成功后可在本次面试中选择。
4. 不支持格式、超过 10 MB 或无法解析时有明确提示，并可换文件或手动继续。
5. 用户最多保存 3 份简历，可以设为默认、切换和删除。
6. 生成的面试方案至少使用一个经用户确认的背景点，且不会虚构简历经历。
7. 删除或修改简历后，已完成面试仍能查看当时使用的背景快照。
