# 新 Issue：简历与职位背景管理功能范围

> - 建议标题：`[提案] 简历与职位背景管理功能范围`
> - 建议标签：`proposal`、`FullSpec`
> - 建议 Milestone：MS1（战略决策）
> - 关联：Issue #9
> - 草稿状态：待团队审核，尚未 Accepted

## 动机 / 用户故事

技术面试中的高价值追问必须建立在用户的目标岗位和真实项目背景上。只选择“后端工程师”会得到通用问题；要求用户先维护复杂简历又会提高开始练习的门槛。因此需要支持简历个性化，同时允许没有简历的用户快速开始。

典型用户故事：

- 例如，作为一名后端工程师，我希望上传简历后，AI 围绕我真实做过的订单系统、消息队列和高并发项目继续追问；其他技术岗位应使用各自简历中的项目和专业内容。
- 作为一名暂时不想上传简历的用户，我希望填写少量背景也能开始练习，而不是被上传步骤阻塞。
- 作为一名准备多个岗位的用户，我希望保存几份简历并选择本次使用哪一份。
- 作为一名上传失败的用户，我希望看到明确错误并能改用手动背景，而不是卡在创建流程。

一句话概括：

**简历和职位背景用于提高追问相关性，但产品不能因此变成简历编辑或招聘工具。**

## 目标用户

- 有 PDF 简历，希望进行个性化英文技术面试训练的求职者。
- 暂无可上传简历，但能提供目标岗位、经验年限和目标公司类型的用户。
- 同时准备少量不同技术岗位，需要切换简历的用户。

## 现有做法及其不足

当前文档和原型存在几处边界不一致：

- 有的描述要求上传 PDF 全文，有的描述又写“不要求上传简历”。
- 当前原型提示支持 PDF、DOC、DOCX，但早期 Proposal 只确定 PDF。
- 当前原型有默认简历、重命名和删除入口，但没有说明最多保存几份、解析失败怎么办。
- 只有职位名称和简历摘要，没有明确 JD 是否属于本期。

这些不一致会直接影响文件上传、解析、存储、面试创建和失败兜底的架构范围，需要先在产品层确定。

## 本期范围与明确不做

### 两个月最终范围

#### 简历管理

- 简历为可选项，不上传也能开始面试。
- 仅支持可提取文本的 PDF，单文件最大 10 MB。
- 每个用户最多保存 3 份简历。
- 支持上传、查看文件名和解析状态、重命名、设为默认、切换和删除。
- 解析状态至少包括：上传中、解析中、可用、解析失败。
- 解析失败时允许删除、重新上传，或切换到手动背景继续。
- 用户删除简历后，后续新面试不再使用该简历；历史会话如何保留快照由架构 Proposal 明确。

#### 职位背景

- 创建面试时，目标岗位为必填项。
- 用户可以粘贴一份可选 JD，帮助系统理解职责、技术栈和岗位要求。
- 用户可以填写可选训练重点，如“项目取舍”“岗位核心能力”“案例分析”。
- 未选择可用简历时，必须填写三项快速背景：目标岗位、经验年限、目标公司类型。
- 简历和 JD 只作为提问与反馈上下文，不直接生成“是否能通过面试”的结论。

### Milestone 实现深度

| Milestone | 做到什么程度 |
|---|---|
| MS1 | 原型展示完整简历管理和失败状态；演示使用 1 份预置简历或三项手动背景，不实现真实上传解析 |
| MS2 | 支持真实上传、解析、使用和删除 1 份文本型 PDF；失败时可用手动背景继续 |
| MS3 | 支持最多 3 份简历、重命名、设为默认、切换；创建面试时支持粘贴 JD |
| MS4 | 完成文件限制、异常提示、删除验证、隐私说明和典型 PDF 兼容性测试 |

### 明确不做

- 不支持 DOC、DOCX、图片简历或其他文件格式。
- 不做扫描 PDF 的 OCR；无法提取文本时明确失败并提供手动兜底。
- 不做教育、经历、技能等复杂字段级结构化解析和人工校正界面。
- 不做简历编辑、模板、润色、翻译、版本对比和导出。
- 不做 ATS 打分、职位匹配、岗位搜索和自动投递。
- 不建立职位收藏库或招聘信息聚合；JD 只服务当前面试创建。
- 不把整份简历作为公开内容，不提供简历分享或社区展示。

## 关键决策与依据

### 决策 1：简历可选，但缺少简历时必须提供最小背景

| 方案 | 个性化程度 | 开始门槛 | 结论 |
|---|---:|---:|---|
| 强制上传简历 | 高 | 高，阻塞无简历或不愿上传的用户 | 不选 |
| 完全不支持简历 | 低 | 低，但难形成项目深挖差异化 | 不选 |
| 可选简历 + 三项背景兜底 | 中到高 | 可控 | 选择 |

这同时解决现有 #9 中“支持简历上传”和“不要求上传简历”的矛盾：**简历是可选的个性化增强能力，不是开始练习的前置条件。**

### 决策 2：首期只支持 PDF

| 方案 | 覆盖范围 | 实现与测试成本 | 结论 |
|---|---:|---:|---|
| PDF | 能覆盖主要简历交付格式 | 可控 | 选择 |
| PDF + DOC/DOCX | 覆盖更广 | 解析差异和异常明显增加 | 两个月不做 |
| PDF + OCR | 覆盖扫描件 | 引入识别质量和额外成本 | 不做 |

原型文案统一为“支持 PDF，最大 10 MB”。

### 决策 3：解析文本用于上下文，不建设简历产品

系统需要从简历中获取项目、职责、技术栈和结果等线索，以生成相关问题。产品层只要求“可用作面试上下文”，不要求生成完整结构化人才档案，也不对简历质量打分。

具体如何截断、检索或摘要简历文本属于架构决策；不得简单承诺每次都把全文无条件发送给模型。

### 决策 4：最多保存 3 份简历

一份简历可以满足单岗位用户，三份足以覆盖同一用户针对少量岗位的版本切换，同时限制文件存储、状态管理和界面复杂度。超过上限时，应提示先删除或替换，而不是静默失败。

## 基本概念与信息结构

| 概念 | 定义 |
|---|---|
| Resume File | 用户上传的原始 PDF 文件 |
| Resume Text | 从 PDF 中提取、可作为面试上下文的文本 |
| Resume Status | 上传中、解析中、可用或解析失败 |
| Default Resume | 创建面试时默认选中的一份可用简历 |
| Job Context | 目标岗位、可选 JD、经验年限、公司类型、训练重点和本次选择的简历 |
| Manual Background | 无可用简历时用于开始练习的三项背景信息 |

关系：

```text
User
  ├─ Resume 0..3
  └─ Candidate Profile

Interview Plan
  └─ Job Context
       ├─ Target Role（必填）
       ├─ Resume（可选）
       ├─ Job Description（可选）
       ├─ Experience Level（无简历时必填）
       ├─ Company Type（无简历时必填）
       └─ Practice Focus（可选）
```

## 验收标准

### Proposal 定稿验收

- [ ] 团队确认简历可选，并确认无简历时的三项背景兜底。
- [ ] 团队确认两个月内只支持文本型 PDF，最大 10 MB。
- [ ] 团队确认每个用户最多保存 3 份简历。
- [ ] 团队确认 JD 为可选的当前面试上下文，不建设职位库。
- [ ] 团队确认不做 OCR、DOC/DOCX、结构化简历、简历优化和 ATS 打分。

### 两个月产品行为验收

1. 用户可以不上传简历，通过目标岗位、经验年限和公司类型创建面试。
2. 用户上传合法 PDF 后能看到上传/解析状态；解析成功后可以在创建面试时选择。
3. 不支持的格式、超过大小限制和无法提取文本的 PDF 都会得到明确提示，并可改走手动背景。
4. 用户最多保存 3 份简历，并可以重命名、设为默认、切换和删除。
5. 面试计划可以体现选定简历或 JD 中的至少一个相关背景点，但不会展示虚构经历。
6. 删除简历后，新创建的面试不能再选择该简历。
