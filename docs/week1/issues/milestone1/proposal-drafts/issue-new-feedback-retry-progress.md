## 动机 / 用户故事

一次模拟面试只能暴露问题，不能证明用户已经改进。通用 AI 的反馈经常是一段泛化评价，或者给出一个无法解释的总分。产品要成为“教练”，必须把诊断定位到真实回答，并让用户立刻重答、看到自己补上了什么、还缺什么。

典型用户故事：

- 作为一名刚完成项目深挖的用户，我希望反馈引用我说过的原句，而不是给一段看起来适用于所有人的建议。
- 作为一名专业表达不完整的用户，我希望系统指出我缺少备选方案、判断依据、验证过程或结果，并解释为什么面试官会继续追问。
- 作为一名想真正改进的用户，我希望直接重答同一道问题，并看到第二次回答比第一次补充了哪些证据。
- 作为一名持续准备面试的用户，我希望从历史中看到反复出现的薄弱项，并知道下一次最值得练什么。

一句话概括：

**反馈必须有原回答证据，并能直接生成一次可验证的复练行动。**

## 目标用户

- 完成至少一次英文技术面试练习，需要具体改进建议的候选人。
- 希望跨多次练习持续改善专业表达、回答结构和英文表达的技术岗位候选人。

## 现有做法及其不足

现有 Proposal 已确定结构化反馈和 Retry 方向，当前原型也有报告、回答改进、复练清单和历史页面，但仍存在以下未决边界：

- 反馈维度在不同文档中出现发音、表达、结构、技术内容等不同组合。
- 没有明确每条反馈必须包含哪些证据。
- “重答这一题”和“加入复练清单”已经出现，但没有定义复练完成后如何判断改进。
- 旧功能全景包含雷达图、统计面板和好句收藏，容易把展示丰富度放在教练闭环之前。

## 本期范围与明确不做

### 两个月最终范围

#### 会话报告

每次完成的练习生成一份报告，按以下四个维度组织：

| 维度 | 关注内容 | 不做的承诺 |
|---|---|---|
| 专业内容与证据 | 是否说明背景、个人贡献、备选方案、判断依据、边界、验证和结果 | 不宣称对所有专业结论做权威正确性认证 |
| 回答结构 | 是否有清楚顺序，重点与问题匹配，行为题是否包含必要 STAR 信息 | 不强制所有问题使用同一种模板 |
| 英文表达 | 用词、语法、专业表达、冗余和中式表达 | 不追求逐句全部改写 |
| 表达清晰度 | 是否因停顿、语速或明显发音问题影响理解 | 不做音素级发音评分 |

每条重要反馈至少包含：

1. 问题所在的轮次和题目。
2. 用户原回答证据。
3. 具体问题及其影响。
4. 修改思路或建议补充的证据。
5. 一个更好的示例表达。
6. 可执行的下一步：同题重答或加入复练清单。

#### 复练

- 用户可以从报告中直接重答同一道问题。
- 用户可以把薄弱回答加入复练清单，并按专业内容与证据、回答结构、英文表达分类查看。
- 复练时保留原问题、原回答、原诊断和本次改进目标。
- 提交新回答后，系统对比“已补充”“仍缺少”，而不是只重新生成一份无关联报告。
- 用户可以使用自己的真实项目数据；示例答案只能作为结构参考，不能鼓励编造经历或指标。

#### 历史与成长

- 历史记录展示面试计划、练习时间、完成状态、会话转录和报告。
- 首页或历史页展示当前待复练项和下一次推荐。
- 跨会话成长只使用可解释的定性变化，例如“上次缺少验证 → 本次说明了压测方法”。
- 统计只服务下一步行动，不建设复杂仪表盘。

### Milestone 实现深度

| Milestone | 做到什么程度 |
|---|---|
| MS1 | 使用固定数据完成项目深挖报告；报告引用原回答，并可进入同题重答/复练清单 |
| MS2 | 根据真实会话生成并保存报告；支持同题重答并保存第二次回答 |
| MS3 | 支持复练清单、前后对比、历史报告、跨会话薄弱项和下一次推荐 |
| MS4 | 校准反馈质量，处理生成失败、证据缺失和不确定结论；完成典型样例与人工 Review |

### 明确不做

- 不用一个 0-100 总分作为报告核心，不做缺乏解释依据的分数排行榜。
- 不做音素级发音评分、口音纠正课程和逐词红绿标注。
- 不做勋章、积分、连续打卡、排名和社交分享。
- 不做大型雷达图、复杂趋势图和为了展示而展示的数据看板。
- 不做独立好句收藏库；更好的表达保留在反馈和复练上下文中即可。
- 不逐句纠正整场转录，只选择最影响面试结果的重点问题。
- 不虚构用户没有说过的项目数据、技术成果或量化指标。
- 不把模型生成的专业判断包装成权威招聘结论或录用概率。

## 关键决策与依据

### 决策 1：选择证据化反馈，不选择裸总分

| 方案 | 用户是否知道怎么改 | 可信度 | 结论 |
|---|---:|---:|---|
| 只给总分 | 低 | 低，难解释分数来源 | 不选 |
| 泛化文字总结 | 中低 | 中，容易套话 | 不选 |
| 原回答证据 + 诊断 + 示例 + 下一步 | 高 | 高，可核对 | 选择 |

以下仅为研发岗位的反馈示例；其他目标岗位应替换为对应的专业问题和证据：

```text
原回答：
“We used goroutines to make it fast.”

问题：
只说明了方案名和模糊结果，没有说明资源边界、替代方案和验证数据。

改进方向：
补充为什么选择 bounded worker pool、如何确定 worker 数和队列长度，以及压测结果。

示例表达：
“We introduced a bounded worker pool to keep memory usage predictable, and load tests showed a 32% reduction in p95 latency.”

下一步：
使用你自己的真实项目数据重答这道题。
```

### 决策 2：报告之后必须能同题重答

| 方案 | 优点 | 不足 | 结论 |
|---|---|---|---|
| 报告是终点 | 实现最简单 | 用户读完仍没有练习行为 | 不选 |
| 推荐下一场新练习 | 能继续使用 | 可能回避当前薄弱回答 | 辅助入口 |
| 同题重答 + 前后比较 | 直接验证是否补上问题 | 需要保存两次回答关系 | 选择 |

复练完成的核心不是“用户又答了一遍”，而是新回答至少针对原诊断进行了改进，并让用户知道仍缺少什么。

### 决策 3：成长记录采用定性证据，不建设复杂评分体系

选择“某个具体回答从缺少什么变为补上什么”的定性锚点。原因是两个月内难以建立可靠、可解释的跨场景统一分数；复杂图表会放大不稳定评分的误导。

可以展示练习次数、时长、待复练数量等事实统计，但不能让事实统计替代产品价值。

### 决策 4：专业反馈评价解释质量，不冒充权威正确性认证

专业内容与证据维度重点判断用户是否说清楚背景、选择、判断依据、边界、验证和结果。对于需要外部事实、代码或业务材料才能确认的结论，反馈使用“建议核实”“缺少证据”等措辞，不直接断言错误。

## 基本概念与信息结构

| 概念 | 定义 |
|---|---|
| Session Report | 一次完成练习对应的结构化报告 |
| Feedback Dimension | 专业内容与证据、回答结构、英文表达或表达清晰度 |
| Feedback Evidence | 支撑诊断的原回答片段 |
| Feedback Item | 一条包含证据、问题、影响、建议和下一步的诊断 |
| Retry Item | 从 Feedback Item 创建的可复练任务 |
| Retry Attempt | 用户对同一问题提交的一次新回答 |
| Attempt Comparison | 新旧回答在原改进目标上的“已补充/仍缺少”比较 |
| Weakness Pattern | 多次会话中重复出现、且有具体 Feedback Item 支撑的薄弱项 |

关系：

```text
Practice Session
  ├─ Interview Turns
  └─ Session Report
       └─ Feedback Items
            ├─ Feedback Evidence
            └─ Retry Item
                 └─ Retry Attempts 1..n
                      └─ Attempt Comparison
```

模型选择、提示词、评估流水线和存储结构属于后续架构与实现 Issue。

## 原型 / Demo

- 在线原型：[SpeakUp 英文技术面试教练](https://spreak-product-prototype.wendymcdonald606998.chatgpt.site/)
- 相关页面：面试报告、回答改进、复练清单、回答复练、练习历史。

原型调整建议：

- 报告固定展示四个维度，但首屏先显示“下一步最值得练”，避免信息堆叠。
- 每条反馈明确显示原回答证据和问题影响。
- “加入复练清单”和“重答这一题”保留，补充重答后的“已补充/仍缺少”对比页或状态。
- 历史页突出待复练项和继续练习，不增加复杂雷达图。
- 删除独立好句收藏入口；优化表达保留在报告和对应复练项中。

## 验收标准

### Proposal 定稿验收

- [ ] 团队确认报告使用专业内容与证据、回答结构、英文表达、表达清晰度四个维度，并随目标岗位适配专业内容。
- [ ] 团队确认每条重要反馈必须引用用户原回答证据。
- [ ] 团队确认报告后必须支持同题重答和复练清单。
- [ ] 团队确认成长记录采用定性证据，不建设统一裸总分。
- [ ] 团队确认不做音素评分、游戏化、复杂仪表盘和独立好句收藏库。

### 两个月产品行为验收

1. 完成练习后，用户能看到一份与本次会话绑定的报告。
2. 至少一条反馈包含题目、原回答、具体问题、影响、改进方向和示例表达。
3. 报告不会引用用户未说过的内容作为“原回答证据”。
4. 用户可以直接重答同一道问题，或把问题加入复练清单。
5. 第二次回答提交后，系统围绕原改进目标展示“已补充”和“仍缺少”。
6. 用户可以从历史查看会话、报告和未完成复练项。
7. 跨会话推荐必须能追溯到至少一个具体反馈项，不能只给泛化建议。
8. 报告不展示无解释的总分，不虚构用户项目数据，不把专业诊断表述为权威录用结论。
