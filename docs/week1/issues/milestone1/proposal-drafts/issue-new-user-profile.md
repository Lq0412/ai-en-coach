# 新 Issue：用户账户与求职档案功能范围

> - 建议标题：`[提案] 用户账户与求职档案功能范围`
> - 建议标签：`proposal`、`FullSpec`
> - 建议 Milestone：MS1（战略决策）
> - 关联：Issue #9
> - 草稿状态：待团队审核，尚未 Accepted

## 动机 / 用户故事

用户的简历、面试记录、反馈和复练项需要归属于同一个人，否则无法形成跨会话的持续训练。但账户本身不是产品卖点，如果首期投入大量时间建设昵称、头像、手机号、会员和订单，会挤占技术面试主链路。

典型用户故事：

- 作为一名求职者，我希望下次打开产品时还能看到自己的简历、面试记录和待复练回答。
- 作为一名同时准备多个岗位的用户，我希望系统记住我的基础求职背景，创建面试时不必重复填写。
- 作为一名上传过简历的用户，我希望能够退出账户，并能删除自己的账户和求职数据。

一句话概括：

**需要一个能承载个性化数据和持续训练的最小账户体系，但不把时间花在与面试教练无关的账户装饰和商业化能力上。**

## 目标用户

- 会多次使用产品、需要保存简历和训练记录的英文技术面试候选人。
- MS1 选定的代表性技术岗位演示用户；具体岗位不属于账户模块的产品边界。

## 现有做法及其不足

当前原型已经有“我的”“设置”“退出登录”“注销账号”等页面，但没有明确账户系统究竟做到什么程度；头像、昵称和手机号容易让团队误判为首期必须实现的功能。

如果完全没有账户，MS1 可以快速演示，但后续无法稳定关联简历、会话、反馈和复练记录。如果直接建设完整账户中心，又会引入多种登录方式、短信、绑定、找回和商业化等大量非核心工作。

## 本期范围与明确不做

### 两个月最终范围

- 提供一种可用的登录/身份识别方式，具体认证技术由后续架构 Proposal 决定。
- 用户登录后可以继续访问自己的求职档案、简历、面试历史、报告和复练清单。
- 求职档案包含：默认目标岗位、经验年限、目标公司类型、默认简历。
- 设置页只保留与核心使用相关的能力：语音偏好、隐私政策、服务条款、退出登录、删除账户。
- 用户删除账户时，应明确告知会删除其求职档案、简历和训练数据；删除过程的技术时限由架构/合规设计补充。

### Milestone 实现深度

| Milestone | 做到什么程度 |
|---|---|
| MS1 | 使用固定演示用户；原型展示“我的”、求职档案、简历和历史入口，不实现真实注册登录 |
| MS2 | 支持一种身份识别方式；用户可跨会话访问 1 份简历、面试记录和报告 |
| MS3 | 完成求职档案编辑、默认简历、语音偏好、退出登录和删除账户入口 |
| MS4 | 完成失败状态、数据删除验证、隐私说明和可用性打磨，不增加新的账户类型 |

### 明确不做

- 不做多套并行登录方式，不同时建设手机号、邮箱、第三方登录和账号绑定。
- 不做手机号展示、短信营销、联系人或通讯录。
- 不做头像装扮、等级、勋章、个人主页和社交关系。
- 不做会员、订单、优惠券、支付、订阅和权益中心。
- 不做企业组织、团队成员、管理员后台和多人协作。
- 不把昵称、头像设置作为两个月发布的核心验收项。

## 关键决策与依据

### 账户范围选择

| 方案 | 能力 | 优点 | 风险/不足 | 结论 |
|---|---|---|---|---|
| A. 无账户，仅本地临时数据 | 打开即用，数据只留在当前设备或会话 | MS1 最快 | 无法跨会话可靠保存，难支撑复练和成长记录 | 仅用于 MS1 演示 |
| B. 最小账户 + 求职档案 | 一种身份方式，保存简历、历史和档案 | 支撑核心闭环，范围可控 | 仍需处理退出和删除 | 两个月选择 |
| C. 完整账户与商业化中心 | 多种登录、会员、订单、权益 | 长期扩展完整 | 工作量大，不验证核心价值 | 不选 |

选择方案 B，MS1 暂时采用方案 A 的固定演示用户。原因是账户只需要解决“数据归谁、下次如何继续”，不需要成为独立产品模块。

### 求职档案字段选择

选择只保存能影响面试内容的字段：

- 默认目标岗位。
- 经验年限区间。
- 目标公司类型。
- 默认简历。

不在档案中收集与面试生成无直接关系的性别、生日、地区、教育详情、社交账号等字段。简历中已有的信息不要求用户重复填写。

## 基本概念与信息结构

| 概念 | 定义 |
|---|---|
| User Account | 唯一标识用户及其数据归属的账户身份 |
| Candidate Profile | 可复用于多次面试的轻量求职背景 |
| User Preference | 语音等不改变面试业务语义的使用偏好 |
| User-owned Data | 简历、职位背景、面试会话、反馈和复练项等用户数据 |

关系：

```text
User Account
  ├─ Candidate Profile
  ├─ User Preference
  ├─ Resumes
  ├─ Interview Sessions
  └─ Retry Items
```

认证协议、Token、数据库表和数据删除任务不在本 Proposal 定义；这些属于后续架构与实现 Issue。

- 相关页面：底部“我的”、个人简历、设置。

原型调整建议：

- “我的”页突出练习数据、默认求职背景、简历和隐私设置。
- 手机号、头像、昵称改成非核心展示或移除，避免形成首期开发承诺。
- 退出登录、删除账户保留为两个月最终状态；MS1 点击时说明为原型行为。

## 验收标准

### Proposal 定稿验收

- [ ] 团队确认采用“最小账户 + 求职档案”，而不是完整账户中心。
- [ ] 团队确认 MS1 使用固定演示用户，不以真实登录为验收项。
- [ ] 团队确认求职档案只包含目标岗位、经验年限、公司类型和默认简历。
- [ ] 团队确认账户删除需要覆盖用户拥有的简历和训练数据。
- [ ] 认证技术方案留给架构 Proposal，不在产品讨论中提前锁定。

### 两个月产品行为验收

1. 已识别用户重新进入后，可以继续查看自己的简历、历史、报告和复练项。
2. 用户修改默认目标岗位或默认简历后，下一次创建面试默认使用新选择。
3. 用户退出后，受保护的个人数据不继续展示。
4. 用户可以发起删除账户，并看到将被删除的数据范围说明。
5. 产品中不存在会员、订单、积分、排行榜或社交入口。
