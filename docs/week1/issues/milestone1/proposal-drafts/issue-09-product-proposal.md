# [提案] SpeakUp 两个月产品方案与功能边界

> 关联：[Issue #9](https://github.com/1024XEngineer/XE3-ESL/issues/9) · [最新在线原型](https://speakup-product-prototype.wendymcdonald606998.chatgpt.site/)

## 动机 / 用户故事

准备外企、出海团队或国际化技术相关岗位英文面试的候选人，常见困难不是完全没有岗位能力，而是无法在追问压力下用英文完整说明自己的经历、判断和结果。背稿只能覆盖固定问题，通用 AI 又容易退化为没有岗位上下文的闲聊。

典型用户故事：

- 作为一名候选人，我希望系统根据目标岗位、岗位描述和简历经历生成面试方案，而不是给所有人同一套问题。
- 作为一名临场表达不稳定的候选人，我希望不同职责的面试官围绕同一岗位依次提问，让我提前适应真实面试的侧重点变化。
- 作为一名完成练习的用户，我希望报告指出目标是否达成、哪句表达需要修改，并能把问题加入错题复练。
- 作为一名还想练其他英语场景的用户，我希望创建一个私有角色自由对话，但不需要浏览复杂的课程或角色市场。

一句话定位：

**SpeakUp 是以英文技术相关岗位模拟面试为核心、以可配置角色对话为扩展的 AI 语音练习教练。**

## 目标用户

### 首要用户

- 准备外企、出海公司、海外岗位或国际化团队英文面试的技术相关岗位候选人。
- 首期覆盖研发、测试、数据/算法、运维/DevOps、技术产品等有明确专业能力和项目经历的岗位。
- 优先验证学生、应届生和 0-5 年经验从业者。
- 具备基础英文阅读或表达能力，但在自我介绍、岗位问题、项目深挖、情景追问或反问环节中容易卡壳。

### 演示数据原则

- 用户自由输入目标岗位；系统不能把岗位或技术栈写死在产品规则中。
- 最新原型中的“SpeakUp 产品经理”和四位具名面试官是当前演示数据，可在评审或演示前替换。
- MS1 只需选一个代表性技术相关岗位完成端到端演示，但产品信息结构必须能够适配其他目标岗位。

### 本期不重点服务

- 完全零基础英语学习者。
- 以考试提分、儿童英语或系统化语言课程为主要目标的用户。
- 需要职位搜索、自动投递、求职社区等完整招聘服务的用户。

## 现有做法及其不足

| 现有做法 | 能解决什么 | 主要不足 |
|---|---|---|
| 背英文稿、搜索面经 | 快速准备固定答案 | 追问变化后容易失效 |
| 通用 AI 对话 | 低成本、随时练习 | 缺少稳定的面试结构、岗位角色和复练闭环 |
| 通用英语 App | 适合日常表达或发音练习 | 难判断项目说明是否包含贡献、取舍、验证和结果 |
| 真人陪练 | 反馈真实 | 成本高、频次低，不适合同题反复练习 |
| 一次性模拟面试 | 能发现问题 | 报告与下一次练习脱节，难证明用户已经改进 |

## 本期范围与明确不做

这里的“本期”指未来两个月完整周期。原型表达期望的最终产品形态，Milestone 决定每一阶段真正实现到什么程度。

### 两个月最终范围

| 模块 | 最终产品能力 | 优先级与原因 |
|---|---|---|
| 最小账户与个人中心 | 一种身份方式；保存简历、历史、报告和错题；基础资料、退出与注销 | P0，持续训练需要稳定的数据归属 |
| 面试背景 | 自由输入目标岗位；编辑岗位名称、岗位描述、简历亮点、考察范围和补充要求 | P0，决定问题是否与用户相关 |
| 简历管理 | 可选简历；PDF/DOC/DOCX，最大 10 MB；默认简历和状态管理 | P0 个性化增强，但不阻塞无简历用户开始练习 |
| 面试方案与面试官 | 生成可编辑方案；按轮次生成不同职责和风格的面试官；允许确认、编辑和补充 | P0，让用户在开始前知道会被怎样考察 |
| 面试训练 | 四轮计划；完整面试与五类专项练习；实时语音和按住/点击说话两种交互 | P0，构成核心使用链路 |
| 智能追问 | 根据岗位、简历和上一轮回答进行澄清、深挖、验证或收束 | P0，形成区别于固定题库的价值 |
| 报告与逐句改进 | 目标达成、语言表现、原句与优化句、改进原因、跟读 | P0，把练习转为可执行反馈 |
| 错题与历史 | 按表达、语法、词汇、发音归类；查看来源、重新练习和历史统计 | P1，支持跨会话持续改进 |
| 自定义角色对话 | 通过五步问答创建私有角色，预览后用实时或气泡模式对话 | P1，复用语音能力，扩展练习场景 |

### Milestone 边界

| Milestone | 交付范围 | 允许 Mock 的部分 |
|---|---|---|
| MS1：战略决策与串联原型 | Proposal 定稿；原型覆盖两个月结构；用一个可替换岗位串联背景、方案、面试官、3 轮追问、报告和复练 | 用户、简历、语音、追问、报告均可使用演示数据，但必须说明 |
| MS2：核心面试闭环可用 | 一种身份、一个岗位背景、1 份 PDF、真实语音/转录、受控追问、基础报告保存 | 多面试官编辑、DOC/DOCX、完整历史、自定义角色可继续 Mock |
| MS3：核心功能闭合 | 四轮面试与专项练习、角色化面试官、最多 3 份简历、逐句改进、错题复练和历史；交付最小自定义角色练习 | 会员、订单等商业化入口不实现 |
| MS4：质量与交付 | 异常恢复、隐私删除、稳定性、测试、部署和展示材料 | 核心流程不得依赖固定脚本 |

### 明确不做

- 不做课程体系、考试题库、儿童英语或场景商城，不把产品扩展为全品类英语学习平台。
- 不做职位搜索、自动投递、求职社区、简历模板、简历编辑、ATS 打分或职位匹配。
- 不做实际会员订阅、权益计费、订单、支付、优惠券、积分、排行榜和社交关系；原型中的相关入口仅为产品外壳。
- 不做角色市场、公开分享、自定义工作流或无限能力 Agent；自定义角色仅服务私有语音练习。
- 不做视频会议、数字人或多名面试官同时在线；多面试官只表示不同轮次的角色化提问者。
- 不承诺音素级发音评分，也不使用一个无法解释的裸总分代表面试能力。
- 不自研 ASR、TTS 或大语言模型。
- 不让 AI 无约束地无限生成问题；使用受控阶段目标和结束条件保证稳定性。

## 关键决策与依据

### 决策 1：面试是主产品，自定义角色是次级扩展

| 方案 | 优点 | 风险 | 结论 |
|---|---|---|---|
| 只做通用角色聊天 | 场景广、容易开始 | 竞争成熟，难形成专业差异 | 不选 |
| 只做一次性模拟面试 | 边界清楚 | 使用频次低，反馈后没有持续练习 | 不选 |
| 面试闭环为主 + 私有角色练习为辅 | 主价值清晰，同时复用语音基础能力 | 需要严格控制优先级 | 选择 |

MS2 前不以自定义角色作为交付阻塞项。只有面试主链路真实可用后，才在 MS3 交付最小角色练习。

### 决策 2：目标岗位自由输入，样例岗位不写死

原型当前使用“SpeakUp 产品经理”演示职位描述、四位面试官和四轮计划。选择这一样例是为了说明流程，不代表只服务产品经理。生成方案后，岗位名称、岗位描述、简历亮点、考察范围和补充要求都应可编辑；面试官职责和问题必须随最终上下文更新。

### 决策 3：保留分角色面试官，不做多人同时面试

四位面试官分别代表 HR、岗位负责人、协作方和高层视角，按轮次依次出现。这样能表达真实面试中的评价侧重点，又不会引入多人实时会话、视频和复杂角色协同。姓名、头像、风格和数量是可编辑的演示配置，不是固定产品规则。

### 决策 4：原型表达终局，Milestone 控制真实程度

原型中的每个核心页面都应能映射到两个月范围，但不以“能点击”冒充“已经实现”。会员、权益和订单虽然出现在最新原型中，本周期只作为外壳，不进入研发验收；页面必须标注“规划中”或在开发版本中隐藏。

### 决策 5：反馈必须落到下一次练习

核心闭环为：

```text
目标岗位/简历背景
  -> 可编辑面试方案与角色化面试官
  -> 完整面试或专项语音练习
  -> 基于回答的追问
  -> 目标达成与语言表现报告
  -> 原句改进/加入错题
  -> 再次练习
```

面试过程中默认不逐句打断；训练结束后集中反馈。专项练习可以给最小提示，完整面试默认不主动提示。

## 基本概念与信息结构

| 概念 | 定义 |
|---|---|
| User Account | 识别用户并关联简历、练习和复盘数据的最小身份 |
| Resume | 可选的面试个性化文件，不是进入练习的强制门槛，也不是评分对象 |
| Interview Context | 岗位名称、岗位描述、简历亮点、考察范围和补充要求 |
| Interview Plan | 根据上下文生成并由用户确认的四轮面试安排 |
| Interviewer | 对应某一轮次的职责、关注点和对话风格；按轮次出现 |
| Practice Stage | 自我介绍、岗位八股、项目深挖、情景追问、反问环节之一 |
| Practice Session | 一次完整面试、专项练习或角色对话 |
| Feedback Item | 基于练习目标或用户原句产生的具体改进项 |
| Retry Item | 加入错题后可再次练习的表达、语法、词汇或发音问题 |
| Custom Role | 用户通过五步问答创建的私有对话角色 |

主要信息流：

```text
创建
  ├─ 模拟面试
  │    -> 输入岗位
  │    -> 编辑面试方案
  │    -> 确认面试官
  │    -> 选择完整面试/专项练习
  │    -> 语音问答
  │    -> 报告/逐句改进/错题复练
  └─ 创建角色
       -> 五步定义
       -> 角色预览
       -> 实时或气泡对话

历史 -> 面试/角色练习记录 -> 报告与再次练习
我的 -> 简历、基础资料、设置与数据管理
```

身份技术、文件存储、语音协议、模型编排和数据库设计由后续架构 Proposal 决定，本 Proposal 只约束产品行为和范围。

## 原型 / Demo

- 在线原型：[SpeakUp 最新在线原型](https://speakup-product-prototype.wendymcdonald606998.chatgpt.site/)
- 原型主要页面：创建、历史、创建模拟面试、确认面试官、面试轮次、语音练习、创建角色、角色预览、角色聊天、练习报告、逐句改进、错题回顾、我的、个人简历、设置。

原型对齐结论：

- 保留“模拟面试”和“创建角色”两个创建入口，但视觉与文案应明确模拟面试为主路径。
- 保留四位可编辑面试官和四轮面试结构；具体姓名、职位和问题随目标岗位变化。
- 保留实时对话与气泡/点击说话两种语音方式。
- 保留 PDF/DOC/DOCX、最大 10 MB 的最终简历文案；MS2 开发版先支持 PDF 时必须明确阶段限制。
- 报告、逐句改进和错题回顾必须相互可达，并能回到再次练习。
- 会员、权益中心、订单入口不进入两个月研发验收；若继续展示，应标记为未开放或产品外壳。

## 验收标准

### Proposal 定稿验收

- [ ] 团队确认面试为主路径、自定义角色为次级路径。
- [ ] 团队确认目标岗位自由输入，产品经理等只作为可替换演示样例。
- [ ] 团队确认四位面试官表示按轮次出现的不同角色，不做多人实时面试。
- [ ] 五个子 Proposal 与本 Proposal 的范围、优先级和 Milestone 无冲突。
- [ ] 原型中的商业化入口被明确为非本周期研发范围。
- [ ] 通过后将标签更新为 `Proposal-Accepted` + `FullSpec`。

### MS1 演示验收

1. 用户输入一个目标技术相关岗位，并可编辑岗位描述、简历亮点、考察范围和补充要求。
2. 系统展示随岗位变化的面试官和四轮计划；原型样例不能被描述为唯一支持岗位。
3. 用户完成至少 3 轮提问、回答和追问，其中至少一次追问明确依赖上一轮回答。
4. 练习结束后展示目标达成、语言表现和至少一条基于原句的改进建议。
5. 用户可以从报告进入逐句改进或错题复练，并能再次练习。
6. 演示明确说明哪些能力使用 Mock、哪些已经真实实现。

### 两个月产品验收

1. 模拟面试主链路可以在非演示岗位下完整运行，问题、面试官和报告不会残留其他样例内容。
2. 完整面试和五类专项练习均可进入语音练习并生成报告。
3. 用户可以跨会话访问简历、历史、报告和错题数据。
4. 自定义角色至少支持私有创建、预览和一次可恢复的语音对话。
5. 核心流程具备录音、识别、模型、网络和报告生成失败时的明确状态与重试入口。
6. 产品中不存在可实际下单或付费的会员/订单流程，也不存在角色市场、社交、数字人或裸总分能力。
