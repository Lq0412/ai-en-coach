## 动机 / 用户故事

希望进入外企、出海团队或国际化技术岗位的候选人，常见问题不是不具备岗位能力，而是无法在英文面试中自然、完整、有证据地说明自己的专业能力。用户可能能阅读英文资料，也能提前准备自我介绍，但一旦面试官围绕项目细节、专业判断和结果继续追问，就容易卡词、回答过短、结构混乱，或者只说结论而说不清理由。

典型用户故事：

- 作为一名准备英文技术岗位面试的候选人，我希望 AI 根据我的目标职位、岗位要求和项目经历继续追问，而不是随机问一组通用英语问题。
- 作为一名英文阅读能力尚可但临场表达不足的候选人，我希望练习后能看到问题对应的原回答证据、修改理由和更好的表达方式。
- 作为一名时间有限的求职者，我希望把薄弱回答直接加入复练，并通过再次回答确认自己是否真的改进。

一句话概括：

**用户需要一个懂技术面试上下文、会基于回答继续追问、并能把反馈转化为下一次复练的英文技术面试教练。**

## 目标用户

### 首要用户

- 准备外企、出海公司、海外岗位或国际化团队英文面试的技术相关岗位候选人。
- 首期覆盖研发、测试、数据/算法、运维/DevOps、技术产品等有明确专业能力与项目经历的岗位。
- 优先验证计算机相关专业学生、应届生和 0-5 年经验技术从业者。
- 有基础英文阅读和表达能力，但在项目讲述、专业判断、岗位问题或行为追问中容易卡壳。

### MS1 演示用户原则

- 从上述目标用户中选择一个代表性技术岗位制作完整演示数据。
- 当前原型使用 Go 后端工程师作为样例，但它不是产品限定岗位，也不是不可修改的 MS1 决策。
- 演示岗位可以在评审前替换；面试计划、问题和反馈必须由目标岗位上下文驱动，不能在产品规则中写死某个岗位或技术栈。
- 演示用户需要有实际项目经历，用于验证如何说明背景、个人贡献、专业取舍、验证过程和结果。

### 本期不服务的用户

- 完全零基础英语学习者。
- 以考试提分、儿童英语、旅游英语或日常陪聊为主要目标的用户。
- 非技术岗位的全量面试准备用户。

## 现有做法及其不足

用户当前通常通过背英文稿、搜索面经、使用通用 AI、使用英语学习 App，或寻找真人陪练来准备面试。

主要不足：

- 背稿只能覆盖固定问题，追问变化后容易失效。
- 通用 AI 可以陪聊，但缺少稳定的技术面试结构和追问深度。
- 通用英语产品偏发音或日常表达，难以评价项目说明是否包含取舍、验证和结果。
- 真人陪练成本高、频次低，不适合反复练同一个薄弱回答。
- 很多产品只给总分或泛化建议，用户不知道下一次具体该练哪一道题、补哪一条证据。

## 本期范围与明确不做

这里的“本期”指未来两个月的完整产品周期；各 Milestone 实现深度见后文。原型应表达两个月最终形态，不等于 MS1 全部实现。

### 两个月最终范围

| 模块 | 两个月最终产品能力 | 核心价值 |
|---|---|---|
| 最小用户体系 | 一个账户身份、求职档案、数据持久化、退出与数据删除入口 | 保存简历、面试和复练记录 |
| 简历与职位背景 | PDF 简历管理、目标岗位、可选 JD、无简历快速背景兜底 | 给追问提供真实上下文 |
| 面试计划 | 根据岗位、简历和训练重点形成四轮面试计划 | 避免进入空白聊天框 |
| 面试训练 | 完整模拟面试与分阶段专项练习，语音为主、文本转录为辅 | 既能全真模拟，也能重复练薄弱环节 |
| 智能追问 | 基于职位、简历和上一轮回答进行澄清、深挖或推进 | 形成区别于固定题库的真实压力 |
| 证据化反馈 | 引用原回答，解释问题，给出优化示例和下一步行动 | 反馈可信、可理解、可执行 |
| 复练闭环 | 同题重答、薄弱回答清单、前后变化提示 | 把“看报告”变成“真正改进” |
| 历史与成长 | 面试历史、报告回看、薄弱项趋势和下一次推荐 | 支持跨会话持续训练 |

### Milestone 边界

| Milestone | 产品范围 | 允许使用 Mock 的部分 |
|---|---|---|
| MS1：战略决策与首个串联版本 | 完成文字 Proposal 和两个月最终原型；选择一个代表性技术岗位，以“项目经历深挖”跑通背景设置、3 轮问答、证据化报告、同题复练 | 用户、岗位样例、简历、语音识别、AI 追问、反馈均可用固定数据或脚本，但必须串成完整流程；样例不得被描述为产品唯一岗位 |
| MS2：核心路径真实可用 | 跑通一个真实项目深挖场景：单一身份、1 份 PDF 简历、真实语音/转录、受控追问、报告保存和同题复练 | 非核心场景和跨会话趋势仍可使用演示数据 |
| MS3：核心功能闭合 | 扩展四轮完整面试、分阶段专项练习、最多 3 份简历、历史记录、薄弱项清单和成长推荐 | 仅非核心展示数据可以 Mock |
| MS4：质量与交付 | 处理失败状态、隐私删除、可用性、稳定性、测试、部署和展示材料；原则上不新增大型核心模块 | Demo 样例可保留，但核心流程不可依赖固定脚本 |

### 明确不做

- 不做泛英语学习平台，以及旅游、点餐、社交、商务闲聊等通用场景。
- 不做完整求职工具箱、职位搜索、自动投递、求职社区。
- 不做简历编辑器、简历模板、ATS 打分、简历优化和职位匹配。
- 不做会员、订单、支付、积分、排行榜、打卡和社交关系。
- 不做数字人、视频面试官、多面试官人格和自定义 Agent。
- 不做音素级发音评分，不用一个裸总分代表用户能力。
- 不自研 ASR、TTS、LLM 等基础模型。
- 不承诺覆盖所有行业和非技术岗位；两个月内以有明确专业能力与项目经历的技术相关岗位为边界。
- 不让 AI 无约束地随机生成无限题目；使用受控面试结构和题目材料保证稳定性。

## 关键决策与依据

### 决策 1：选择英文技术面试教练，不做通用 AI 口语 App

| 方案 | 优点 | 主要问题 | 结论 |
|---|---|---|---|
| 通用 AI 口语陪练 | 用户面广、场景多 | 竞争成熟，边界过宽，难形成两个月差异化 | 不选 |
| 单纯英文模拟面试 | 场景明确、容易演示 | 只提问不复练，仍接近一次性聊天 | 不选 |
| 英文技术面试教练 | 用户目标和结果明确，可用技术追问与复练形成差异 | 需要控制追问和反馈质量 | 选择 |

选择依据：两个月内最值得验证的不是“用户愿不愿意与 AI 说英语”，而是“基于技术上下文的追问和证据化复练，是否比通用 AI 更能帮助用户准备真实面试”。

### 决策 2：原型表达最终形态，Milestone 控制实现深度

备选方案：

1. 原型只画 MS1 会实现的页面。
2. 原型表达两个月最终产品，但在 Proposal 中单独标注每个 Milestone 的实现范围。
3. 原型把所有可能功能都画出来，开发时再决定。

选择方案 2。方案 1 无法帮助团队看清两个月终局，方案 3 会把想法误当承诺。最终原型只展示已经进入两个月范围的核心功能；明确不做的功能不进入原型。

### 决策 3：MS1 选择“项目经历深挖”作为唯一完整场景

| 候选场景 | 实现难度 | 技术面试差异化 | 是否能验证追问与反馈 | 结论 |
|---|---:|---:|---:|---|
| 自我介绍 | 低 | 低 | 较弱 | 后续覆盖 |
| 行为面试 | 中 | 中 | 中 | 后续覆盖 |
| 项目经历深挖 | 中 | 高 | 强 | MS1 选择 |
| 岗位核心能力问答 | 中 | 中 | 容易退化为知识测验 | 后续覆盖 |
| 综合案例与协作 | 高 | 高 | 强，但首期范围过大 | MS3 覆盖 |

项目深挖能同时验证简历/职位上下文、基于回答继续追问、专业证据诊断和同题复练，并且能适配多种技术岗位，是最短的差异化验证路径。

### 决策 4：做“教练闭环”，不以聊天时长或总分为核心

核心闭环固定为：

```text
职位/简历背景
  -> 受控面试计划
  -> 英文语音回答
  -> 基于回答继续追问
  -> 引用原回答的结构化反馈
  -> 同题重答
  -> 记录薄弱项并推荐下一次练习
```

用户在面试中不被逐句纠错；训练结束后再集中反馈。训练模式允许最小提示，模拟模式不主动提示。

## 基本概念与信息结构

| 概念 | 定义 |
|---|---|
| User Account | 用于识别用户并关联其求职档案、简历和训练记录的最小身份 |
| Candidate Profile | 目标岗位、经验年限、目标公司类型等长期求职背景 |
| Resume | 用户上传的 PDF 简历及其可用状态；是面试上下文，不是评分对象 |
| Job Context | 一次面试使用的目标岗位、可选 JD、简历和训练重点 |
| Interview Plan | 根据 Job Context 形成的四轮训练安排 |
| Interview Round | 完整面试中的一个主题轮次，如综合沟通、岗位能力、项目深挖、综合案例与协作；具体内容随目标岗位适配 |
| Practice Session | 用户在完整面试或一个专项阶段下进行的一次连续练习 |
| Interview Turn | AI 问题/追问与用户回答构成的一轮交互 |
| Feedback Evidence | 从用户原回答中引用、用于支撑诊断结论的文本或语音片段 |
| Retry Item | 从反馈中产生、可让用户重新回答的具体薄弱项 |

主要信息流：

```text
创建面试
  -> 设置目标职位、简历与训练重点
  -> 确认四轮面试计划
  -> 选择完整模拟或分阶段专项练习
  -> 语音问答与追问
  -> 查看会话报告
  -> 加入复练/同题重答
  -> 历史与下一次推荐
```

模块细节由以下子 Proposal 定义：

- `[待创建] 用户账户与求职档案功能范围`
- `[待创建] 简历与职位背景管理功能范围`
- `[待创建] 面试创建、练习与智能追问功能范围`
- `[待创建] 反馈、复练与成长记录功能范围`

架构设计必须以这些产品概念和边界为输入；身份方案、文件存储、模型调用和数据表设计不在本 Proposal 决定。

## 原型 / Demo

- 在线原型：[SpeakUp 英文技术面试教练](https://spreak-product-prototype.wendymcdonald606998.chatgpt.site/)
- 本地原型入口：`prototype/public/spreak-prototype.html`
- 当前原型覆盖：创建面试、四轮计划、专项练习、语音对话、会话历史、证据化报告、复练清单、用户页、简历管理和设置。

原型对齐要求：

- 原型展示两个月最终核心能力，不使用“全部在 MS1 实现”的表述。
- MS1 演示从一个代表性技术岗位的项目深挖进入，完成至少 3 轮问答；当前 Go 后端仅为可替换样例。
- 创建页允许用户输入目标岗位，原型文案和 Proposal 不把任何具体岗位写成唯一选项。
- 明确展示“追问引用上一轮回答”“报告引用用户原话”“报告后同题重答”三个差异点。
- 补充不上传简历时的三项背景兜底和简历解析失败状态。
- 将简历支持格式收敛为 PDF；弱化非核心头像、昵称、手机号设置。
- 不出现会员、订单、积分、排行榜、通用英语、自定义角色和数字人入口。

## 验收标准

### Proposal 定稿验收

- [ ] 团队确认产品定位、目标用户、两个月最终范围和明确不做清单。
- [ ] 团队确认目标岗位由用户输入，首期边界为技术相关岗位，而非单一岗位或技术栈。
- [ ] 团队确认 MS1 唯一完整场景为一个代表性技术岗位的项目经历深挖，具体岗位只是演示数据。
- [ ] 四个子 Proposal 已完成讨论并与本 Proposal 无冲突。
- [ ] 原型中的页面均能映射到两个月核心功能；被排除的功能不再出现。
- [ ] MS1、MS2、MS3、MS4 的实现边界可被独立检查，不再用“原型里有”等同于“本阶段已实现”。
- [ ] Proposal 通过后，标签更新为 `Proposal-Accepted` + `FullSpec`，后续变更通过新 Proposal 记录。

### MS1 产品演示验收

1. 用户可以输入一个目标技术岗位，并使用与该岗位相符的演示简历或三项快速背景继续。
2. 系统展示一份随目标岗位适配的四轮面试计划，但只要求项目经历深挖主链路真实可操作。
3. 用户完成至少 3 轮提问、回答和追问；至少一次追问明确基于上一轮回答。
4. 练习结束后展示结构化报告，至少包含原回答证据、问题、修改理由、优化示例和下一步练习。
5. 用户可以从报告进入同题重答或复练清单。
6. 演示明确说明哪些能力为 Mock、哪些能力已实现，不能把固定脚本包装成真实模型能力。
