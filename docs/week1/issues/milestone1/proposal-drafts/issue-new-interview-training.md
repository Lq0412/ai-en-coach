## 动机 / 用户故事

固定题库能帮助用户背答案，却不能模拟真实面试中的连续追问；完全开放的 AI 对话虽然灵活，但容易偏题、深度不稳定，也难以复现和评价。产品需要在“受控面试结构”和“根据回答动态追问”之间取得平衡。

典型用户故事：

- 作为一名准备技术岗位英文面试的用户，我希望系统先根据职位和简历生成一份清楚的面试计划，而不是把我扔进空白聊天框。
- 作为一名项目经历很多的用户，我希望 AI 追问我的个人贡献、替代方案、验证过程和量化结果，而不是只问下一个固定问题。
- 作为一名想练薄弱环节的用户，我希望既能参加一次完整模拟，也能单独反复练项目深挖或岗位相关的综合案例。
- 作为一名容易卡词的用户，我希望训练模式可以提供最小提示，但完整模拟模式保持真实面试节奏。

一句话概括：

**产品需要提供有边界的完整面试计划，并在边界内根据用户的真实回答继续追问。**

## 目标用户

- 准备研发、测试、数据/算法、运维/DevOps、技术产品等技术相关岗位英文面试，需要练习专业表达和临场追问的候选人。
- MS1 优先验证一个代表性技术岗位的项目经历深挖场景；当前原型中的 Go 后端工程师只是可替换的演示样例。

## 现有做法及其不足

现有文档把“自我介绍、行为面试、项目深挖、基础技术问答”描述为四个独立场景；当前原型则使用 Go 后端样例展示“综合沟通、岗位技术基础、项目经历深挖、系统设计与协作”四轮面试，并允许分阶段专项练习。两种表达没有统一，而且样例岗位容易被误解为产品限制，会导致题库、会话、进度和页面结构不一致。

此外，当前原型已经展示实时对话、按键对话、提示、示范和多轮追问，但尚未明确哪些属于核心、训练模式与模拟模式有什么差异，以及 AI 可以在什么范围内改变问题。

## 本期范围与明确不做

### 两个月最终范围

#### 创建面试

1. 用户设置目标岗位、可选简历、可选 JD 和训练重点。
2. 系统根据目标岗位生成一份可确认的四轮面试计划，轮次名称可保持稳定，但内容和问题必须随岗位适配。
3. 用户确认后创建面试；不提供多面试官人格选择。

#### 四轮完整面试

| 轮次 | 核心内容 | 典型验证点 |
|---|---|---|
| 1. 综合沟通 | 自我介绍、求职动机、代表性行为题 | 表达是否有重点，经历是否与岗位相关 |
| 2. 岗位核心能力 | 与目标岗位相关的高频专业问题 | 能否用英文准确解释专业概念、方法和边界 |
| 3. 项目经历深挖 | 项目背景、个人贡献、技术取舍、验证和结果 | 是否能经受连续追问并提供证据 |
| 4. 综合案例与协作 | 根据岗位适配系统设计、测试方案、数据案例、产品案例或协作情境 | 能否结构化说明方案、权衡、风险与协作 |

#### 两种练习方式

- 完整模拟：按计划连续进行，默认不展示提示和示范，接近真实面试节奏。
- 分阶段专项练习：选择自我介绍、行为题、岗位核心能力、项目深挖、综合案例或反问等阶段，允许暂停、重答和查看最小提示。

#### 语音与追问

- 以英文语音回答为主，展示转录文本；用户可以播放问题。
- 产品可提供实时连续语音或按键录音交互，但两者共享同一面试会话语义。
- 每次会话包含提问、回答、追问/推进三种行为。
- 追问必须基于职位背景、简历/JD 或上一轮回答中的可识别信息。
- 追问目标优先覆盖：澄清事实、确认个人贡献、追问取舍、要求验证证据、要求量化结果、收束进入下一题。
- 受控题目材料定义面试主题和评价目标，AI 负责在主题内自然追问，不无约束随机扩展。

### Milestone 实现深度

| Milestone | 做到什么程度 |
|---|---|
| MS1 | 原型展示随岗位适配的四轮最终形态；选择一个代表性技术岗位，仅打通“项目经历深挖”3 轮问答、报告和复练，可使用固定脚本与模拟语音 |
| MS2 | 一个真实项目深挖会话可使用职位/简历上下文，完成真实语音或录音转写、受控追问和结束报告 |
| MS3 | 四轮完整面试和分阶段专项练习可用；支持训练/模拟模式差异、暂停/继续和会话历史 |
| MS4 | 完成网络、录音、识别、模型超时、重试、恢复和稳定性打磨，不再新增面试角色或新业务场景 |

### 明确不做

- 不做通用英语自由聊天、日常生活、旅游、商务闲聊等场景。
- 不做用户自定义面试官人格、评分维度或任意 Agent。
- 不做数字人、视频面试、表情识别和多人面试官会议。
- 不做无限动态出题，不允许追问脱离本次职位、题目目标和用户回答。
- 不做算法刷题执行环境、在线代码编辑器和自动判题平台。
- 不把岗位核心能力练习做成百科知识测验；重点仍是英文解释、专业判断、取舍和边界。
- 不在面试过程中逐句打断纠错，集中反馈放在会话结束后。
- 不承诺 MS1 同时实现四轮真实模型能力。

## 关键决策与依据

### 决策 1：采用“岗位自适应四轮计划 + 分阶段专项练习”

| 方案 | 优点 | 不足 | 结论 |
|---|---|---|---|
| 四个互不相关的场景入口 | 简单、容易理解 | 无法表达一次完整技术面试的节奏 | 不选为主结构 |
| 只做一次完整模拟 | 真实感强 | 用户不能针对薄弱环节反复练 | 不选 |
| 四轮完整计划 + 阶段专项练习 | 同时支持全真模拟和针对性复练 | 状态稍多，但可由统一会话模型承载 | 选择 |

四轮是稳定的信息结构，不是固定题目：综合沟通和项目深挖适用于所有目标岗位；岗位核心能力和综合案例根据研发、测试、数据、运维或技术产品等岗位改变内容。行为面试作为“综合沟通”中的核心阶段，同时也可以单独专项练习。

### 决策 2：MS1 选择项目经历深挖，不锁定目标岗位

| 方案 | 能验证的核心能力 | 结论 |
|---|---|---|
| 自我介绍 | 语音、基本反馈 | 容易实现，但难证明技术差异化 |
| 行为面试 | STAR 结构与追问 | 有价值，但通用面试产品也能覆盖 |
| 项目经历深挖 | 简历上下文、技术取舍、连续追问、证据反馈 | 选择 |

以下仅是当前后端岗位演示数据的示例，不构成产品岗位限制：

```text
Q1: Tell me about a backend project where you solved a difficult scalability problem.
Q2: Why did you choose a bounded worker pool instead of starting one goroutine per task?
Q3: How did you validate the queue size and what measurable result did you get?
```

Q2 必须来自用户在 Q1 中提到的方案，Q3 必须继续追问验证和结果。其他目标岗位应替换为对应的专业问题。即使 MS1 使用固定脚本，也要通过这条因果链表达最终行为，而不是把 Go 或后端内容写入通用产品规则。

### 决策 3：受控材料 + AI 动态追问

| 方案 | 稳定性 | 个性化 | 结论 |
|---|---:|---:|---|
| 完全固定题库 | 高 | 低 | 只作为兜底 |
| 完全开放生成 | 低 | 高 | 不选 |
| 受控主题/目标 + 上下文追问 | 中高 | 高 | 选择 |

每个面试阶段至少定义主题、核心问题、可追问目标、结束条件和反馈维度。AI 可以调整措辞与追问顺序，但不能改变本阶段目标。

### 决策 4：训练模式与模拟模式分离提示行为

- 训练模式：允许提示关键词、查看回答框架、暂停和重答。
- 模拟模式：默认不提供提示，不允许任意回退；结束后统一反馈。
- 两种模式使用相同的职位和面试内容，避免维护两套题库。

## 基本概念与信息结构

| 概念 | 定义 |
|---|---|
| Interview Plan | 根据本次 Job Context 生成、经用户确认的四轮安排 |
| Interview Round | 完整面试中的一个主题轮次 |
| Practice Stage | 可从轮次中拆出、供单独重复训练的阶段 |
| Practice Mode | 完整模拟或分阶段专项练习 |
| Session | 用户一次连续练习及其状态 |
| Turn | 一个问题/追问与一个用户回答 |
| Follow-up Goal | 澄清、贡献、取舍、验证、结果或收束等追问目的 |
| Session State | 未开始、进行中、暂停、生成报告中、完成或失败 |

核心状态流：

```text
Draft Plan
  -> Confirmed Plan
  -> Session Started
  -> AI Question
  -> User Answer
  -> Follow-up or Advance
  -> Session Completed
  -> Feedback Generated
```

语音协议、模型编排、状态存储和超时策略属于架构 Proposal；本 Proposal 只约束用户可见行为。

原型调整建议：

- 四轮信息结构按本 Proposal 统一；岗位核心能力和综合案例的内容随目标岗位变化。
- “行为面试”明确放入综合沟通轮，并保留专项练习入口。
- MS1 演示默认直接进入第 3 轮项目经历深挖，具体演示岗位可以替换，避免让评审误以为产品只支持当前样例或四轮都已实现。
- 在至少一次追问中标明“基于你刚才提到的 worker pool”，让动态追问可感知。
- 训练模式显示提示；完整模拟模式不显示提示。
- 明确显示录音、转录、AI 思考、网络异常和恢复状态，但 MS1 可以 Mock。

## 验收标准

### Proposal 定稿验收

- [ ] 团队确认目标岗位由用户输入，不固定为 Go 后端或任何单一岗位。
- [ ] 团队确认岗位自适应四轮面试的名称、顺序和范围。
- [ ] 团队确认行为面试归属于综合沟通，并可单独专项练习。
- [ ] 团队确认 MS1 只完整演示项目经历深挖。
- [ ] 团队确认完整模拟和专项训练的提示规则不同。
- [ ] 团队确认采用受控材料 + AI 上下文追问，不做开放式无限出题。

### 两个月产品行为验收

1. 用户输入目标岗位并选择可选简历/JD 后，可以看到并确认与该岗位相关的四轮面试计划；计划不能继续显示其他样例岗位的专属内容。
2. 用户可以选择完整模拟，也可以选择至少一个阶段进行专项练习。
3. 一次项目深挖会话至少包含 3 个用户回答和 2 次追问。
4. 至少一次追问明确依赖用户上一轮回答或简历中的信息，不能只是固定下一题。
5. AI 的问题保持在本阶段目标内；偏离时可以收束回当前面试主题。
6. 训练模式可以查看最小提示，模拟模式默认不显示提示。
7. 用户可以暂停/继续可恢复的会话，并能从历史中查看转录。
8. 录音、识别或模型失败时有明确状态和可重试入口，不会丢失已完成轮次。
