## 一、Milestone 管理

### MS1 目标

**MS1：战略决策与首个可串联版本**

### MS1 交付边界

本轮只交付下面四类东西：

1. 产品 Proposal 定稿
2. 架构主干空骨架
3. 技术选型裁定
4. Live Demo

### MS1 不交付

- 不做完整 MVP
- 不做真实 ASR / TTS / S2S 全量接入
- 不做商业化系统

### Milestone 完成标准

- 所有关联 Issue 都有明确验收标准
- 每个 Issue 至少有一个明确负责人
- 每个 Issue 都挂了正确的 Milestone
- MS1 评审时能从 Milestone 直接看出“这一轮做成了什么”

## 二、Issue 管理

### Issue 总原则

- 一个 Issue 只做一件事
- 一天内可以完成为宜，超过就拆分
- 描述写验收标准，不写空泛口号
- 每个 Issue 至少挂 2 个标签：1 个状态/优先级 + 1 个技术域
- 所有 Issue 都必须关联 Milestone

### MS1 建议 Issue 结构

| Issue | 类型 | 责任 | 关键验收 |
|---|---|---|---|
| 产品 Proposal 定稿 | `[提案]` | 产品/调研主责 | 八字段齐全，结论可进入评审 |
| 架构主干空骨架 | `[功能]` 或 `[重构]` | 后端主责 | 模块能串联，接口为真、实现可桩化 |
| 技术选型裁定 | `[调研]` | 技术预研主责 | 有明确主选、备选和取舍依据 |
| Live Demo | `[功能]` | 全栈主责 | 能演示主链路，不要求大而全 |

### Issue 文案写法

推荐统一用下面结构：

```markdown
## 背景
为什么要做这件事，关联哪份文档或哪项决策。

## 要做的事
3-5 条可验证的事项，确保看完就能开工。

## 验收标准
写成可检查的结果，而不是过程描述。

## 关联
Milestone、依赖 Issue、参考文档。
```

## 三、Label 管理

### 推荐最小标签集

- `proposal`
- `Proposal-Accepted`
- `Proposal-Denied`
- `Proposal-NoPlan`
- `FullSpec`
- `MiniSpec`
- `P0`
- `P1`
- `P2`
- `前端`
- `后端`
- `全栈`
- `调研`
- `bug`
- `Need-Document`
- `Documented`

### 打标规则

- 提案类 Issue 创建时先打 `proposal`
- 定稿后用 `Proposal-Accepted` 替换 `proposal`
- 大规格用 `FullSpec`，小规格用 `MiniSpec`
- 需要后续补文档的功能，收尾时打 `Need-Document`
- 文档补完后改为 `Documented`

## 四、PR 管理

### PR 原则

- 每个 PR 只做一件事
- PR 必须能对应到一个 Issue
- PR 不能夹带无关改动
- AI 生成代码必须可复述、可验证、可 review

### PR 描述模板

```markdown
## 功能描述
做了什么，用户/开发者如何感知。

## 实现思路
关键文件、技术选型、为何这样改。

## 测试方式
可复制的验证步骤。

## 相关 Issue / 备注
关联 issue、已知限制、额外说明。
```

### Review 门槛

- 能讲清改动理由
- 能讲清改动范围
- 能讲清测试方式
- 通过必要验证后再合并

## 五、Release 管理

### Release 触发条件

当一个 Milestone 结束，并且关键交付物已经可演示、可追踪、可复盘时，创建 Release。

### Release 描述要求

- 列清本轮交付了哪些功能
- 明确哪些是主链路，哪些是后续工作
- 不写空泛总结，写能被 reviewer 复核的清单

## 六、MS1 执行节奏

### 评审前

- 核对所有 Issue 是否挂了 MS1
- 核对 Proposal 是否定稿
- 核对架构主干是否能串联
- 核对技术选型是否有主备方案

### 评审中

- 先讲目标，再讲范围，再讲取舍
- 只展示能证明方向成立的最小闭环
- 不把评审做成功能堆砌

### 评审后

- 把已完成的 Issue 收拢到 Milestone 说明中
- 更新状态标签
- 进入下一轮开发或拆分新的 Issue

## 七、推荐的实际操作顺序

1. 锁定 MS1 目标与交付边界
2. 保持 Proposal Issue 为单一主线
3. 用架构主干空骨架承接技术落地
4. 用技术选型 Issue 固化主链路决策
5. 用 Live Demo 验证可串联性
6. 评审结束后再做 Release 与文档状态收口
