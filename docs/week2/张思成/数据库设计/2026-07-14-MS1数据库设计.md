# MS1 数据库设计

> 日期：2026-07-14  
> 范围：SpeakUp MS1 英文面试主链路  
> 输入文档：`docs/superpowers/specs/2026-07-14-ms1-interview-prd-design.md`

## 1. 核心对象

本阶段先不直接进入字段设计，而是先确认系统中的核心业务对象和它们之间的关系。

MS1 需要进入数据库建模的核心对象如下：

| 对象 | 含义 |
|---|---|
| `User` | 用户账户。使用邮箱和密码登录，拥有自己的简历、面试计划、音频和历史记录。 |
| `AuthSession` | 登录态。记录用户当前登录会话，用于登录校验、退出登录和过期控制。 |
| `Resume` | 用户上传的简历文件。MS1 只支持 PDF，每个用户最多 3 份。 |
| `Experience` | 从简历中解析出来的一条经历，可以是项目、实习或工作经历。 |
| `InterviewPlan` | 一次面试准备计划。锁定目标岗位和用户确认过的一条经历快照。 |
| `Interviewer` | 面试计划下的一位面试官。默认生成 4 位，用户可以编辑、删除和添加。 |
| `InterviewSession` | 用户和某一位面试官进行的一场独立面试。 |
| `Turn` | 面试里的一轮问答，包含 AI 问题、用户有效回答、转录和音频索引。 |
| `FeedbackItem` | 针对某一个 `Turn` 生成的一条证据反馈。 |
| `RetryAttempt` | 用户针对某一个原始 `Turn` 做的一次同题复练。 |
| `AudioAsset` | 音频文件索引。真正的音频文件放对象存储，数据库只保存归属、位置和元信息。 |

## 2. 第一版 ER 关系

### 2.1 简历链路

```text
User 1 -> N Resume
Resume 1 -> N Experience
```

含义：

- 一个用户可以有多份简历，MS1 限制最多 3 份。
- 一份简历可以解析出多条经历。
- 一条经历属于某份简历，也属于某个用户。

### 2.2 面试链路

```text
User 1 -> N InterviewPlan
InterviewPlan 1 -> N Interviewer
Interviewer 1 -> N InterviewSession
InterviewSession 1 -> N Turn
Turn 1 -> 1 FeedbackItem
Turn 1 -> N RetryAttempt
```

含义：

- 一个用户可以创建多个面试计划。
- 一个面试计划包含 1-4 位面试官。
- 一位面试官可以开启多场独立面试。
- 一场面试固定最多 4 个有效 `Turn`。
- 每个有效 `Turn` 必须生成一条 `FeedbackItem`。
- 每个 `Turn` 可以被用户复练多次，每次复练生成一条新的 `RetryAttempt`。

### 2.3 音频关系

```text
Turn 0/1 -> 1 AudioAsset
RetryAttempt 0/1 -> 1 AudioAsset
```

含义：

- 原回答音频由 `Turn` 引用。
- 复练回答音频由 `RetryAttempt` 引用。
- 面试进行中时，`Turn` 可能已经生成问题，但还没有有效回答音频，所以音频关系允许暂时为空。
- 原回答和每次复练都分别保存独立音频，不互相覆盖。

### 2.4 完整关系草图

```text
User
├── AuthSession
├── Resume
│   └── Experience
├── AudioAsset
└── InterviewPlan
    ├── Experience Snapshot
    └── Interviewer
        └── InterviewSession
            └── Turn
                ├── AudioAsset
                ├── FeedbackItem
                └── RetryAttempt
                    └── AudioAsset
```

### 2.5 关键建模决策

`InterviewPlan` 不应该只引用 `Experience`，还必须保存一份创建计划时的 `Experience Snapshot`。

原因：

- 用户后续可能删除简历。
- 用户后续可能重新上传简历。
- 用户后续可能修改解析出来的经历。
- 历史面试、报告和复练不能因为简历变化而被改变。

因此：

```text
Experience = 当前简历解析结果
Experience Snapshot = 创建面试计划那一刻确认下来的历史上下文
```

`RetryAttempt` 建议主要归属于 `Turn`，同时引用 `FeedbackItem`。

原因：

- 复练的是同一道原问题，所以它本质上属于 `Turn`。
- 复练对比的是原反馈缺口，所以它也需要知道对应的 `FeedbackItem`。

## 3. 账号与权限相关表

MS1 账号体系只覆盖邮箱密码注册、登录、退出、登录态校验、数据权限和注销账号。

MS1 暂不实现：

- 邮箱验证
- 密码找回
- 手机号登录
- 第三方登录
- 账号绑定

### 3.1 `users`

`users` 表表示一个用户账户。

建议字段：

| 字段 | 含义 |
|---|---|
| `id` | 用户唯一 ID。其他表通过 `user_id` 指向它。 |
| `email` | 登录邮箱，系统内唯一。 |
| `password_hash` | 密码哈希，不能存明文密码。 |
| `default_resume_id` | 当前默认简历 ID。用户有多份简历时使用。 |
| `deleted_at` | 注销或软删除时间。为空表示账户仍可用。 |
| `created_at` | 创建时间。 |
| `updated_at` | 更新时间。 |

关键约束：

```text
email 唯一
password_hash 非空
```

密码不能以明文形式保存。数据库中不应该出现：

```text
password = "123456"
```

而应该保存密码哈希，例如：

```text
password_hash = "$argon2id$v=19$..."
```

登录时不需要也不应该解密密码，而是用密码哈希算法验证用户输入是否匹配 `password_hash`。

### 3.2 `auth_sessions`

`auth_sessions` 表表示用户的一次登录状态。

建议字段：

| 字段 | 含义 |
|---|---|
| `id` | 登录会话 ID。 |
| `user_id` | 当前登录态属于哪个用户。 |
| `token_hash` | 登录 token 的哈希值。数据库不直接保存原始 token。 |
| `expires_at` | 登录态过期时间。 |
| `revoked_at` | 退出登录时间。为空表示尚未退出。 |
| `created_at` | 创建时间。 |
| `updated_at` | 更新时间。 |

关系：

```text
User 1 -> N AuthSession
```

含义：

- 一个用户可以有多个登录会话。
- 例如电脑浏览器一个、手机浏览器一个。

登录流程：

```text
用户提交邮箱和密码
后端查询 users.email
验证密码是否匹配 password_hash
生成一段随机 session_token
把 hash(session_token) 存入 auth_sessions.token_hash
把原始 session_token 写入安全 Cookie 或返回给前端保存
```

接口鉴权流程：

```text
前端请求接口时带上 session_token
后端计算 hash(session_token)
查询 auth_sessions.token_hash
检查：
  1. 会话是否存在
  2. expires_at 是否仍然有效
  3. revoked_at 是否为空
  4. 对应 user 是否存在且未注销
通过后，当前请求用户就是 auth_sessions.user_id
```

退出登录流程：

```text
把当前 auth_sessions.revoked_at 设置为当前时间
```

之后同一个 token 再请求接口，就会被视为无效。

### 3.3 用户数据权限原则

所有用户私有业务表都建议直接保存 `user_id`。

例如：

```text
resumes.user_id
experiences.user_id
interview_plans.user_id
interview_sessions.user_id
turns.user_id
audio_assets.user_id
```

这样做不是多余，而是为了让权限校验简单、安全、查询快。

查询用户私有数据时，不能只按业务 ID 查询：

```text
SELECT * FROM audio_assets
WHERE id = 'audio_001';
```

而应该带上当前登录用户：

```text
SELECT * FROM audio_assets
WHERE id = 'audio_001'
AND user_id = current_user_id;
```

即使用户 A 猜到了用户 B 的音频 ID，只要 `user_id` 不匹配，后端也不能返回数据。

### 3.4 注销账号策略

PRD 要求注销后删除账户、简历原文件、解析结果、经历、计划、面试官、场次、音频、转录、报告、反馈和复练记录。

因此账号注销需要同时处理两类数据：

```text
关系数据库数据
对象存储文件
```

建议流程：

```text
1. 用户二次确认注销。
2. 后端查询该用户所有 resumes.storage_key 和 audio_assets.storage_key。
3. 删除对象存储里的 PDF 和音频文件。
4. 删除或软删除 users 相关数据库记录。
5. 让该用户所有 auth_sessions 失效。
6. 删除完成后退出登录。
```

MS1 可以先采用硬删除加级联删除策略；如果担心删除失败或需要审计，可以先软删除用户，再由后台任务异步清理。

### 3.5 第 3 步结论

账号与权限第一版表：

```text
users
  1 -> N auth_sessions
```

并且所有用户私有业务表都冗余保存：

```text
user_id
```

核心原则：

```text
登录靠 auth_sessions
数据隔离靠每张私有表的 user_id
密码和 token 都只存 hash
注销必须同时删除数据库记录和对象存储文件
```

## 4. 简历与经历相关表

本阶段设计 `resumes` 和 `experiences` 两张表，覆盖 PDF 简历上传、解析状态、默认简历、解析出的经历列表，以及后续创建面试计划时的经历确认。

### 4.1 业务规则回顾

PRD 对简历和经历有几个关键要求：

```text
每个用户最多保存 3 份简历
MS1 只支持 PDF
单份 PDF 最大 10 MB
简历有上传和解析状态
一份简历可以解析出多条项目、实习或工作经历
默认选中最近一条经历
解析失败或无经历时允许手动填写
创建面试计划时必须保存用户确认后的经历快照
删除简历不影响历史面试计划和报告
```

因此数据库要区分两个概念：

```text
Resume / Experience = 当前简历文件和当前解析结果
Experience Snapshot = 创建面试计划时复制下来的历史上下文
```

`Experience Snapshot` 不放在 `experiences` 表里，而是后续放在 `interview_plans` 表里。

### 4.2 `resumes`

`resumes` 表表示用户上传的一份 PDF 简历。

建议字段：

| 字段 | 含义 |
|---|---|
| `id` | 简历唯一 ID。 |
| `user_id` | 简历所属用户。 |
| `file_name` | 用户上传时的原始文件名。 |
| `mime_type` | 文件类型。MS1 应为 `application/pdf`。 |
| `file_size_bytes` | 文件大小，用于校验 10 MB 限制和展示。 |
| `storage_key` | PDF 在对象存储中的位置。不是公开 URL。 |
| `parse_status` | 解析状态。 |
| `parse_error` | 解析失败原因。解析成功时为空。 |
| `is_default` | 是否为默认简历。 |
| `parsed_at` | 解析完成时间。 |
| `deleted_at` | 删除时间。若采用硬删除，可不保留。 |
| `created_at` | 创建时间。 |
| `updated_at` | 更新时间。 |

`parse_status` 建议枚举：

```text
uploaded    已上传，等待解析
parsing     解析中
available   解析成功，可使用
failed      解析失败
```

也可以把 `uploaded` 命名为 `pending`，但要和前端文案区分：

```text
数据库状态：uploaded / parsing / available / failed
页面文案：上传中 / 解析中 / 可用 / 解析失败
```

关系：

```text
User 1 -> N Resume
```

约束：

```text
resumes.user_id 必填
resumes.storage_key 必填
同一用户最多 3 份未删除简历
同一用户最多 1 份默认简历
```

其中“最多 3 份”和“最多 1 份默认简历”不一定完全靠数据库约束实现，MS1 可以先由业务逻辑保证：

```text
上传前查询用户当前未删除简历数量
如果 >= 3，拒绝上传

设置默认简历时
先把该用户其他简历 is_default 设为 false
再把目标简历 is_default 设为 true
```

### 4.3 `experiences`

`experiences` 表表示从某份简历里解析出来的一条经历。

建议字段：

| 字段 | 含义 |
|---|---|
| `id` | 经历唯一 ID。 |
| `user_id` | 经历所属用户。冗余保存，便于权限校验。 |
| `resume_id` | 来源简历 ID。 |
| `type` | 经历类型，例如项目、实习、工作。 |
| `title` | 经历标题，例如项目名称或岗位名称。 |
| `organization` | 公司、学校、组织或项目所属方。 |
| `start_date` | 开始日期。允许为空。 |
| `end_date` | 结束日期。允许为空。 |
| `sort_order` | 在简历中的出现顺序。用于日期缺失时选择默认经历。 |
| `background` | 项目或经历背景。 |
| `responsibility` | 用户个人职责。 |
| `achievements` | 成果、指标或影响。 |
| `raw_extract_json` | 解析模型输出的原始结构化结果，便于排查和后续优化。 |
| `created_at` | 创建时间。 |
| `updated_at` | 更新时间。 |

关系：

```text
Resume 1 -> N Experience
User 1 -> N Experience
```

这里 `experiences.user_id` 理论上可以从 `resume_id` 推导出来，但仍建议保存。

原因：

```text
权限查询更直接
历史排查更方便
避免每次都 join resumes 才能知道这条经历属于谁
```

### 4.4 默认经历选择规则

PRD 要求：

```text
按结束日期倒序排列，默认选中最近一条经历。
日期缺失时，默认选中简历中最先出现的一条经历。
```

对应数据库设计：

```text
end_date 用于判断最近经历
sort_order 用于记录简历原文中的出现顺序
```

默认选择逻辑可以放在业务层：

```text
优先选择 end_date 最新的经历
如果所有经历都没有 end_date，则选择 sort_order 最小的经历
```

不建议在 `resumes` 表里固定保存 `default_experience_id` 作为第一版必需字段。

原因：

```text
默认经历更多是创建面试计划时的推荐选择
用户最终确认哪条经历，应该体现在 interview_plans 的快照里
```

如果后续产品希望用户在简历页保存“默认经历”，可以再给 `resumes` 增加：

```text
default_experience_id
```

但 MS1 第一版可以先不加。

### 4.5 手动填写经历

PRD 要求解析失败或没有识别到经历时，用户可以手动填写，不要求重新上传。

这里有两种设计：

方案 A：手动经历也写入 `experiences`

```text
experiences.resume_id = 某份解析失败的 resume_id
experiences.type = "manual"
```

优点：

```text
经历列表统一
后续创建计划可以照常选择 Experience
```

缺点：

```text
手动经历依然挂在一份简历下面，语义稍微有点绕
```

方案 B：手动经历不写入 `experiences`，只在创建计划时写入 `interview_plans.experience_snapshot`

优点：

```text
更贴合“这只是本次面试确认的背景”
```

缺点：

```text
如果用户还没创建计划，手动填写内容不容易暂存
```

MS1 建议采用方案 A：

```text
手动填写经历也保存到 experiences
并用 type 或 source 字段标记来源
```

为了表达来源，可以给 `experiences` 增加：

| 字段 | 含义 |
|---|---|
| `source` | `parsed` 或 `manual`，表示模型解析还是用户手动填写。 |

因此 `experiences` 第一版建议补充：

```text
source
```

### 4.6 简历删除策略

删除简历时，PRD 要求：

```text
删除原 PDF 和当前解析结果
历史计划只保留确认后的文本
不保留对已删除文件的引用
```

对应设计：

```text
删除 resumes 时，级联删除 experiences
删除对象存储中的 PDF 文件
interview_plans 不能依赖 experiences 才能展示历史
```

因此 `interview_plans` 后续需要保存：

```text
resume_id                  可选引用，简历删除后可置空
source_experience_id        可选引用，经历删除后可置空
experience_snapshot_json    必填，保存用户确认后的经历文本
```

这样删除简历后：

```text
当前简历文件没了
当前解析结果没了
历史面试计划仍然能展示当时确认过的岗位和经历
```

### 4.7 第 4 步结论

简历与经历第一版表：

```text
users
  1 -> N resumes

resumes
  1 -> N experiences
```

推荐表：

```text
resumes
- id
- user_id
- file_name
- mime_type
- file_size_bytes
- storage_key
- parse_status
- parse_error
- is_default
- parsed_at
- deleted_at
- created_at
- updated_at

experiences
- id
- user_id
- resume_id
- source
- type
- title
- organization
- start_date
- end_date
- sort_order
- background
- responsibility
- achievements
- raw_extract_json
- created_at
- updated_at
```

核心原则：

```text
PDF 文件放对象存储，resumes 只保存 storage_key
解析出来的经历放 experiences
用户最终确认用于面试的经历，要在 interview_plans 中保存快照
删除简历不能影响历史计划和报告
```

## 5. 面试计划与面试官相关表

本阶段设计 `interview_plans` 和 `interviewers` 两张表，覆盖目标岗位、练习要求、经历快照、默认 4 位面试官、面试官编辑，以及 1-4 位面试官数量限制。

### 5.1 业务规则回顾

PRD 对面试计划与面试官有几个关键要求：

```text
用户输入目标岗位
用户选择或确认一条经历
创建计划时保存岗位和经历快照
系统默认生成 4 位面试官
用户可以编辑面试官姓名、职责、风格和关注点
用户可以删除或重新添加面试官
最终必须保留 1-4 位面试官
多位面试官代表多场独立面试，不是多人同场面试
```

因此需要明确区分：

```text
InterviewPlan = 一次面试准备计划，保存共享上下文
Interviewer = 计划下的一位面试官，表达不同追问视角
InterviewSession = 用户和某一位面试官的一场真实面试
```

第 5 步只设计前两者，`InterviewSession` 放到第 6 步设计。

### 5.2 `interview_plans`

`interview_plans` 表表示一次面试准备计划。

它的核心职责是锁定本次准备的上下文：

```text
用户是谁
目标岗位是什么
选择了哪份简历
确认了哪条经历
当时确认后的经历文本是什么
练习重点和补充要求是什么
```

建议字段：

| 字段 | 含义 |
|---|---|
| `id` | 面试计划唯一 ID。 |
| `user_id` | 计划所属用户。 |
| `resume_id` | 创建计划时使用的简历 ID。简历删除后可置空。 |
| `source_experience_id` | 创建计划时来源经历 ID。经历删除后可置空。 |
| `target_role` | 目标岗位，必填。 |
| `job_description` | 岗位描述，可为空。 |
| `practice_focus` | 练习重点，可为空。 |
| `extra_requirements` | 用户补充要求，可为空。 |
| `experience_snapshot_json` | 用户确认后的经历快照，必填。 |
| `status` | 计划状态。 |
| `created_at` | 创建时间。 |
| `updated_at` | 更新时间。 |

关系：

```text
User 1 -> N InterviewPlan
Resume 0/1 -> N InterviewPlan
Experience 0/1 -> N InterviewPlan
```

这里 `resume_id` 和 `source_experience_id` 建议允许为空。

原因：

```text
用户可能删除简历
删除简历会删除当前解析出的 experiences
历史计划不能因此无法展示
```

真正用于历史展示和面试上下文的是：

```text
experience_snapshot_json
```

### 5.3 `experience_snapshot_json`

`experience_snapshot_json` 是 `interview_plans` 里最重要的字段之一。

它保存的是用户创建计划时确认过的经历内容，而不是简单引用 `experiences.id`。

建议结构：

```json
{
  "source": "parsed",
  "type": "project",
  "title": "AI Interview Coach Prototype",
  "organization": "SpeakUp",
  "startDate": "2026-06",
  "endDate": "2026-07",
  "background": "Built an English interview practice prototype for job candidates.",
  "responsibility": "Designed the interview flow and implemented the feedback module.",
  "achievements": "Completed the first playable prototype and validated the core interview loop."
}
```

字段可以根据前端表单略有调整，但原则是：

```text
快照必须足够支撑后续生成问题、生成反馈、展示历史报告
不要只保存 experience_id
```

如果用户在创建计划时修改了项目名称、背景、职责或成果，快照中应该保存修改后的版本。

### 5.4 `status`

`interview_plans.status` 用于描述计划本身的状态。

第一版建议：

```text
draft       草稿，尚未确认面试官
ready       已生成并确认面试官，可以开始面试
archived    用户归档或后续不再使用
```

MS1 也可以更简单，只用：

```text
ready
```

因为真实进度更多由 `interviewers` 和 `interview_sessions` 计算得出。

例如：

```text
某个计划是否有进行中面试
某个计划是否全部完成
最近一次面试是什么时间
```

这些不建议直接写死在 `interview_plans.status` 里，而是从下面的 session 数据聚合出来。

### 5.5 `interviewers`

`interviewers` 表表示某个面试计划下的一位面试官。

它的核心职责是保存这一位面试官的追问视角：

```text
姓名
职责
风格
关注点
排序
```

建议字段：

| 字段 | 含义 |
|---|---|
| `id` | 面试官唯一 ID。 |
| `user_id` | 所属用户。冗余保存，便于权限校验。 |
| `plan_id` | 所属面试计划。 |
| `name` | 面试官姓名。 |
| `role` | 面试官职责或身份，例如技术负责人、业务面试官、HR。 |
| `style` | 面试风格，例如温和、追问严格、结构化。 |
| `focus_areas` | 关注点，可以是文本或 JSON。 |
| `sort_order` | 排序。 |
| `created_at` | 创建时间。 |
| `updated_at` | 更新时间。 |

关系：

```text
InterviewPlan 1 -> N Interviewer
User 1 -> N Interviewer
```

这里同样建议保存 `interviewers.user_id`。

原因：

```text
用户私有数据权限校验更直接
查询某个面试官时不必一路 join 到 interview_plans 才知道所属用户
```

### 5.6 面试官数量限制

PRD 要求：

```text
一个计划默认生成 4 位面试官
用户最终必须保留 1-4 位面试官
不能保存 0 位或超过 4 位
```

这个规则建议主要在业务层实现：

```text
创建计划后默认插入 4 条 interviewers
删除面试官前检查删除后是否仍 >= 1
新增面试官前检查新增后是否 <= 4
确认计划前检查 interviewers 数量是否在 1-4
```

数据库层可以配合唯一约束或索引，但“不超过 4 条子记录”这类约束用数据库表达会比较绕，MS1 不建议复杂化。

### 5.7 多面试官语义

需要特别注意：

```text
多面试官不表示多人同时在线
多面试官表示同一个计划下的多个独立面试入口
```

因此关系不是：

```text
InterviewPlan -> InterviewSession -> 多个 Interviewer
```

而是：

```text
InterviewPlan -> Interviewer -> InterviewSession
```

原因：

```text
每次 Session 只有一位面试官
每位面试官拥有独立进度、问答、报告和复练历史
完成面试官 A 的面试，不会自动完成面试官 B
同一位面试官也可以重复开始多场 Session
```

这也是为什么 `InterviewSession` 应该挂在 `Interviewer` 下面，而不是直接挂在 `InterviewPlan` 下面。

### 5.8 历史页查询视角

历史首页按计划展示时，需要从 `interview_plans` 聚合下面的数据：

```text
计划目标岗位
计划创建或更新时间
最近面试官
最近一次 session 时间
是否有进行中 session
是否有已完成 session
```

所以常见查询关系是：

```text
interview_plans
  -> interviewers
    -> interview_sessions
```

为了支持这个查询，后续建表时建议给这些字段加索引：

```text
interview_plans.user_id
interviewers.plan_id
interview_sessions.interviewer_id
interview_sessions.plan_id
interview_sessions.user_id
```

其中 `interview_sessions.plan_id` 虽然可以通过 `interviewer_id` 推导，但仍可冗余保存，方便历史聚合查询。

### 5.9 第 5 步结论

面试计划与面试官第一版表：

```text
users
  1 -> N interview_plans

interview_plans
  1 -> N interviewers
```

推荐表：

```text
interview_plans
- id
- user_id
- resume_id
- source_experience_id
- target_role
- job_description
- practice_focus
- extra_requirements
- experience_snapshot_json
- status
- created_at
- updated_at

interviewers
- id
- user_id
- plan_id
- name
- role
- style
- focus_areas
- sort_order
- created_at
- updated_at
```

核心原则：

```text
InterviewPlan 保存共享上下文和经历快照
Interviewer 保存某一位面试官的追问视角
一个计划必须有 1-4 位面试官
多面试官代表多场独立面试入口
Session 后续应该挂在 Interviewer 下面
```

## 6. 面试场次与问答相关表

本阶段设计 `interview_sessions` 和 `turns` 两张表，覆盖单场面试、面试状态恢复、4 次有效问答、AI 问题、用户回答、打断状态、转录和原回答音频索引。

### 6.1 业务规则回顾

PRD 对面试场次和问答有几个关键要求：

```text
每次 Session 只有一位面试官
每位面试官可以产生多场相互独立的面试
一场完整面试固定包含 4 个有效 Turn
只有获得可用转录的回答才计为完成一个 Turn
空白、过短或识别失败的回答不增加次数
第四次有效回答保存后自动结束并进入报告生成
用户可以提前结束，但只生成未完成报告，不进入同题复练
当前回答断线时，已完成 Turn 不丢失，当前题重答
报告生成失败可以单独重试，不要求重做面试
```

因此需要区分：

```text
InterviewSession = 一场面试的整体状态
Turn = 一道有效问答的持久化记录
```

注意：

```text
未完成、无有效转录、断线丢弃的回答，不应该创建为有效 Turn
```

如果后续需要排查实时语音过程，可以单独设计事件日志表；MS1 第一版不必引入。

### 6.2 `interview_sessions`

`interview_sessions` 表表示用户与某一位面试官进行的一场独立面试。

建议字段：

| 字段 | 含义 |
|---|---|
| `id` | 面试场次唯一 ID。 |
| `user_id` | 所属用户。 |
| `plan_id` | 所属面试计划。冗余保存，便于历史聚合查询。 |
| `interviewer_id` | 本场面试对应的面试官。 |
| `status` | 场次状态。 |
| `current_turn_index` | 当前进行到第几问。用于恢复进行中面试。 |
| `started_at` | 开始时间。 |
| `completed_at` | 完成时间。 |
| `ended_early_at` | 用户提前结束时间。 |
| `duration_seconds` | 面试总时长。可以完成后计算写入。 |
| `report_status` | 报告生成状态。 |
| `report_summary_json` | 报告顶部汇总内容。 |
| `report_error` | 报告生成失败原因。 |
| `created_at` | 创建时间。 |
| `updated_at` | 更新时间。 |

关系：

```text
Interviewer 1 -> N InterviewSession
InterviewPlan 1 -> N InterviewSession
User 1 -> N InterviewSession
```

虽然 `plan_id` 可以通过 `interviewer_id` 间接得到，但建议在 `interview_sessions` 里冗余保存。

原因：

```text
历史页通常按计划聚合 sessions
查询某个计划下最近面试记录时更直接
权限校验和排查更方便
```

### 6.3 `interview_sessions.status`

第一版建议：

```text
not_started     已创建但尚未开始，可选
in_progress     进行中
completed       已完成 4 个有效 Turn
ended_early     用户提前结束
failed          场次异常失败
```

也可以不创建 `not_started` 状态：

```text
用户点击开始后才创建 InterviewSession
创建即 in_progress
```

MS1 更推荐后者：

```text
未开始状态属于 Interviewer 的页面展示状态
真正的 InterviewSession 从用户点击开始面试后创建
```

这样可以避免产生很多没有实际内容的空 session。

### 6.4 `report_status`

报告生成状态建议从 `interview_sessions.status` 中拆出来。

原因：

```text
面试完成和报告生成不是同一件事
第四次有效回答保存后，session 可以 completed
但报告可能还在 generating 或 failed
```

建议枚举：

```text
pending       尚未开始生成
generating    生成中
ready         已生成
failed        生成失败
```

这样能支持 PRD 中的要求：

```text
报告生成失败可以单独重试，不要求重新完成面试
```

### 6.5 `turns`

`turns` 表表示一场面试中的一道有效问答。

建议字段：

| 字段 | 含义 |
|---|---|
| `id` | Turn 唯一 ID。 |
| `user_id` | 所属用户。 |
| `session_id` | 所属面试场次。 |
| `turn_index` | 第几问，建议使用 1-4。 |
| `fixed_goal` | 固定目标，例如项目概览、个人贡献、方案取舍、验证与结果。 |
| `question_text` | AI 生成的完整问题文本。 |
| `played_question_text` | 用户实际听到的 AI 问题文本。发生打断时可能短于完整问题。 |
| `was_interrupted` | AI 提问是否被用户打断。 |
| `answer_audio_asset_id` | 用户原回答音频 ID。 |
| `answer_transcript` | 用户原回答转录文本。 |
| `answered_at` | 用户有效回答完成时间。 |
| `created_at` | 创建时间。 |
| `updated_at` | 更新时间。 |

关系：

```text
InterviewSession 1 -> N Turn
Turn 0/1 -> 1 AudioAsset
Turn 1 -> 1 FeedbackItem
Turn 1 -> N RetryAttempt
```

`turn_index` 建议使用 1-4，而不是 0-3。

原因：

```text
更贴近页面文案：第 1 问、第 2 问、第 3 问、第 4 问
报告卡展示更直接
```

### 6.6 四个固定目标

PRD 固定了一场完整面试的 4 个目标：

| turn_index | fixed_goal |
|---|---|
| 1 | `project_overview` 项目概览 |
| 2 | `personal_contribution` 个人贡献 |
| 3 | `tradeoff_decision` 方案取舍 |
| 4 | `validation_result` 验证与结果 |

`fixed_goal` 存在的意义：

```text
让报告知道这一问应该评价什么
让动态追问有稳定骨架
让历史记录能解释每个问题的目标
```

具体问题文本仍然应该动态生成：

```text
第 2 问引用第 1 问回答
第 3 问引用前面提到的方案
第 4 问追问验证、边界、结果或量化证据
```

所以：

```text
fixed_goal = 稳定目标
question_text = 本次实际问题
```

### 6.7 打断状态怎么存

PRD 要求：

```text
AI 播放问题时用户可以打断
被打断的 AI 内容标记为已中断
界面只展示用户实际听到的部分
未播放文本不能当作 AI 已经说过的内容进入用户可见历史
```

对应字段：

```text
question_text
played_question_text
was_interrupted
```

示例：

```text
question_text:
"Can you walk me through the background of this project and explain why it mattered for the target role?"

played_question_text:
"Can you walk me through the background of this project..."

was_interrupted:
true
```

这样报告和历史页可以展示：

```text
用户实际听到的问题版本
```

同时系统内部仍然保留完整生成问题，便于排查。

### 6.8 什么情况下创建 Turn

建议规则：

```text
AI 问题生成成功后，可以先创建 Turn 草稿
但只有拿到有效音频和有效转录后，才算一个有效 Turn
```

如果希望更简单，也可以：

```text
等用户有效回答完成后，再一次性创建 Turn
```

MS1 推荐第二种，先简单：

```text
有效回答完成
拿到音频 storage_key
拿到可用转录
创建 audio_assets
创建 turns
更新 session.current_turn_index
```

这样可以避免数据库里出现很多无效 Turn。

但有一个例外：

```text
如果需要支持刷新页面后恢复“当前 AI 问题”，就需要在回答前保存当前问题。
```

第一版可以用 `interview_sessions` 里的临时状态或前端会话状态解决；如果后续恢复要求更强，再增加：

```text
pending_question_text
pending_fixed_goal
```

或单独建实时会话状态表。

### 6.9 `current_turn_index`

`interview_sessions.current_turn_index` 用于恢复进行中面试。

建议语义：

```text
表示已经完成的有效 Turn 数量
```

例如：

```text
0 = 还没有完成有效回答，下一题是第 1 问
1 = 已完成第 1 问，下一题是第 2 问
2 = 已完成第 2 问，下一题是第 3 问
3 = 已完成第 3 问，下一题是第 4 问
4 = 已完成第 4 问，面试完成
```

这样比“当前正在第几问”更不容易歧义。

页面展示时可以转换：

```text
下一题编号 = current_turn_index + 1
```

### 6.10 提前结束

用户可以提前结束，但 PRD 说：

```text
只生成未完成报告
不进入同题复练
```

对应设计：

```text
interview_sessions.status = ended_early
interview_sessions.ended_early_at = 当前时间
turns 中只保留已经完成的有效 Turn
report_status 可以进入 generating / ready
```

反馈与复练规则：

```text
已完成 Turn 可以生成反馈
但未完成场次不开放 RetryAttempt
```

这个限制可以在业务层判断：

```text
只有 session.status = completed 才允许同题复练
```

### 6.11 第 6 步结论

面试场次与问答第一版表：

```text
interviewers
  1 -> N interview_sessions

interview_sessions
  1 -> N turns
```

推荐表：

```text
interview_sessions
- id
- user_id
- plan_id
- interviewer_id
- status
- current_turn_index
- started_at
- completed_at
- ended_early_at
- duration_seconds
- report_status
- report_summary_json
- report_error
- created_at
- updated_at

turns
- id
- user_id
- session_id
- turn_index
- fixed_goal
- question_text
- played_question_text
- was_interrupted
- answer_audio_asset_id
- answer_transcript
- answered_at
- created_at
- updated_at
```

核心原则：

```text
InterviewSession 表示一场独立面试
Turn 表示一次有效问答
一场完整面试固定 4 个有效 Turn
current_turn_index 建议表示已完成有效 Turn 数量
报告状态 report_status 要和场次状态 status 分开
AI 打断需要同时保存完整问题和实际播放问题
无有效转录的回答不应该计入 Turn
```

## 7. 音频文件表

本阶段设计 `audio_assets` 表，覆盖原回答音频和复练回答音频的文件索引。

### 7.1 核心原则

音频本体不建议存进关系数据库。

推荐架构：

```text
关系数据库：保存音频元数据和 storage_key
对象存储：保存真实音频文件
```

例如：

```text
D1 / PostgreSQL / MySQL:
audio_assets 表

R2 / S3 / OSS / COS:
真实 .webm / .mp3 / .wav 文件
```

数据库里不要保存永久公开 URL。

应该保存：

```text
storage_key = users/u_123/sessions/s_456/turns/t_001/original.webm
```

播放时由后端校验权限，再返回短期签名 URL，或者由后端代理音频流。

### 7.2 `audio_assets`

`audio_assets` 表表示一份受保护音频文件的索引。

建议字段：

| 字段 | 含义 |
|---|---|
| `id` | 音频唯一 ID。 |
| `user_id` | 音频所属用户。 |
| `storage_key` | 音频在对象存储中的路径。不是公开 URL。 |
| `mime_type` | 音频类型，例如 `audio/webm`。 |
| `file_size_bytes` | 文件大小。 |
| `duration_ms` | 音频时长。 |
| `transcript` | ASR 转录文本。可冗余保存。 |
| `transcript_status` | 转录状态。 |
| `created_at` | 创建时间。 |
| `updated_at` | 更新时间。 |

关系：

```text
User 1 -> N AudioAsset
Turn 0/1 -> 1 AudioAsset
RetryAttempt 0/1 -> 1 AudioAsset
```

### 7.3 为什么不直接把 transcript 只存在 turns

可以在 `turns.answer_transcript` 保存一份，也可以在 `audio_assets.transcript` 保存一份。

建议第一版允许适度冗余：

```text
audio_assets.transcript = 音频文件对应的转录
turns.answer_transcript = 原回答在业务上下文中的转录
retry_attempts.transcript = 复练回答在业务上下文中的转录
```

原因：

```text
AudioAsset 可以独立表达文件与转录状态
Turn / RetryAttempt 查询报告时不需要额外 join 也能拿到文本
后续如果重跑 ASR，可以对比不同版本
```

MS1 可以保持简单：写入音频时同步写业务表转录，`audio_assets.transcript` 作为文件级备份。

### 7.4 音频访问流程

推荐后端代理播放：

```text
前端请求 /api/audio/:audioId/stream
后端根据 session_token 得到 current_user_id
后端查询 audio_assets:
  id = audioId
  user_id = current_user_id
如果查不到，返回 404 或 403
如果查到，用 storage_key 从对象存储读取文件
后端把音频流返回给前端
```

优点：

```text
对象存储地址不暴露给前端
权限逻辑统一在后端
适合理解和实现 MS1
```

如果后续要优化性能，可以改为短期签名 URL。

### 7.5 第 7 步结论

推荐表：

```text
audio_assets
- id
- user_id
- storage_key
- mime_type
- file_size_bytes
- duration_ms
- transcript
- transcript_status
- created_at
- updated_at
```

核心原则：

```text
音频文件放对象存储
数据库只保存 storage_key 和元数据
不保存永久公开 URL
播放前必须校验 user_id
原回答和复练回答分别保存独立 audio_assets
```

## 8. 反馈与复练相关表

本阶段设计 `feedback_items` 和 `retry_attempts` 两张表，覆盖逐题证据反馈、改进目标、同题复练和版本对比。

### 8.1 业务规则回顾

PRD 要求：

```text
每个有效 Turn 必须生成一条 FeedbackItem
报告中按问答顺序展示 4 张回答复盘卡
反馈必须引用用户原话
反馈只评价项目证据是否讲清楚
同一道题可以重复复练
每次复练保存新音频和新转录
复练只检查原诊断缺口
报告卡展示最近一次复练结果，并可展开查看全部版本
```

### 8.2 `feedback_items`

`feedback_items` 表表示针对某个 `Turn` 的一条反馈。

建议字段：

| 字段 | 含义 |
|---|---|
| `id` | 反馈唯一 ID。 |
| `user_id` | 所属用户。 |
| `session_id` | 所属面试场次。 |
| `turn_id` | 对应的 Turn。 |
| `status` | `evidence_complete` 或 `needs_more_evidence`。 |
| `evidence_quote` | 引用用户原话的关键证据。 |
| `diagnosis` | 中文诊断说明。 |
| `improvement_goal` | 下一次改进目标。 |
| `example_answer` | 示例表达，不能虚构用户没有提供的事实。 |
| `gap_json` | 结构化缺口，供复练对比使用。 |
| `created_at` | 创建时间。 |
| `updated_at` | 更新时间。 |

关系：

```text
Turn 1 -> 1 FeedbackItem
InterviewSession 1 -> N FeedbackItem
User 1 -> N FeedbackItem
```

`feedback_items.turn_id` 应该唯一，保证一个 Turn 只有一条主反馈。

### 8.3 `gap_json`

`gap_json` 用来保存复练时要检查的缺口。

示例：

```json
{
  "missing": [
    {
      "key": "personal_contribution",
      "label": "个人贡献",
      "description": "需要说明自己具体负责的任务，而不是只说团队完成了项目。"
    },
    {
      "key": "result_metric",
      "label": "结果指标",
      "description": "需要补充验证结果或量化影响。"
    }
  ]
}
```

这样复练时不需要重新生成一套无关评分，而是对照原缺口判断：

```text
哪些已补充
哪些仍缺少
```

### 8.4 `retry_attempts`

`retry_attempts` 表表示用户针对某个原始 Turn 的一次同题复练。

建议字段：

| 字段 | 含义 |
|---|---|
| `id` | 复练唯一 ID。 |
| `user_id` | 所属用户。 |
| `turn_id` | 被复练的原始 Turn。 |
| `feedback_item_id` | 对应原反馈，用于读取原缺口。 |
| `attempt_index` | 第几次复练，从 1 开始。 |
| `audio_asset_id` | 本次复练回答音频。 |
| `transcript` | 本次复练转录。 |
| `filled_gap_json` | 已补充内容。 |
| `missing_gap_json` | 仍缺少内容。 |
| `created_at` | 创建时间。 |

关系：

```text
Turn 1 -> N RetryAttempt
FeedbackItem 1 -> N RetryAttempt
RetryAttempt 0/1 -> 1 AudioAsset
```

`attempt_index` 应该在同一个 `turn_id` 下递增。

例如：

```text
turn_001
  retry_attempt attempt_index = 1
  retry_attempt attempt_index = 2
  retry_attempt attempt_index = 3
```

### 8.5 复练开放条件

PRD 要求：

```text
用户提前结束时只生成未完成报告，不进入同题复练
```

因此业务层应限制：

```text
只有 interview_sessions.status = completed
才允许创建 retry_attempts
```

已完成 session 中的任意 Turn 都可以复练，不限制次数。

### 8.6 第 8 步结论

推荐表：

```text
feedback_items
- id
- user_id
- session_id
- turn_id
- status
- evidence_quote
- diagnosis
- improvement_goal
- example_answer
- gap_json
- created_at
- updated_at

retry_attempts
- id
- user_id
- turn_id
- feedback_item_id
- attempt_index
- audio_asset_id
- transcript
- filled_gap_json
- missing_gap_json
- created_at
```

核心原则：

```text
每个有效 Turn 必须有一条 FeedbackItem
反馈围绕用户原话证据，不做虚假技术裁判
RetryAttempt 主要属于 Turn，同时引用 FeedbackItem
每次复练都保存独立音频和转录
复练只对比原反馈缺口
```

## 9. 状态字段与枚举

本阶段统一整理主要状态字段。状态字段要少而清晰，避免页面和后端各自发明一套状态。

### 9.1 简历解析状态


建议枚举：

### 9.2 面试计划状态

字段：


建议枚举：

```text
draft       草稿，尚未确认面试官
```

MS1 可以先只使用 `ready`，后续再扩展。

## 10. 删除策略

本阶段整理普通删除、简历删除、账号注销的策略。

### 10.1 普通关系删除

推荐策略：

```text
users 删除 -> 级联删除 auth_sessions、resumes、experiences、interview_plans、audio_assets 等用户数据
resumes 删除 -> 级联删除 experiences
interview_plans 删除 -> 级联删除 interviewers、interview_sessions、turns、feedback_items、retry_attempts
interviewers 删除 -> 级联删除 interview_sessions
interview_sessions 删除 -> 级联删除 turns、feedback_items、retry_attempts
turns 删除 -> 级联删除 feedback_items、retry_attempts
```

注意：

```text
数据库级联只能删除数据库记录
对象存储文件需要业务代码额外删除
```

### 10.2 简历删除

删除简历时：

```text
删除对象存储中的 PDF
删除 resumes 记录
删除该 resume 下的 experiences
不删除 interview_plans
interview_plans.resume_id / source_experience_id 可以置空
interview_plans.experience_snapshot_json 保持不变
```

这样历史报告不会坏。

### 10.3 音频删除

删除音频记录前，要先知道对象存储路径：

```text
audio_assets.storage_key
```

建议流程：

```text
查询要删除的 audio_assets
删除对象存储文件
删除 audio_assets 数据库记录
```

如果对象存储删除失败，不建议先删除数据库记录，否则会留下无法追踪的孤儿文件。

### 10.4 账号注销

账号注销是全量删除。

建议流程：

```text
1. 用户二次确认。
2. 后端锁定该用户，防止并发创建新数据。
3. 查询该用户所有 resumes.storage_key。
4. 查询该用户所有 audio_assets.storage_key。
5. 删除对象存储中的 PDF 和音频。
6. 删除数据库中该用户相关数据。
7. 使所有 auth_sessions 失效。
8. 返回注销完成并退出登录。
```

如果担心第 5 步部分失败：

```text
可以先把 users.deleted_at 置为当前时间
禁止用户继续登录和访问数据
后台任务继续重试删除文件和数据库记录
```

MS1 第一版可以先实现同步删除，但文档上要明确对象存储也必须清理。

### 12.5 最终原则

```text
用户私有数据都带 user_id
大文件放对象存储，数据库只保存 storage_key
历史不可变内容用 snapshot 保存
当前解析结果和历史面试上下文分开
Session 表示一场独立面试
Turn 表示一次有效问答
FeedbackItem 和 Turn 一一对应
RetryAttempt 可以无限追加，不覆盖原回答
删除账号时必须同时清理数据库和对象存储
```
