# MS1 表关系说明

> 日期：2026-07-14  
> 用途：用业务语言解释数据库表之间的关系，辅助理解 ER Model  
> 对应 SQL：`ms1-postgres-schema.sql`

## 1. 总体关系

MS1 第一版一共 11 张核心表：

```text
users
auth_sessions
resumes
experiences
interview_plans
interviewers
interview_sessions
turns
audio_assets
feedback_items
retry_attempts
```

可以把它理解成一条主链路：

```text
用户
  -> 上传简历
  -> 简历解析出经历
  -> 基于一条经历创建面试计划
  -> 计划下生成多个面试官
  -> 每个面试官可以开启多场面试
  -> 每场面试有 4 次有效问答
  -> 每次问答生成一条反馈
  -> 每次问答可以反复复练
  -> 原回答和复练回答都有自己的音频记录
```

## 2. 用户与登录

相关表：

```text
users
auth_sessions
```

关系：

```text
users 1 -> N auth_sessions
```

含义：

一个用户可以有多个登录态。比如同一个账号可以在电脑浏览器登录一次，也可以在另一个浏览器登录一次。

`users` 负责保存账号本体：

```text
邮箱
密码哈希
默认简历
注销时间
```

`auth_sessions` 负责保存登录状态：

```text
哪个用户登录了
登录 token 的哈希
什么时候过期
是否已经退出
```

重要原则：

```text
数据库不保存明文密码
数据库不保存原始登录 token
```

## 3. 用户与简历

相关表：

```text
users
resumes
experiences
```

关系：

```text
users 1 -> N resumes
resumes 1 -> N experiences
users 1 -> N experiences
```

含义：

一个用户最多可以上传 3 份简历。

一份简历可以解析出多条经历，例如：

```text
项目经历 A
实习经历 B
工作经历 C
```

`resumes` 表保存的是简历文件的索引，不保存 PDF 文件本体。

真实 PDF 文件放对象存储，例如：

```text
users/u_123/resumes/r_001/source.pdf
```

数据库里只保存：

```text
resumes.storage_key
```

`experiences` 表保存从简历里解析出来的经历。

例如：

```text
title = "AI Interview Coach Prototype"
type = "project"
background = "项目背景"
responsibility = "个人职责"
achievements = "项目成果"
```

为什么 `experiences` 里也有 `user_id`？

虽然一条经历可以通过 `resume_id` 找到用户，但直接保存 `user_id` 可以让权限校验更简单：

```text
查询 experience 时同时判断 user_id
防止用户 A 访问用户 B 的经历
```

## 4. 简历与面试计划

相关表：

```text
resumes
experiences
interview_plans
```

关系：

```text
resumes 0/1 -> N interview_plans
experiences 0/1 -> N interview_plans
```

含义：

一个面试计划通常来自某份简历和某条经历。

但是：

```text
interview_plans 不能只依赖 experiences
```

因为用户以后可能删除简历，或者重新解析简历。

所以 `interview_plans` 里有一个非常关键的字段：

```text
experience_snapshot_json
```

它保存的是“创建面试计划那一刻，用户确认过的经历内容”。

也就是说：

```text
experiences = 当前简历解析结果
experience_snapshot_json = 历史面试使用的固定上下文
```

这样即使用户删除了简历，历史面试报告仍然能正常展示。

## 5. 面试计划与面试官

相关表：

```text
interview_plans
interviewers
```

关系：

```text
interview_plans 1 -> N interviewers
```

含义：

一个面试计划下有 1 到 4 位面试官。

默认可以生成 4 位，例如：

```text
技术负责人
业务面试官
项目深挖面试官
HR 面试官
```

`interview_plans` 保存共享背景：

```text
目标岗位
岗位描述
练习重点
补充要求
经历快照
```

`interviewers` 保存单个面试官的视角：

```text
姓名
职责
风格
关注点
排序
```

重要理解：

```text
多面试官不是多人同时面试
多面试官是多个独立面试入口
```

所以不是：

```text
一个 session 里有多个 interviewer
```

而是：

```text
一个 interviewer 可以开启多场 session
```

## 6. 面试官与面试场次

相关表：

```text
interviewers
interview_sessions
```

关系：

```text
interviewers 1 -> N interview_sessions
```

含义：

同一位面试官可以被用户反复练多次。

例如：

```text
技术负责人
  -> 第一次面试 session
  -> 第二次面试 session
  -> 第三次面试 session
```

每一场 `interview_sessions` 都是独立记录：

```text
独立进度
独立问题
独立回答
独立报告
独立复练历史
```

`interview_sessions` 里保存：

```text
当前状态
已经完成几问
开始时间
完成时间
是否提前结束
报告生成状态
报告总结
```

## 7. 面试场次与问答

相关表：

```text
interview_sessions
turns
```

关系：

```text
interview_sessions 1 -> N turns
```

含义：

一场完整面试固定有 4 次有效问答，也就是 4 条 `turns`。

每个 `turn` 代表：

```text
AI 问了一个问题
用户给了一次有效回答
系统拿到了可用转录
```

如果用户回答太短、空白、断线、识别失败，就不应该算成有效 `turn`。

`turns` 里保存：

```text
第几问
这一问的固定目标
AI 完整问题
用户实际听到的问题
是否打断
原回答音频 ID
原回答转录
回答完成时间
```

4 个固定目标是：

```text
1. project_overview        项目概览
2. personal_contribution   个人贡献
3. tradeoff_decision       方案取舍
4. validation_result       验证与结果
```

## 8. 问答与反馈

相关表：

```text
turns
feedback_items
```

关系：

```text
turns 1 -> 1 feedback_items
```

含义：

每一次有效问答，都必须生成一条反馈。

一场完整面试有 4 个 `turns`，所以应该有 4 条 `feedback_items`。

`feedback_items` 保存：

```text
证据是否完整
引用的用户原话
诊断说明
下一次改进目标
示例表达
结构化缺口
```

这里的反馈不是打分系统，也不是判断用户技术水平对不对。

它只回答：

```text
用户有没有把真实项目证据讲清楚
哪些信息讲清楚了
哪些信息还缺
下一次应该补什么
```

## 9. 问答与复练

相关表：

```text
turns
feedback_items
retry_attempts
```

关系：

```text
turns 1 -> N retry_attempts
feedback_items 1 -> N retry_attempts
```

含义：

用户可以针对同一道原问题反复复练。

例如：

```text
第 2 问：请说明你的个人贡献
  -> 第 1 次复练
  -> 第 2 次复练
  -> 第 3 次复练
```

每次复练都会创建一条新的 `retry_attempts`，不会覆盖原回答，也不会覆盖之前的复练。

`retry_attempts` 保存：

```text
复练的是哪个 turn
参考的是哪条 feedback
第几次复练
复练音频 ID
复练转录
已经补充了哪些缺口
仍然缺少哪些内容
```

为什么复练同时关联 `turns` 和 `feedback_items`？

因为：

```text
复练的是同一道原问题，所以要关联 turn
复练对比的是原反馈缺口，所以要关联 feedback_item
```

## 10. 音频索引表

相关表：

```text
audio_assets
turns
retry_attempts
```

关系：

```text
audio_assets 1 -> N turns
audio_assets 1 -> N retry_attempts
```

更准确地说：

```text
turns.answer_audio_asset_id -> audio_assets.id
retry_attempts.audio_asset_id -> audio_assets.id
```

含义：

`audio_assets` 不保存音频文件本体。

它只保存：

```text
这个音频属于谁
这个音频在对象存储里的 storage_key
音频类型
文件大小
音频时长
转录文本
转录状态
```

真实音频文件在对象存储里，例如：

```text
users/u_123/sessions/s_456/turns/t_001/original.webm
users/u_123/retry-attempts/r_789/retry.webm
```

数据库里保存：

```text
storage_key = "users/u_123/sessions/s_456/turns/t_001/original.webm"
```

所以 ER 图里会出现 `audio_assets`，但不会出现真实的 `.webm` 文件。

播放音频时的流程：

```text
前端请求播放 audio_id
后端查询 audio_assets
后端检查 audio_assets.user_id 是否等于当前登录用户
检查通过后，后端从对象存储读取 storage_key 对应的文件
后端返回音频流，或者生成短期签名 URL
```

## 11. 删除关系

### 11.1 删除简历

删除简历时：

```text
删除 resumes
删除这份简历解析出的 experiences
删除对象存储里的 PDF 文件
不删除 interview_plans
```

原因：

```text
历史面试计划已经保存了 experience_snapshot_json
历史报告不能因为简历删除而消失
```

### 11.2 删除面试计划

删除面试计划时，通常会删除下面所有内容：

```text
interview_plans
  -> interviewers
    -> interview_sessions
      -> turns
        -> feedback_items
        -> retry_attempts
```

但要注意：

```text
数据库删除 audio_assets 记录，不等于删除对象存储里的音频文件
```

删除音频文件需要业务代码额外处理对象存储。

### 11.3 注销账号

注销账号时，要删除：

```text
用户账号
登录态
简历记录
经历记录
面试计划
面试官
面试场次
问答记录
反馈记录
复练记录
音频索引
对象存储里的 PDF
对象存储里的音频
```

这是全量删除。

## 12. 最简记忆版

如果只记一条主线，可以记这个：

```text
users
  -> resumes
    -> experiences

users
  -> interview_plans
    -> interviewers
      -> interview_sessions
        -> turns
          -> feedback_items
          -> retry_attempts

turns / retry_attempts
  -> audio_assets
```

再记三个关键原则：

```text
1. 用户私有表都带 user_id，用于权限隔离。
2. PDF 和音频文件放对象存储，数据库只存 storage_key。
3. 历史面试上下文必须存 snapshot，不能只依赖当前简历解析结果。
```
