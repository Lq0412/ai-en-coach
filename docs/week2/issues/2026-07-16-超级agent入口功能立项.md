# [提案] 增加超级 Agent 统一入口，用自然语言调用 App 内能力

> 调研依据：[SpeakUp 超级 Agent 入口调研与立项论证](../张思成/2026-07-16-超级Agent入口调研与立项论证.md)

## 动机 / 用户故事

SpeakUp 当前采用传统 App 导航模式。用户需要先理解产品的信息架构，再通过菜单和按钮进入历史记录、面试 Review、继续练习或专项训练。

但用户真正想完成的通常不是“打开某个页面”，而是一个具体目标，例如：

> 帮我复盘上周那场 Java 后端面试。

传统操作要求用户先判断该目标对应哪个功能，再找到历史模块、筛选面试记录并进入 Review。随着产品能力增加，这种操作方式会带来更高的功能发现和导航成本。

本提案希望增加一个支持语音与文字输入的超级 Agent 入口。用户只需描述目标，AI 负责识别意图、查询相关数据并返回可操作气泡。用户确认后，可以直接进入对应对象的页面。

一句话概括：

**用户只需要说出想完成什么，由 AI 负责找到并调用正确的 App 能力。**

## 目标用户

- 已完成至少一次模拟面试，希望复盘历史表现的用户；
- 不熟悉 SpeakUp 功能结构，希望快速找到对应能力的用户；
- 习惯通过语音表达目标，不希望逐层点击页面的用户；
- 希望在面试、Review 和复练之间连续操作的用户。

## 现有做法及其不足

当前完成一次历史面试复盘，用户需要：

```text
产生复盘目标
  -> 判断应该进入历史模块
  -> 找到目标面试
  -> 打开面试记录
  -> 进入 Review
```

存在以下问题：

1. 用户需要先理解“历史记录”和“Review”等产品概念；
2. 用户需要记住功能所在位置；
3. 历史记录较多时，需要额外筛选；
4. 产品功能增加后，菜单和页面层级会继续变复杂；
5. 面试、Review 和复练之间仍然是相互分离的操作路径。

Pendo 的功能采用报告显示，平均软件产品中约80%的功能很少或从未被使用。该数据不能直接代表 SpeakUp，但说明功能发现不足是软件产品的普遍问题。

与此同时，CNNIC数据显示，截至2025年12月，中国生成式人工智能用户达到6.02亿，普及率为42.8%。用户通过自然语言向 AI 描述目标的习惯正在快速形成。

移动端输入研究也显示，中文语音输入速度约为手机键盘输入的2.8倍。SpeakUp 本身就是语音练习产品，因此语音目标输入与现有场景具有较高一致性。

## 本期范围与明确不做

### 本期范围

本期先以“查找并 Review 某场历史面试”为首个完整闭环，不直接建设覆盖全部功能的万能 Agent。

#### 统一入口

- 在首页提供清晰可发现的 Agent 入口；
- 支持语音输入；
- 支持文字输入，适配不方便说话的场景；
- 用户可以查看并修改语音识别结果。

#### 意图识别

首期支持以下表达：

- “帮我复盘上周的面试”；
- “Review 我最近一次 Java 后端面试”；
- “打开昨天那场产品经理模拟面试”；
- “看看我最近一次面试表现怎么样”。

Agent 至少需要识别：

- 用户意图：查看面试 Review；
- 时间范围：昨天、上周、最近一次或具体日期；
- 目标岗位：如 Java 后端、产品经理；
- 目标面试记录。

#### 面试检索与结果气泡

- 根据用户指令检索当前用户的历史面试；
- 找到唯一结果时，展示包含关键信息的确认气泡；
- 气泡至少展示面试日期、岗位和进入 Review 的操作按钮；
- 用户点击气泡后，直接进入该场面试的 Review 页面，而不是进入历史模块首页；
- 没有匹配结果时，明确告知用户并提供可执行的下一步；
- 存在多个匹配结果时，展示候选记录供用户选择；
- 信息不足时，通过一次简短追问补充条件。

示例：

> 找到了 7 月 10 日的 Java 后端模拟面试<br>
> 时长 28 分钟<br>
> **查看 Review**

#### 原有路径

- 保留历史记录和 Review 的传统页面入口；
- Agent 作为统一快捷入口，不作为唯一入口；
- Agent 识别或调用失败时，用户仍可通过传统导航完成任务。

### 明确不做

- 本期不支持任意 App 功能的通用工具调用；
- 不支持跨多个模块的长链路自动执行；
- 不自动修改或删除用户数据；
- 不替代传统导航；
- 不在意图不明确时静默选择某场面试；
- 不自动进入错误成本较高的页面；
- 不建设系统级 Siri、Apple Intelligence 或 Gemini 集成；
- 不在本期生成跨多场面试的综合问题分析；
- 不在本期自动创建完整学习计划；
- 不以语音搜索关键词后打开模块首页作为完成标准。

## 关键决策与依据

### 决策1：先验证 Review，不直接建设万能 Agent

Review 是高频、对象明确且风险较低的场景，能够完整验证以下关键链路：

```text
自然语言输入
  -> 意图识别
  -> 历史数据检索
  -> 结果确认
  -> App 深链跳转
```

如果该链路能够降低任务耗时并提高 Review 使用率，再逐步扩展继续练习、专项复练和进步报告等能力。

### 决策2：语音与文字同时支持

语音能够降低移动端输入成本，也符合英语陪练场景；但办公室、公共交通等环境不一定适合说话，因此文字输入必须同时保留。

### 决策3：先展示确认气泡，再进入页面

Agent 可能错误识别时间、岗位或面试记录。气泡需要展示日期和岗位等关键信息，让用户确认 AI 是否找对对象。

对于存在多个候选结果或置信度不足的请求，不能直接跳转。

### 决策4：直接进入具体对象，不只打开模块首页

如果用户说“复盘上周的 Java 面试”，Agent 只打开历史模块首页，用户仍需手动检索，无法真正降低任务成本。

因此，首期必须支持带具体面试记录参数的 Review 深链。

### 决策5：Agent 是统一入口，不是语音版菜单

Agent 的核心价值是理解目标、识别参数并调用能力，而不是将固定关键词映射到一级页面。

后续可以逐步扩展：

```text
检索最近三场面试
  -> 读取 Review
  -> 归纳重复问题
  -> 推荐最高优先级弱项
  -> 进入对应专项练习
```

但该跨模块链路不属于本期验收范围。

## 基本概念与信息结构

| 概念 | 定义 |
|---|---|
| Agent Entry | 用户通过语音或文字表达目标的统一入口 |
| User Utterance | 用户输入的原始自然语言指令 |
| Intent | 用户希望完成的任务，本期只支持历史面试 Review |
| Entity | 从指令中提取的时间、岗位和面试记录等参数 |
| Tool | Agent 可以调用的 App 内部能力 |
| Tool Result | 工具检索或执行后返回的结构化结果 |
| Action Bubble | 展示匹配结果、确认信息和后续按钮的气泡 |
| Deep Link | 可以直接进入某场面试 Review 的页面路径 |
| Clarification | 指令信息不足或结果不唯一时的澄清问题 |

建议流程：

```text
用户语音或文字输入
  -> 转写并确认文本
  -> 识别 Review 意图
  -> 提取时间与岗位
  -> 检索历史面试
      -> 唯一结果：展示确认气泡
      -> 多个结果：展示候选面试
      -> 没有结果：提示重新描述或进入历史
  -> 用户确认
  -> 打开对应 Review
```
