# [提案] 引入基于 Mem0 的用户长期记忆系统

## 动机 / 用户故事

SpeakUp 当前只能在单次会话中使用上下文。进入新会话后，用户需要重新说明目标岗位、学习偏好和需要继续练习的内容。

典型用户故事：

> 作为持续练习英文面试的用户，我希望 AI 能记住我的目标、偏好和长期薄弱点，让下一次训练能够接着上一次继续。

本提案建议直接引入官方 Mem0，建设跨会话的用户长期记忆能力。

## 本期范围

- 使用官方 `mem0ai`，不自行实现记忆引擎；
- 从对话中提取适合长期保存的用户信息；
- 在生成回复前检索相关记忆；
- 将记忆作为用户背景注入模型上下文；
- 支持记忆的查询、修改、删除和历史记录；
- 按 `user_id` 隔离不同用户的数据；
- Mem0 不可用时降级为无记忆对话，不影响原有链路。

首期保存以下内容：

- 目标岗位和学习目标；
- 回答风格和反馈偏好；
- 用户确认过的项目背景；
- 重复出现的训练薄弱点；
- 用户明确要求下次继续的内容。

## 明确不做

本期不做：

- 不自行实现向量检索和记忆去重算法；
- 不保存完整聊天记录作为长期记忆；
- 不保存密码、Token 等敏感数据；
- 不建设知识图谱；
- 不迁移全部历史会话；
- 不将 #45 的面试知识库内容保存为用户记忆；
- 不用记忆系统代替面试、训练和报告等业务数据。

## 关键决策

### 决策一：完全使用官方 Mem0

记忆的提取、搜索、更新、去重和删除使用 Mem0 原生能力：

- `add`
- `search`
- `get_all`
- `get`
- `update`
- `delete`
- `delete_all`
- `history`

项目只负责 Mem0 配置、接口适配、用户隔离和业务接入。

### 决策二：记忆系统保持独立

`assistant` 只负责编排，通过统一接口调用记忆模块，不直接访问 Mem0 或记忆数据库。

```text
用户输入
-> 检索相关记忆
-> 注入模型上下文
-> 生成回复
-> 将本轮有效信息交给 Mem0
```

### 决策三：区分用户记忆与知识库

- Mem0 保存用户目标、偏好和个人背景；
- #45 保存题目卡、Rubric 和面试知识；
- 知识库内容不得被记录成用户个人经历。

### 决策四：记忆只作为参考信息

召回的记忆不能覆盖系统提示、当前用户输入和业务数据。

如果当前输入与历史记忆冲突，以用户当前输入为准。

## 验收标准

- [ ] 用户在会话 A 中说明目标岗位后，会话 B 能召回该信息；
- [ ] 用户偏好可以影响后续回复；
- [ ] 支持查看、修改和删除记忆；
- [ ] 支持查看记忆变更历史；
- [ ] 不同用户之间的记忆完全隔离；
- [ ] 删除后的记忆不会再次被召回；
- [ ] 知识库内容不会被保存为用户经历；
- [ ] Mem0 不可用时，原有对话和训练仍能正常运行；
- [ ] `assistant` 不直接依赖 Mem0 SDK 和存储实现。
