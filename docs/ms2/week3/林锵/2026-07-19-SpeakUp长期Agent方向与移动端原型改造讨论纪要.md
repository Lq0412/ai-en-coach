# SpeakUp 长期 Agent 方向与移动端原型改造讨论纪要

> 日期：2026-07-19
> 性质：用户反馈整理、产品方向澄清、移动端原型改造建议
> 当前结论：批准验证新方向，不代表已经完成全面 Pivot 决策

## 一、讨论背景

一位创业师兄反馈了以下需求：

- 不能保证每天有完整、固定的学习时间；
- 不想每次都先创建一次正式练习；
- 更希望像微信一样直接发送语音消息；
- 希望可以连续发送多句话，而不是 AI 每句话都立刻抢答；
- 希望 AI 像一个长期认识自己的联系人，能够记住上下文并跨天继续；
- 不需要实时语音电话，更接近异步但连续的自然聊天。

这次反馈暴露的并不只是“缺少一个语音按钮”，而是当前产品对用户行为的基本假设可能过于传统：产品默认用户先决定要学习，再进入一次正式训练；但忙碌用户的真实英语需求往往先发生在工作和生活中，用户只想先把事情说出来。

## 二、需求成立到什么程度

### 已经较可信的部分

1. 正式练习的启动成本对碎片时间用户偏高。
2. 固定一问一答容易产生机械感，不能承载完整表达。
3. 异步语音是用户已经熟悉的沟通方式，不需要额外教育。
4. 面试属于高意图但阶段性的需求，难以独自承担长期留存。

### 尚未被验证的部分

1. 用户是否会长期与英语 Agent 交流，而不只是首次觉得新鲜。
2. 连续发送多段语音是否为更大目标人群的普遍行为。
3. 长期记忆是否能显著提升复访和付费。
4. 用户是否愿意为这一体验付费，而不是直接使用通用 AI。
5. 用户想要的身份更接近朋友、教练，还是工作沟通搭档。

因此更准确的判断是：

> 问题高度可信，解决方向值得立即验证；但单次访谈不足以证明完整产品形态和商业模式已经成立。

## 三、Agent 不是“意图识别入口”

意图识别只能解决“用户想做什么”。如果系统听到“我要准备客户会议”后，只是打开场景练习页面，本质仍然是用自然语言替代点击导航。

真正的长期 Agent 至少包含以下层次：

```text
语音或文字输入
    ↓
判断多条消息是否属于同一件事
    ↓
理解用户的意图与现实事件
    ↓
结合长期记忆和当前上下文
    ↓
直接回应或选择合适能力
    ├─ 帮助组织表达
    ├─ 润色英文消息
    ├─ 发起客户或会议排练
    ├─ 调用面试模拟
    └─ 复盘并跟进现实结果
    ↓
结果回到原对话并沉淀为记忆
```

OpenClaw、Hermes 的类比有帮助：它们体现的是长期在线、跨会话记忆、技能调用和任务延续。SpeakUp 不应复制一个通用个人助手，而应做一个边界清楚的垂直版本。

建议定位为：

> SpeakUp 是一个记得用户的异步语音优先职业英语联系人。用户可以把现实中说不清的事直接发给它，必要时它再把真实沟通转化为专项排练。

## 四、当前移动端原型的问题

当前首页以“今天想练什么？”为主标题，并突出三个入口：

- 创建模拟面试；
- 场景口语练习；
- 继续上次练习。

页面底部虽然已经具有聊天输入框，但它在信息层级上仍然从属于三个任务入口。用户首先感知到的还是“选择学习功能”，而不是“联系一个认识我的人”。

当前场景练习链路大致为：

```text
选择场景练习
→ 描述场景
→ 系统整理场景
→ 查看场景
→ 场景确认
→ 开始练习
→ 第 1/4 轮
→ 结束并保存
```

这套流程适合一次正式训练，但不适合用户只有两三分钟、只想讲述一件真实事件的情况。

当前“第 1/4 轮、先听问题再回答、开始说话、结束并保存”等设计，也强化了课程和考试心智。问题并非移动端视觉不够现代，而是产品仍以任务、轮次和完成状态为中心。

## 五、建议的移动端最终结构

### 1. 一个稳定 Agent，多条自然话题

第一版不建议做角色广场，也不建议让用户创建多个互相割裂的 Agent。

更合适的关系是：

- 用户始终面对同一个 SpeakUp；
- 不同现实事件形成不同话题线程；
- 英语水平、工作背景和表达习惯跨线程共享；
- Alex、客户、面试官等角色只在专项排练时临时出现。

可能的话题包括：

- 向客户解释项目延期；
- 准备下周的英文面试；
- 海外会议发言；
- 入职自我介绍；
- 复盘刚刚发生的一次英文沟通。

### 2. 首页直接变成持续对话页

Agent 不是首页上的一张功能卡，Agent 就是产品本身。移动端默认进入最近一条对话，而不是进入功能选择大厅。

首页可采用以下结构：

```text
☰        SpeakUp · 职业英语伙伴        ＋

SpeakUp：昨天你说周五要和 Alex 讨论延期，
材料后来准备得怎么样了？

用户语音  ▶ 0:18
用户语音  ▶ 0:07

SpeakUp 语音  ▶ 0:22
可按需展开文字转写

┌────────────────────────┐
│ 要不要让我扮演 Alex，追问你两分钟？ │
│ [开始排练]  [先不用]                │
└────────────────────────┘

＋              按住说话              键盘
```

首次使用、还没有上下文时，可以在输入框上方提供弱提示，例如“准备一次会议”“说说刚发生的事”“练习面试”“润色一段英文”。它们不应继续成为占据首页的三张大卡片。

### 3. 不增加复杂底部导航

第一版可以继续沿用当前左侧抽屉，收敛为：

- 新建话题；
- 最近事项；
- 记忆与进步；
- 我的设置。

“练习记录”和“错题回顾”可以合并到“记忆与进步”，避免过强的题库和考试感。

## 六、现有页面的具体改法

| 当前页面或组件 | 建议调整 |
|---|---|
| “今天想练什么？”首页 | 改为持续对话时间线 |
| 三个大功能入口 | 降级到输入框“＋”菜单或上下文建议 |
| 场景练习描述页 | 用户直接在原对话中用语音描述 |
| “场景已整理”卡片 | 改成聊天内的轻量排练邀请 |
| 独立场景确认页 | 默认取消，复杂任务才需要确认 |
| “第 1/4 轮”练习页 | 改成自然排练，弱化固定轮数和考试感 |
| 独立完成报告 | 结果直接返回原对话，并保存少量有效记忆 |
| 错题回顾 | 改为“我的表达”或“记忆与进步” |
| 继续上次练习 | 由 Agent 根据上下文自然追问或推荐 |
| 多种语音模式按钮 | 合并为用户容易理解的语音消息与键盘输入 |

## 七、连续语音的交互原则

产品不应要求用户点击“说完了”，也不应向用户暴露一个固定的 8—15 秒等待规则。

建议行为如下：

1. 每段语音发送后立即成为一个消息气泡。
2. Agent 不在第一段语音结束后立刻抢答，而是内部判断表达是否完整。
3. 用户再次开始录音时，系统暂停或取消尚未发送的回复。
4. 连续发送的多段语音自动归入同一组上下文。
5. Agent 可以自然回复一至两条语音，文字转写按需展开。
6. 用户可以在 Agent 回复过程中继续补充或打断。

这属于对话节奏和消息分组能力，不是一个需要用户主动切换的产品模式。

## 八、面试功能的正确优先级

当前理解基本正确：面试不再是首页和产品主路径，但不能简单判断为“最低价值功能”。需要区分不同维度：

| 维度 | 面试优先级 |
|---|---|
| 首页展示 | 低，不应继续统治首页 |
| 当前重构开发 | 较低，暂时不必继续扩展复杂流程 |
| Agent 能力库 | 中等，是重要专项能力 |
| 获客和付费 | 可能较高，仍然是高意图入口 |

现有面试能力可以先冻结和保留，后续从 Agent 对话中调用：

```text
用户讲述下周的面试
→ Agent 自动带入简历、岗位和历史表现
→ 只询问缺失信息
→ 对话内出现“开始模拟面试”卡片
→ 进入现有专注面试模式
→ 完成后回到原对话
→ 结果进入长期成长记录
```

因此，面试从产品主角降级为高价值专项技能，而不是被删除。

## 九、建议的产品优先级

### P0：先验证长期 Agent 主循环

- 持续对话首页；
- 多段异步语音；
- 自然语音回复；
- 最小上下文与可见记忆；
- 跨天继续同一现实事项。

### P1：跑通真实沟通与排练闭环

- 从聊天中识别真实沟通困难；
- 聊天内出现短排练邀请；
- 进入临时专注练习；
- 练习结果返回原对话；
- 跟进现实使用结果。

### P2：接入现有面试能力

- Agent 自动补全已有信息；
- 压缩固定 1/4 配置流程；
- 将面试作为聊天内可调用技能；
- 复用现有报告、复练和学习资产。

### P3：验证通过后再扩张

- 角色广场；
- 多 Agent；
- 社区；
- 大量角色模板；
- 更广泛的自动化能力。

## 十、哪些资产保留，哪些主路径重做

### 可以保留

- 当前移动端视觉风格；
- 顶部栏、侧边抽屉和输入框基础结构；
- 面试模拟能力；
- JD、简历和候选人背景；
- Alex 等角色排练页的基础框架；
- 原音、转写、表达诊断、复练和报告资产。

### 需要重做或降级

- 首页的三大任务入口；
- 强制先选择功能的路径；
- 场景描述、确认、开始的多层跳转；
- 所有场景固定四轮的结构；
- 将练习报告与原对话割裂的结果页；
- 过强的“错题、轮次、完成训练”学习心智。

当前原型不是完全作废，它更像已经实现的能力库和专注练习 Demo；但它不能直接作为长期产品的主壳层。

## 十一、需要避免的方向偏差

### 1. 不要从“面试太窄”跳成“什么都能聊”

交互方式可以通用，但产品边界仍应聚焦：用户现实中有一件想用英语说清楚、但没有说顺的事。

### 2. 不要把 Agent 做成新的功能导航器

如果 Agent 只是识别意图并打开页面，用户仍然需要理解产品内部的模块划分，产品本质没有改变。

### 3. 不要先做角色广场

角色广场会增加选择成本、切碎长期记忆，也没有直接解决师兄提出的碎片时间和自然连续表达问题。

### 4. 不要把记忆等同于保存全部聊天

记忆应至少区分：

- 用户背景：岗位、英语水平、长期目标；
- 事件上下文：客户、项目、会议、面试等未完成事项；
- 学习记忆：常见表达问题、已经掌握的表达。

用户需要能够查看、修改和删除“它记得什么”。

## 十二、下一步验证建议

在全面重构前，先用 12—30 名具有真实职业英语需求的用户进行 14 天左右验证。重点观察真实行为，而不是只询问用户喜不喜欢概念：

- 用户是否会在没有“开始练习”提示时主动发第一条语音；
- 是否自然连续发送两段以上语音；
- 是否会跨天继续同一件现实事件；
- 是否会从日常聊天进入一次短排练或面试；
- 是否在现实沟通中使用了 Agent 给出的表达；
- 用户能否说清 SpeakUp 比通用 AI 多保存了什么价值；
- 是否愿意为继续使用付出真实费用。

如果用户只觉得聊天自然，却不进入排练、不产生现实结果，产品可能退化为陪聊；如果用户仍然只在面试前使用，面试仍然是核心事件型产品；只有“现实对话—专项排练—现实结果—长期改进”连接起来，新方向才真正成立。

## 十三、当前阶段的一句话结论

> SpeakUp 的下一步不是给现有面试产品增加一个语音聊天功能，而是验证一个以长期对话为产品本体、以面试和场景排练为可调用能力的职业英语 Agent。对话是入口，练习是动作，面试是高价值专项模式，长期可见的现实表达改善才是最终价值。
