# 把规范"长"在流程里——用 Claude Code Skills 替代没人看的开发文档

> 5 人团队 · 8 周交付 · 10 个 AI Skill 串起完整研发流水线

---

每个团队都有开发规范。CONTRIBUTING.md、代码规范、分支策略、Issue 模板……写得再认真，**也没人看**。

我们在参加七牛云 X-Engineer 实训营时想了一个办法：把规范写成 Claude Code Skills——不是让人读的文档，而是 **AI 在每一步自动执行的规则**。本文聊聊我们怎么做的。

## 问题：规范文档为什么没用？

三个原因：

1. **时机不对**。写代码时不会专门翻 CONTRIBUTING.md，写完才想起来格式不对
2. **记不住**。10 条规则，写到第 3 条就忘了前两条
3. **没有后果**。格式错了也能合入，规范就形同虚设

核心矛盾是：**规范存在文档里，但工作在编辑器里**。两者不在同一个时空。

## 解法：把规范写成 Skill

Claude Code 的 Skill 机制允许我们把团队约定写成结构化提示词，放进 `.claude/skills/` 目录。开发者工作时，AI 助手**自动加载对应 Skill**，在正确的时机执行规范。

我们写了 10 个 Skill，按研发流水线串起来：

```
写 Issue          提交代码          提 PR           发版
   │                │               │               │
   ▼                ▼               ▼               ▼
issue-standard   pr-commit       pr-review     release-tag
                 code-discipline  branch-commit
                 
         ┌──────────────────────────┐
         │  milestone-manager       │  ← 贯穿全程，管排期和进度
         └──────────────────────────┘
```

## 七个核心 Skill 的设计思路

### 1. issue-standard：一个 Issue 一天能做完

传统 Issue 常犯两个毛病：要么太粗（"完成用户模块"），要么太细（"改按钮颜色"）。

我们的规则很暴力：**一个 Issue 一个人、一天能量完**。超过一天就拆。描述不写"做什么"，写"验收标准"——读完 Issue 能直接判断做没做完。

### 2. code-discipline：决策阶梯

AI 写代码容易过度设计。我们定了一个"决策阶梯"，每一级够用就停：

```
真需要吗？(YAGNI)
  → 标准库能做吗？
    → 平台原生能力够吗？
      → 现有依赖已解决吗？
        → 能一行搞定吗？
          → 最后才写最小实现
```

前面几级把 90% 的过度设计拦住了。

### 3. pr-commit：先对齐，再套模板

不强行统一格式，而是**先对齐项目已有习惯**——读 `git log`、读现有 PR、读项目约定文件。无约定时才用默认模板。这避免了一个常见问题：新规范跟项目历史风格打架。

同时规定了分支命名（`feat/xxx`）和 PR 四段描述（背景、改动、验证、风险）。

### 4. branch-commit & pr-review：管节奏、管质量

branch-commit 管的是**提交频率**——每人每天至少一次推送，避免攒一周才提 PR 的灾难。

pr-review 管的是**合入门槛**——不经过 Review 不能合入 dev，AI 生成的代码走同样流程，且提交者必须能讲清每段逻辑。

### 5. milestone-manager & release-tag：8 周 4 个里程碑

```
MS1 (W1-2)：方向成立吗？   → v0.1.0  可运行骨架
MS2 (W3-4)：核心路径通吗？  → v0.2.0  核心功能可演示
MS3 (W5-6)：MVP 做完了吗？  → v0.3.0  全部功能可试用
MS4 (W7-8)：交付真实吗？    → v1.0.0  功能冻结 + 文档
```

每个 MS 一个核心问题 + 一个可打 tag 的交付物。进度是否正常，一眼判断。

## 这套体系的核心洞察

**规范不应该是一份文档，而应该是工作流的一部分。** Skill 把规范从"需要记得"变成了"自动执行"——开发者在写代码时，AI 在正确的时机提醒正确的规则。

三个关键设计原则：

1. **触发时机比内容更重要**。每个 Skill 定义了明确的触发词——不是让人记住什么时候用，而是 AI 在匹配到场景时自动激活
2. **优先对齐项目习惯**。规范不是从天而降的，先看项目已有风格再适配
3. **互相引用、形成体系**。Skill 之间通过 `[[pr-commit]]` 等链接形成知识图谱，不是 10 个孤岛

## 实际效果

用了一周，几个明显变化：

- Issue 拆分粒度明显均匀了——不再有"一个 Issue 做三天"的情况
- Commit message 格式自动统一——AI 在 commit 时就按规范生成
- PR 描述不再是"fix bug"——四段模板自动填好

**规范终于不是"写了没人看"的东西了**——因为它不再需要人专门去看。
