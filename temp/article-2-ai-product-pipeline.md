# 一个命令，三步分析——AI 产品调研流水线

> product-killshot → web-to-design-md → ai-design-smell 三连击

---

做产品前要调研竞品。传统做法：手动翻竞品网站、截图、分析功能、写报告。几个竞品下来，**两三天没了**。

我们给 AI 搭了一条流水线——三个 Skill 串联，一个命令跑完全程。聊聊怎么做的。

## 三步流水线

```
竞品网址/描述
     │
     ▼
┌─────────────────────┐
│ 1. product-killshot │  → 竞品功能分析 + 差异化建议
└────────┬────────────┘
         ▼
┌─────────────────────┐
│ 2. web-to-design-md │  → 提取竞品设计系统（配色/字体/间距/交互）
└────────┬────────────┘
         ▼
┌─────────────────────┐
│ 3. ai-design-smell  │  → 审查自己的 UI 是否有"AI 模板味"
└─────────────────────┘
```

## Step 1：product-killshot — 竞品功能分析

这个 Skill 做的事情：

1. 读你的项目代码，理解产品定位
2. 自动搜索市场上头部竞品
3. 从两个不同用户视角分别分析
4. 合成一份带优先级的优化报告

输出不是"竞品有什么功能"的罗列，而是**"我们应该做什么、先做什么"**。

关键设计：执行策略不是"把所有竞品功能列出来"，而是**找切换成本和十倍改进点**——用户为什么离开竞品选我们？我们怎么做能让用户离不开？

## Step 2：web-to-design-md — 提取设计系统

给定竞品网址，自动提取：

- 配色方案（主色、辅色、灰度）
- 字体层级（标题/正文/标签的字号字重）
- 间距系统
- 组件风格（按钮圆角、阴影、卡片样式）
- 交互气质（动效节奏、留白密度）

输出一份 `DESIGN.md`，另一个 Agent 仅凭这份文档就能复现同样的设计语言。

**价值不在于"抄设计"，而在于理解竞品的设计决策**——为什么他们用这个配色？他们的信息层级怎么组织的？

## Step 3：ai-design-smell — 审查 AI 设计味

做完自己的 UI 后，跑一遍 ai-design-smell。它会识别五种"AI 模板化设计"特征：

- 卡片套娃
- 蓝紫渐变 + 发光按钮
- 文字层级不分明
- 什么都居中
- 标题堆大图标

每条都有对应的修复方向。相当于给 UI 做了一次自动体检。

## 流水线的设计思路

单个 Skill 只解决一个问题。**三个串起来，形成完整的产品分析闭环**——知道竞品在做什么 → 理解他们怎么做设计 → 避免自己踩进 AI 模板化的坑。

Skill 之间互相独立，可以单独用，也可以组合。这种"可组合的工具链"比一个大而全的脚本更灵活——因为实际工作中，你不需要每次都跑全流程。

## 实际跑一次的效果

以我们的 AI 英语口语陪练项目为例：

1. product-killshot 发现：市面口语 App 普遍在"发音评测"上做得浅——只给分数不说问题在哪。这成了我们的核心差异化方向。
2. web-to-design-md 提取了几个竞品的设计系统，发现它们几乎都用了蓝紫渐变 + 大圆角卡片——这就是我们要避开的"模板味"。
3. ai-design-smell 在我们出第一版 UI 时拦住了至少 5 处模板化设计。

**三个 Skill 跑完不到 10 分钟，替代了至少两天的调研工作量。**
