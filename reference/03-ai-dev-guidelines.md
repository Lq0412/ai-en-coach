# AI 时代的开发规范

## 背景

1024XEngineer 的项目在 `CLAUDE.md`（即 `AGENTS.md`）中定义了给 **LLM Coding Agent** 的行为规范。这不是给人看的传统编码规范，而是专门为 AI 辅助开发设计的原则。

## ByteMind 的 CLAUDE.md（完整翻译）

**取舍说明**: 这些规范偏向谨慎而非速度。对于简单任务，用你自己的判断。

### 1. 先想再写 (Think Before Coding)

**不要假设。不要隐藏困惑。暴露权衡。**

实施之前：

- 明确陈述你的假设。如果不确定，就问。
- 如果存在多种理解，把它们呈现出来——不要默默选一个。
- 如果存在更简单的方法，说出来。在合适的时候提出质疑。
- 如果有不清楚的地方，停下来。说清楚哪里困惑。问出来。

### 2. 简单优先 (Simplicity First)

**最少代码解决问题。不要写推测性的代码。**

- 不写超出需求的功能
- 不为只使用一次的代码创建抽象
- 不添加未被要求的"灵活性"或"可配置性"
- 不为不可能发生的场景做错误处理
- 如果你能 50 行写完却写了 200 行，重写它

问自己："资深工程师会说这是过度设计吗？"如果会，简化。

### 3. 外科手术式修改 (Surgical Changes)

**只碰必须改的。只清理你自己弄乱的东西。**

修改现有代码时：

- 不要"顺便改进"旁边的代码、注释或格式
- 不要重构没坏的东西
- 匹配现有风格，即使你会用不同的写法
- 如果你注意到无关的坏代码，提出来——不要直接删除

当你的修改制造了孤立代码时：

- 删除**你的修改**造成的无用 import/变量/函数
- 不要删除已有的坏代码，除非被要求

**检验标准**: 每一行改动都应该能直接追溯到用户的需求。

### 4. 目标驱动执行 (Goal-Driven Execution)

**定义成功标准。循环直到验证通过。**

把任务转化为可验证的目标：

- "加个验证" → "为无效输入写测试，然后让它们通过"
- "修一下这个 bug" → "写一个能复现 bug 的测试，然后让它通过"
- "重构 X" → "确保重构前后测试都通过"

对多步骤任务，先给出简要计划：

```
1. [步骤] → 验证: [检查点]
2. [步骤] → 验证: [检查点]
3. [步骤] → 验证: [检查点]
```

强成功标准让你可以独立循环。弱标准（"让它能用"）需要不断澄清。

---

**这些规范有效的标志**: diff 中不必要的改动更少，因过度复杂而重写的次数更少，澄清性问题在实施前而不是在犯错后才出现。

## 可以直接复用到 UniSpeaking

在你的项目根目录创建一个 `CLAUDE.md`，把上面的内容放进去（可以按需要调整）。这样当 Claude Code 在这个项目中工作时，会自动遵循这些原则。

## 这个文件的运作机制

CLAUDE.md / AGENTS.md 会被 Agent 的 `prompt.go` 加载到系统提示词中：

```
系统 Prompt 组装顺序:
① 基础 Prompt (default.md)
② 模式 Prompt (build/plan)
③ 环境/技能/工具信息
④ 激活的技能 Prompt
⑤ AGENTS.md 行为规范  ← 在这里
```

所以它不是"文档"，而是 **Agent 行为的一部分**。
