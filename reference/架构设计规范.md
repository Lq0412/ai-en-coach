> 所属体系：XEngineer 营软件工程规范（与产品设计规范、GitHub 过程管理规范并列）

---

架构解决的核心问题：**如何将产品实现出来，并支撑后续迭代**。

架构分两层：**模块架构**（单个模块内部）和 **系统架构**（模块如何划分、配合、接口定义）——后者是本营重点。

系统架构的载体：入库的空骨架代码（经 PR Review 合并），辅以架构说明文档。草稿写在 Issue 中，定稿只读，调整时另写新文档描述变化，不覆盖旧版本。

---

## 一、总架构先行

MS1 的核心任务之一是确定总架构——完成支撑主干串联的模块划分、关键数据模型与接口契约，形成可持续叠加功能的骨架。

### 代码即架构

系统架构推荐以"代码即架构"的方式表达——在仓库中用代码明确定义模块、接口与数据模型。

**核心理念**：「串联已完成、模块内部实现为空」——代码可编译，但尚未产生真实行为，数据可为 mock。

**后端（以 Go 为例）**：直接定义 package、interface 与 struct，方法体留空或桩实现，但模块间相互调用的关键处要写出来使串联成立。

**前后端边界**：前端不直接调用后端 HTTP 接口，而是封装一层类 SDK（业务语义的 model 层）。举例：

```
api.user.list({ page: 1, size: 10, search: 'gmail' })
```

这层 SDK 的 TypeScript 类型将产品设计中的"基本概念与信息结构"在代码中固定下来。

### 重要原则

- **系统架构应由人主导设计**。搭空骨架的价值在于想清楚模块边界与协作关系。
- **不得以 AI 生成的粗浅实现入库冒充架构**。
- 总架构一旦定稿，后续每轮在其上叠加功能——产品定义完成后可直接进入开发，无需为每个功能重做架构。
- **总架构就绪前不得绕过它直接开发。**

## 二、设计步骤

架构设计承接产品设计定稿，将产品需求落实为可运行的系统方案。四个主要步骤：

1. **需求梳理**：从产品定稿的关键决策与"基本概念与信息结构"出发，梳理需求之间的内在联系，从技术实现视角思考每个需求的实现方式。

2. **风险分析**：识别技术实现中复杂、难度高的环节，列为详细设计重点，尽早排除不确定性。

3. **模块拆分**：将系统拆分为职责清晰的模块，明确模块间配合方式及各自对外接口。

4. **详细设计**：针对重点模块与高风险环节，确定内部实现方案，包括关键数据结构与核心流程。

### 对应产出

| 产出 | 说明 |
|:-----|:-----|
| **空骨架代码** | 系统架构的主要载体。模块划分与对外接口以代码写清，接口为真实定义，实现留空或桩实现（数据可为 mock）。经 PR Review 后合并入库。**骨架是"模块间对接与主干串联的契约"，是 MS1 核心产出，应尽早稳定。** |
| **架构说明（辅助、可选）** | 承载代码难以表达的内容，如模块划分依据与整体协作逻辑，可辅以模块关系图。结构直观的项目可省略。 |
| **重点模块及难点的详细设计文档** | 对关键模块与高风险点实现方案的说明，可随功能推进逐轮深化。 |

## 三、跟踪架构演进

每个 Milestone 复盘时，记录本轮架构相对上一轮的变更内容及变更原因。对架构演进的跟踪本身即是衡量架构能力的方式。

### 两类警惕信号

| 坏状态 | 根因 |
|:-----|:-----|
| **频繁大改** | 主干 MS1 没有定准 |
| **架构稳定但功能交不出来** | 架构不支撑迭代 |

### 理想状态

MS1 确定的主干后期仅作微调，主要在其上叠加功能——架构稳定且功能持续交付，即为架构能力强的体现。

> **参考实例**：xgo 项目总架构长期稳定，从 v1.6 到 v1.7 系列规律发布，单个 milestone 闭合数十个 issue，架构未大改而功能持续交付。

## 四、评价标准

架构设计过线的四项要求：

1. 空骨架已入库、主干串联，模块划分与对外接口清晰
2. 关键技术风险已识别，重点模块有详细设计，不确定性在编码前排除
3. 后续功能能在既有架构上落地，无需反复推翻
4. 每轮演进有记录，变化可解释

### 典型不合格

- 将架构写成目录罗列与技术栈清单
- 以 AI 生成的粗浅实现入库、未经真正的架构思考便当作架构完成
