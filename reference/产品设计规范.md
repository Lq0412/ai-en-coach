> 来源：[1024XEngineer/techcamp Wiki](https://github.com/1024XEngineer/techcamp/wiki/01-%E4%BA%A7%E5%93%81%E8%AE%BE%E8%AE%A1%E8%A7%84%E8%8C%83)
> 编者：mcell，更新于 2026-07-07
> 所属体系：XEngineer 营软件工程规范（与架构设计规范、GitHub 过程管理规范并列）

---

产品设计是营内"最难也最关键"的环节——功能能不能做对，首先取决于产品设计有没有把"做什么、不做什么"讲清楚。规范化在这里是营里对每个组的第一要求。

产品设计文档是工程文档：草稿阶段在 Issue 中推进，进入开发后即视为只读，变更走新文档而非覆盖旧文档。

---

## 一、为什么先做产品设计

软件的价值取决于"做对了东西"，而不只是做完了东西。方向错了——不是用户需要的，或偏离了整体目标——即使实现再完美也是浪费，且越晚发现代价越大。

产品设计的任务是在开发之前厘清：**做什么、为谁做、解决什么问题、不做什么**，并让团队达成共识，开发才能走在正确的方向上。

---

## 二、产品设计是决策，不是功能清单

产品设计的核心是**取舍**。每个设计点都要回答"有哪些做法、选了哪个、为什么"：

- **备选方案**：列出多种可能的方向
- **选择与理由**：明确选了什么、放弃了什么，给出依据（用户价值、成本、风险、现阶段边界）
- **不做什么**：明确划出本期不做的东西，与"做什么"同等重要

只列功能、不做取舍的文档，不算是合格的产品设计。

---

## 三、设计流程

产品设计不是一步到位，而是从调研到定稿逐步收敛，一般包括以下步骤：

### 3.1 行业与竞品调研

调研行业常见做法、竞品方案及取舍，梳理各方做法，明确哪些值得借鉴、哪些要避开。这一步属于设计前的**需求分析**，不做强制标准化，团队可用腾讯文档、飞书、Wiki 等任意载体。Mentor 在这一阶段给的是方向性引导，而非直接给结论。

### 3.2 讨论与达成共识

在调研基础上，团队讨论并收敛出统一方向，形成初步设计。共识必须在此阶段达成，防止后期分歧。

### 3.3 细化设计（两步法，顺序不可颠倒）

**第一步 — 文字描述**：先用文字把设计决策讲清楚——解决什么问题、包含什么排除什么、边界在哪里。这是纯决策表达，不涉及界面。

**第二步 — 原型**：在文字定稿的基础上，制作原型/界面，将交互和形态具体化。

- **无界面产品**（编译器、SDK、CLI 工具）：只需第一步
- **有界面产品**：两步都需要，且必须**先文字、后原型**

> **无界面示例**：xgo proposals 是语法级的文字描述，无原型，例如 #2797 Function Decorators
> **有界面示例**：XBuilder UI proposals 如 #494 v1.3 UI Design、#3214 animation UI。营内无专职设计师，原型直接贴在产品设计文字文档下方。

设计文档和原型可通过 draft PR（不合并，定稿后回到 Issue）逐行 Review，具体要求见 GitHub 过程管理规范。

### 3.4 定稿为 Proposal

细化设计完成后定稿为 **Proposal（产品提案）**，即最终交付物。结构与要求见下一章。

### 原型做法随阶段不同

- **MS1**：尚无基础代码，原型可用原型工具独立制作
- **MS2**：已有真实工程代码，原型应基于真实代码搭建——使用仅展示、不合入的 PR（本质是 web coding），明确"产品应该长什么样、达成什么行为"。让这样的不合入 PR 跑起来成为可交互的界面，Mentor/助教会帮助实现，不是学生的硬性工作量。

---

## 四、Proposal 的编写要求

每个有分量的需求都应先形成一份 Proposal（产品提案），写入对应 Issue，将"做什么、不做什么"表达为可 Review、可实施、可验收的决策。

### 4.1 五条原则

1. **一次一个增量**：一份 Proposal 确定一个清晰的设计点，不试图覆盖整个系统；相邻决策拆成后续 Proposal。宁可提一个能通过、能实现、能验证的窄提案，也不要提一个试图解决所有未来问题的大提案。

2. **简单直接**：能用真实例子说明的，先给例子再抽象；能用一条规则表达的行为，不要引入多个机制。

3. **减法设计**：主动去除冗余和不必要的负担（样板代码、手动步骤、需要用户额外记忆的特殊情况），而非为了求全而堆叠功能。

4. **边界清晰**：明确什么被解决、什么不被解决，什么是合法情况、什么是错误，既有的行为是否改变。

5. **从真实场景抽象**：需求应来自多个真实用户故事的共性点，而非机械地从单个 case 加功能。

### 4.2 应包含的内容

| 字段 | 说明 |
|:-----|:-----|
| 动机 / 用户故事 | 触发的真实场景或痛点，共性需求是什么（一句话概括） |
| 目标用户 | 面向谁 |
| 现有做法及其不足 | 用户目前怎么解决的，仍有哪些冗余或手动负担；如不适用则写"无" |
| 本期范围与明确不做 | 包含什么；**明确列出不做的东西**（非 trivial 提案必须写） |
| 关键决策与依据 | 核心规则或方案；有哪些做法，选了哪个，为什么（含竞品/替代方案对比） |
| 基本概念与信息结构 | 核心领域概念如何定义，信息/数据如何组织——这是给架构设计的输入 |
| 原型 / Demo | 有界面的产品附上文字定稿后的原型；无界面产品跳过 |
| 验收标准 | 做到什么程度算完成——尽量用可验证的具体例子表述 |

**以例子定义行为**：例子不是配图，而是规范本身。展示"现状 / 方案后 / 边界与非法情况"比长篇抽象描述更利于 Reviewer 理解、也利于实施落地。AI 可以显著提高 Proposal 的效率，但**取舍由人决定、且可被否决**——详尽是为了辅助判断，不是替代判断。

**详略随影响面而定**：小改动走短提案，把上述项表达清楚即可。持续性、影响面广的改动展开边界、替代方案与测试。不对小改动上全模板，不对大改动马虎了事——对应 `MiniSpec` / `FullSpec`。

## 五、状态流转

设计讨论在 draft PR 中展开，定稿后回到 Issue。Proposal 状态通过 Issue Label 标记：

| 标签 | 含义 |
|:-----|:-----|
| `proposal`（草案） | 设计中，开放讨论与修改 |
| `Proposal-Accepted`（通过） | 定稿，进入开发；从此只读，变更走新版本 |
| `Proposal-Denied`（拒绝） | 方案不成立；记录原因并归档。拒绝是正常结果，不是失败 |
| `Proposal-NoPlan`（暂不排期） | 方案成立但本期不排；记录并保留，后续可启用 |

按变更粒度区分 `FullSpec` / `MiniSpec`：大影响面用 FullSpec，小改动用 MiniSpec。

> **真实示例**：#2802 Flat Mode 已通过并标为 FullSpec；#2667 defer 命令语法被拒绝。

---

## 六、验收

产品设计文档须满足以下要求方可通过：

- 设计方向有行业与竞品调研支撑，有据可循
- 明确了做什么、不做什么，本期边界清晰
- 关键设计点有备选方案与选择依据，而非只列功能
- 基本概念与信息结构清晰，可直接交予架构设计
- 有界面的产品，文字定稿后有对应原型
- 状态与规格标签完整可追溯
