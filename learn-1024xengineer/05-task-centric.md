# Task-Centric 架构

## 核心洞察

普通 AI 产品是 **Chat-Centric**（以聊天轮次为中心），CialloClaw 是 **Task-Centric**（以任务为中心）。

### Chat-Centric 的问题

```
用户: "帮我分析这个数据"
AI: "好的，结果是..."
用户: "不对，换一种方式"
AI: "试试这样..."
```

聊天模式下，上下文靠消息历史维护，任务没有结构、不能恢复、无法追踪进度。

### Task-Centric 的做法

```
task (对外主对象)
  ├── 状态: pending → running → done / cancelled
  ├── 描述: 用户原始意图
  ├── 上下文: 锁定的工作区快照
  └── runs (内部执行链)
       ├── run_1 (第一次尝试)
       │   ├── step_1 → tool_call → result
       │   ├── step_2 → tool_call → result
       │   └── step_3 → 完成
       └── run_2 (用户补充需求后的续接)
           └── ...
```

## Task 对象的生命周期

```
create → running → paused → running → done
              ↓                    ↓
          cancelled            failed → recovered
```

CialloClaw 的设计：

- **对外**只暴露 `task` 对象（用户看到的是"任务工作台"）
- **对内**保留 `run / step / event / tool_call`（用于执行、排障和回放）
- 用户可以在工作台查看所有任务，恢复未完成的任务

## 状态机 vs 消息列表

| | Chat-Centric | Task-Centric |
|---|---|---|
| 状态管理 | 消息数组 | 任务状态机 |
| 中断恢复 | 重新描述需求 | 从上次暂停处继续 |
| 审批 | 无 | 高风险操作需用户确认 |
| 审计 | 无 | 每次工具调用都有审计日志 |
| 多任务 | 上下文混乱 | 每个任务独立状态 |

## 对 UniSpeaking 的启示 — 最重要的一篇

口语教练天然适合 Task-Centric 设计：

**不要把口语练习当成"和 AI 聊天"，而应该是"完成一个学习任务"。**

### 任务类型示例

```
task: "今日发音练习 /θ/ 和 /ð/"
  ├── step_1: 听标准发音
  ├── step_2: 跟读录音
  ├── step_3: AI 评分 + 纠错
  ├── step_4: 针对薄弱点再练 3 次
  └── step_5: 生成练习报告

task: "场景对话：餐厅点餐"
  ├── step_1: 场景介绍 + 关键词预习
  ├── step_2: 角色扮演对话（AI 扮演服务员）
  ├── step_3: 对话回放 + 纠错
  └── step_4: 重练搞砸的句子
```

### 带来的好处

1. **用户看到进度**: "今天完成了 3 个发音任务，/θ/ 从 55 分提升到 78 分"
2. **中断可恢复**: 聊到一半被打断，下次打开从当前 step 继续
3. **错题本**: 每个 task 里标红的句子自动汇总到"需要复习"列表
4. **学习计划**: 可以排 7 天的发音任务，每天解锁下一个

这才是"教练"而不是"聊天机器人"。
