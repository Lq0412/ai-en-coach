# Issue 规范

一个 Issue 只做一件事。写好 Issue 比写好代码更重要——它决定了谁会接、能不能做完、做完怎么验。

## 规范来源与适用优先级

本项目优先参考 `reference/GitHub过程管理规范.md`、`reference/产品设计规范.md` 和 `reference/一些安排与共识.md`。这些文件来自 1024XEngineer/techcamp 过程规范，比本 skill 内置模板更贴近当前团队实践。

使用本 skill 时按以下优先级判断：

1. **先看目标仓库已有做法**：不要为了“统一”强行批量改历史 Issue、标签或 Milestone。若仓库现有调研 Issue 只使用 `调研` 标签，就沿用该风格。
2. **普通调研 Issue 轻量处理**：调研阶段不强制四段模板、不强制优先级标签或技术域标签；只要求背景清晰、目标明确、验收标准或产出标准具体。
3. **Proposal 才走完整规范**：进入产品/架构设计定稿时，使用 `proposal`、`FullSpec` / `MiniSpec`、`Proposal-Accepted` / `Proposal-Denied` / `Proposal-NoPlan` 等标签。
4. **不要把本地配置当作组织强制规范**：`.Codex/skills` 是执行辅助，不是覆盖团队现有流程的最高规则。

## 触发场景

- 从需求文档或讨论中创建 Issue
- 把大任务拆成多个 Issue
- 给 Issue 打标签、关联 Milestone
- 检查 Issue 是否写到位了

## 核心原则

1. **范围可控**：一个 Issue 对应一个清晰问题、一个工程文档草稿、一个任务拆解或一个设计澄清点。
2. **背景清晰、目标明确、验收具体**：读完 Issue 应能判断为什么做、要产出什么、做到什么程度算完成。
3. **标题不能空泛**：避免“优化一下 XX”“完善一下 XX”这类不可验收标题。
4. **尊重 Issue 类型差异**：调研、讨论、Proposal、实现任务的结构要求不同，不要用同一套模板硬套。
5. **PR 可追溯**：进入开发后，每个 PR 应关联对应 Issue，且改动范围与 Issue 一致。

## Issue 模板

### 标题格式

```
[类型] 简短描述（中文，≤50 字）
```

**类型前缀**（与 commit type 对齐，见 [pr-commit](../pr-commit/SKILL.md)）：

| 前缀 | 什么时候用 |
|------|-----------|
| `[提案]` | 产品 Proposal / 设计方案（走 Proposal 生命周期） |
| `[功能]` | 新功能 |
| `[修复]` | Bug 修复 |
| `[重构]` | 重构，不改对外行为 |
| `[文档]` | 文档 |
| `[调研]` | 技术调研、市场调研 |
| `[杂项]` | 构建、CI、配置 |

**示例：**
- `[提案] 用户认证系统重构 — 产品 Proposal`
- `[功能] 文件上传模块 WebSocket 集成`
- `[修复] 表单提交后残留数据未清理`
- `[调研] PostgreSQL vs MySQL 性能对比`

### 普通任务正文结构（推荐，不强制套给调研汇总）

```markdown
## 背景
<!-- 为什么要做这个？关联哪份文档/决策？ -->

## 要做的事
<!-- 3-5 条可验证的事项，非开发也能看懂 -->

## 验收标准
<!-- 做完之后怎么验证？具体步骤 + 预期结果 -->

## 关联
<!-- 依赖哪个 Issue？阻塞哪个 Issue？哪个 MS？哪个文档？ -->
```

普通实现任务、明确的技术调研任务、子任务适合使用以上四段。已有调研汇总、会议结论、个人研究沉淀不必为了模板感重写。

### 调研 Issue 的最低要求

调研阶段在 `reference/产品设计规范.md` 中明确属于设计前需求分析，不做强制标准化。创建或修改调研 Issue 时优先保证：

- 问题背景：为什么现在要调研
- 调研目标：要回答哪些问题
- 产出形式：对比表、推荐结论、风险清单、PoC 验证项等
- 关联关系：主 Issue、Milestone、后续会回填到哪个设计/技术决策

标签沿用仓库现有风格。若同类调研 Issue 只挂 `调研`，不要额外添加 `P1`、`后端`、`全栈` 等标签。

### 完整示例

```markdown
## 背景
技术调研确定使用 [方案名称] 作为 [模块] 方案。见 [相关调研文档]。

## 要做的事
- 封装 [核心模块]，输出 [格式/协议]
- 建立与 [外部服务] 的连接，实现 [功能]
- 实现 [关键逻辑]，[触发条件] 后 [时间窗口] 内返回结果
- 与 [下游模块] 对接，传入 [数据]

## 验收标准
1. [输入 A] → [预期输出]
2. [输入 B] → [预期输出]
3. [边界条件] → [预期行为]
4. [异常场景] → [恢复行为]

## 关联
- 依赖：#12（[依赖的 Issue]）
- Milestone：MS2（核心路径跑通）
- 参考：[相关调研文档]
```

## 标签体系

标签体系的核心服务对象是产品/架构决策流转，而不是给所有 Issue 打满元数据。

### 提案状态标签（产品/设计 Issue 专用）

| 标签 | 含义 | 何时打 |
|:-----|:-----|:-----|
| `proposal` | 该 Issue 是一个产品提案/设计方案 | 创建时即打 |
| `Proposal-Accepted` | 定稿，进入开发；从此只读 | 讨论通过后替换 `proposal` |
| `Proposal-Denied` | 方案不成立；记录原因并归档 | 被否决时。拒绝是正常结果，不是失败 |
| `Proposal-NoPlan` | 方案成立但本期不排 | 讨论通过但推迟时 |

**提案典型生命周期**：`proposal` → 讨论 → `Proposal-Accepted`（+ `FullSpec` / `MiniSpec`）→ 开发

### 规格粒度标签

| 标签 | 含义 |
|:-----|:-----|
| `FullSpec` | 影响面大的完整规格——需完整 Proposal 八字段 |
| `MiniSpec` | 小改动的精简规格——简要表达即可 |
| `sub-task` | 从 Proposal 拆出的实现子任务（上下文过大时拆分） |

### 文档状态标签

| 标签 | 含义 |
|:-----|:-----|
| `Need-Document` | 功能已完成待补用户文档 |
| `Documented` | 用户文档已提供 |

### 优先级标签

| 标签 | 含义 | 何时用 |
|:-----|:-----|:-----|
| `P0` | 阻塞级，本 MS 不做就交不了 | 核心路径 |
| `P1` | 本 MS 应该做 | 重要但非阻塞 |
| `P2` | 下个 MS 可以做 | 有缓冲再做 |

### 技术域标签

以下标签可作为团队内部扩展使用，但不是 1024XEngineer 过程规范的强制项。使用前先看目标仓库是否已有同类标签和实际习惯。

| 标签 | 含义 |
|:-----|:-----|
| `前端` / `后端` / `全栈` | 涉及的技术域 |
| `调研` | 桌面研究或技术调研 |
| `bug` | 已有功能出问题 |

不要强制每个 Issue 至少 2 个标签。若目标仓库尚未采用优先级/技术域标签，不要为了规范感批量新增。

## 拆分规则

满足 **任意一条** 就拆分：

- 涉及 2 个以上不同的技术域（如前端的 UI + 后端的 API）
- 预估工作量超过 1 人天
- 没有一个人能独立做完（依赖链太长）
- 描述里出现「以及」「同时」「另外」

**示例：** 「文件上传功能 + 上传进度 UI」→ 拆成：
- `[功能] 文件上传 API 与存储集成`（后端）
- `[功能] 上传进度展示组件`（前端）

## 创建 Issue 后检查

- [ ] 标题带类型前缀
- [ ] 背景、目标、产出或验收标准清楚
- [ ] 若是 Proposal，标签为 `proposal` + `FullSpec` / `MiniSpec`
- [ ] 若只是调研，标签沿用仓库现有调研风格
- [ ] 如仓库使用 Milestone，则关联到正确 Milestone
- [ ] 不批量重写与当前任务无关的历史 Issue

## Proposal 流程


当 Issue 类型为 `[提案]` 时，走特殊流程：

1. **创建 Issue**：打 `proposal` 标签，按 Proposal 八字段填写（动机/目标用户/现有做法不足/本期范围与明确不做/关键决策与依据/基本概念与信息结构/原型/验收标准）
2. **设计草案 Review**：将设计内容提交为 draft PR，利用行内评论逐条讨论。**此 PR 只 Review、不合并**。
3. **定稿回到 Issue**：讨论定稿后把内容落回 Issue，附上 PR 链接。更新标签为 `Proposal-Accepted`（+ `FullSpec` / `MiniSpec`）。
4. **进入开发**：Issue 成为开发起点与后续跟踪。

### Proposal 五大原则

1. **一次一个增量**：一份 Proposal 确定一个清晰的设计点
2. **简单直接**：能用例子说明的先给例子，再抽象
3. **减法设计**：主动去除冗余，而不是为求全而堆叠
4. **边界清晰**：明确什么被解决、什么不被解决
5. **从真实场景抽象**：需求应来自多个真实用户故事的共性点
