# Issue 规范

一个 Issue 只做一件事。写好 Issue 比写好代码更重要——它决定了谁会接、能不能做完、做完怎么验。

## 触发场景

- 从需求文档或讨论中创建 Issue
- 把大任务拆成多个 Issue
- 给 Issue 打标签、关联 Milestone
- 检查 Issue 是否写到位了

## 核心原则

1. **一个 Issue 一个人能做、一天能量完。** 超过一天 → 拆分。
2. **描述写「验收标准」而不是「做什么」。** 读完 Issue 应该能直接判断「做完了没有」。
3. **必须关联 Milestone。** 不挂 MS 的 Issue 等于没有排期。

## Issue 模板

### 标题格式

```
[类型] 简短描述（中文，≤50 字）
```

**类型前缀**（与 commit type 对齐，见 [pr-commit](../../dev-tools/pr-commit/SKILL.md)）：

| 前缀 | 什么时候用 |
|------|-----------|
| `[提案]` | 产品 Proposal / 设计方案（走 Proposal 生命周期） |
| `[功能]` | 新功能 |
| `[修复]` | Bug 修复 |
| `[重构]` | 重构，不改对外行为 |
| `[文档]` | 文档 |
| `[调研]` | 技术调研、市场调研 |
| `[杂项]` | 构建、CI、配置 |

**示例：**
- `[提案] 高压冲突场景口语训练 — 产品 Proposal`
- `[功能] ASR 语音识别模块 WebSocket 集成`
- `[修复] 对话打断后 TTS 音频残留`
- `[调研] Whisper vs Deepgram ASR 延迟对比`

### 正文结构（必填四段）

```markdown
## 背景
<!-- 为什么要做这个？关联哪份文档/决策？ -->

## 要做的事
<!-- 3-5 条可验证的事项，非开发也能看懂 -->

## 验收标准
<!-- 做完之后怎么验证？具体步骤 + 预期结果 -->

## 关联
<!-- 依赖哪个 Issue？阻塞哪个 Issue？哪个 MS？哪个文档？ -->
```

### 完整示例

```markdown
## 背景
市场调研 Day 5 评审确定方向 D（高压冲突口语）。T1 访谈 6/8 人提到「现有 App 太温和」。
技术预研确定使用 Deepgram 作为 ASR 方案。见 [ASR 选型调研](../../research/tech-预备调研/ASR选型调研.md)。

## 要做的事
- 封装原生音频采集模块（AudioRecord / AVAudioEngine），输出 PCM 16kHz 流
- 建立 WebSocket 连接至 Deepgram API，实现流式语音识别
- 实现服务端 VAD 端点检测，用户说完后 300ms 内返回识别文本
- 与 LLM 模块对接，传入 ASR 文本 + 对话上下文

## 验收标准
1. 说出「Hello, can you hear me」→ 控制台输出完整识别文本
2. 中文「今天天气真好」→ 正确识别，不混入英文
3. 用户沉默 2 秒 → VAD 触发关闭，不产生额外收费
4. WebSocket 断线 → 5 秒内自动重连

## 关联
- 依赖：#12（后端 WebSocket 服务搭建）
- Milestone：MS2（核心路径跑通）
- 参考：[ASR 选型调研](../../research/tech-预备调研/ASR选型调研.md)
```

## 标签体系

> 标签体系来自 [GitHub 过程管理规范](../../../reference/GitHub过程管理规范.md)，用于标记产品设计决策结果、规格粒度、文档状态，使整个项目可追溯。

### 提案状态标签（产品/设计 Issue 专用）

| 标签 | 含义 | 何时打 |
|:-----|:-----|:-----|
| `proposal` | 该 Issue 是一个产品提案/设计方案 | 创建时即打 |
| `Proposal-Accepted` | 定稿，进入开发；从此只读 | 讨论通过后替换 `proposal` |
| `Proposal-Denied` | 方案不成立；记录原因并归档 | 被否决时。拒绝是正常结果，不是失败 |
| `Proposal-NoPlan` | 方案成立但本期不排 | 讨论通过但推迟时 |

**提案典型生命周期**：`proposal` → 讨论 → `Proposal-Accepted`（+ `FullSpec` / `MiniSpec`）→ 开发

### 规格粒度标签

| 标签 | 含义 |
|:-----|:-----|
| `FullSpec` | 影响面大的完整规格——需完整 Proposal 八字段 |
| `MiniSpec` | 小改动的精简规格——简要表达即可 |
| `sub-task` | 从 Proposal 拆出的实现子任务（上下文过大时拆分） |

### 文档状态标签

| 标签 | 含义 |
|:-----|:-----|
| `Need-Document` | 功能已完成待补用户文档 |
| `Documented` | 用户文档已提供 |

### 优先级标签

| 标签 | 含义 | 何时用 |
|:-----|:-----|:-----|
| `P0` | 阻塞级，本 MS 不做就交不了 | 核心路径 |
| `P1` | 本 MS 应该做 | 重要但非阻塞 |
| `P2` | 下个 MS 可以做 | 有缓冲再做 |

### 技术域标签

| 标签 | 含义 |
|:-----|:-----|
| `前端` / `后端` / `全栈` | 涉及的技术域 |
| `调研` | 桌面研究或技术调研 |
| `bug` | 已有功能出问题 |

每个 Issue **至少打 2 个标签**：1 个状态/优先级 + 1 个技术域。 |

## 拆分规则

满足 **任意一条** 就拆分：

- 涉及 2 个以上不同的技术域（如前端的 UI + 后端的 ASR）
- 预估工作量超过 1 人天
- 没有一个人能独立做完（依赖链太长）
- 描述里出现「以及」「同时」「另外」

**示例：** 「ASR 集成 + 发音评测 UI」→ 拆成：
- `[功能] ASR WebSocket 流式识别集成`（后端）
- `[功能] 对话实时转文字 UI 展示`（前端）

## 创建 Issue 后必做

- [ ] 标题带类型前缀
- [ ] 四段正文（背景/要做的事/验收标准/关联）齐全
- [ ] 挂了至少 2 个标签
- [ ] 关联了正确的 Milestone

## Proposal 流程

> 详见 [产品设计规范](../../../reference/产品设计规范.md) 和 [GitHub 过程管理规范](../../../reference/GitHub过程管理规范.md)。

当 Issue 类型为 `[提案]` 时，走特殊流程：

1. **创建 Issue**：打 `proposal` 标签，按 Proposal 八字段填写（动机/目标用户/现有做法不足/本期范围与明确不做/关键决策与依据/基本概念与信息结构/原型/验收标准）
2. **设计草案 Review**：将设计内容提交为 draft PR，利用行内评论逐条讨论。**此 PR 只 Review、不合并**。
3. **定稿回到 Issue**：讨论定稿后把内容落回 Issue，附上 PR 链接。更新标签为 `Proposal-Accepted`（+ `FullSpec` / `MiniSpec`）。
4. **进入开发**：Issue 成为开发起点与后续跟踪。

### Proposal 五大原则

1. **一次一个增量**：一份 Proposal 确定一个清晰的设计点
2. **简单直接**：能用例子说明的先给例子，再抽象
3. **减法设计**：主动去除冗余，而不是为求全而堆叠
4. **边界清晰**：明确什么被解决、什么不被解决
5. **从真实场景抽象**：需求应来自多个真实用户故事的共性点
