# Milestone 管理

把项目按 **8 周 / 4 个 Milestone** 拆成可验收的迭代。每个 MS 必须有一个核心问题和一个可打 tag 的交付物。

## 4 个 MS 结构

| MS | 周 | 核心问题 | 交付物 |
|:---|:---|:---------|:-------|
| **MS1** | 1-2 | 方向和主干成立吗？ | 产品决策 + 空骨架可运行 |
| **MS2** | 3-4 | 核心路径跑通了吗？ | 核心功能可演示 |
| **MS3** | 5-6 | MVP 真的做完了吗？ | 全部功能完成，可试用 |
| **MS4** | 7-8 | 交付真实、表达清楚吗？ | 功能冻结 + 文档 + Tag v1.0 |

## 创建 Milestone

在 GitHub 上创建时填三个东西：

```markdown
## 标题
MS1：战略决策与首个可串联版本

## 描述
### 核心问题
方向和主干成立吗？

### 交付物
- [ ] 产品取舍决策（提案文档：5 问完整回答 + 方案对比）
- [ ] 模块骨架可运行（接口定义、数据模型、目录结构可编译）
- [ ] 技术选型裁定（[列出关键选型项]）
- [ ] Live Demo（核心链路可串联展示）

### 不交付
- [ ] 完整 MVP 功能
- [ ] 真实第三方集成（仅桩实现）
- [ ] 商业化方案

## 截止
2026-07-19
```

### 标题命名规则

```
MS{N}：{一句话使命}
```

示例：
- `MS1：战略决策与首个可串联版本`
- `MS2：MVP 深化与架构落实`
- `MS3：功能闭合与质量达标`
- `MS4：打磨、交付与发布`

## Issue 关联

### 关联到 MS

每个 Issue 的右侧面板 → Milestone → 选对应 MS。

### 关联检查

创建 Issue 或评审 Issue 时，必须确认 **Milestone 字段不为空**。

没有挂 MS 的 Issue 等于没有排期。

### 工作量配平

| MS | 建议 Issue 数 | 每人 Issue 数 |
|:---|:-------------|:-------------|
| MS1 | 10-15 | 2-3 |
| MS2 | 15-20 | 3-4 |
| MS3 | 15-20 | 3-4 |
| MS4 | 5-10 | 1-2 |

超出这个范围 → 检查是否把 Issue 拆得太碎或太大。

## 进度跟踪

### MS 看板

每个 MS 开始后，用 GitHub 的 Milestone 进度条关注百分比。低于预期时主动提醒：

| 现象 | 处理 |
|:-----|:-----|
| MS 过了一半，Issue 完成率 <30% | 检查 scope 是否膨胀，是否需要砍 |
| 某个成员 Issue 全部卡住 | 排查是否被阻塞、是否需要帮助 |
| MS 接近结束，还有 >3 个未开始 | 决定：移入下个 MS / 降优先级 / 放弃 |

### MS 进度速报

报告进度时使用简洁格式：

```
MS1 进度：7/12 已关闭，2 个进行中，3 个未开始
```
